## 需求 1：比价页必须承载四类运价的独立比价场景（不包含 Air 附加费独立比价）

- **出处**：`frontend/src/pages/RateCompare.tsx:23-79`（当前只支持海运港口 id 对的比较）；`backend/app/services/freight_rate_service.py:99-139`（只查 `FreightRate`）；`docs/运价管理_三类分离_业务需求_20260424.md` 需求 2 已把全量运价分为 5 类
- **业务目标**：营业人员在做客户报价或 Demo Step2 PKG 自动填写前，经常需要「挑同一条航线里最便宜/最合适的那一条」。该诉求在不同运价类型上的具体形态如下，必须全部覆盖：
  1. **海运 FCL（Ocean）比价**：选定 POL + POD → 横向对比多个船司在 20GP / 40GP / 40HQ 三箱型上的价格 + BAF 附加 + 有效期 + 是否直航 + 航程。主场景。
  2. **海运 NGB 比价**：与 FCL 同维度同字段，但起运港集中在宁波/乍浦/舟山等 NGB 范畴港口。由宁波分公司半月更新，营业单独查看本批是否到位。
  3. **空运周价（Air）比价**：选定起运地 + 目的地（均为字符串，机场代码或城市名）→ 横向对比多个航司在同一「生效周」下的 7 日价（price_day1..7），附服务描述与生效周窗口。
  4. **拼箱 LCL 比价**：选定 POL + POD → 横向对比多条记录的「每 CBM 价 / 每 TON 价」（W/M 二选一业务）+ 开船日 + 中转港 + 有效期。
- **业务规则**：
  1. 比价场景强制按「运价类型」切分 —— 不同类型的「同航线」定义不同（海运看港口 id，空运看机场字符串），列头不可混；不做跨类合并对比。
  2. **Air 附加费（MYC/MSC）不作为独立比价类型**。理由：附加费按区域+kg 阶梯、没有具体 POL/POD，不存在「同航线多供应商」对比语义；本轮明确放入「不做范围」。
  3. 每次比价结果的条数含义是「同一筛选条件下的候选报价数」，不是「运价总数」；0 条是合法业务结果（表示本航线目前无可用报价）。
- **验收条件**：楢崎在 Demo 现场依次操作四类比价（Ocean/NGB/Air/LCL），每类都能返回一个列头合适、数值能看懂的对比表；且空运的 7 日价必须是 7 列展开、不能被折叠成一个平均数。
- **边界**：
  - 不做「一次同时对比 Ocean + Air」这种跨类对比。
  - 不做 Air 附加费的独立比价 tab。
  - 不做「历史价 vs 当前价」的纵向对比（v2.0）。
- **业务风险**：当前前端只给海运框，楢崎若在 Demo 里点「比价」而想看空运，会发现输入控件根本不支持机场字符串；直接暴露「中台只懂海运」的印象。

## 需求 2：UI 交互方案 —— 选定方案 α（顶部 4 tab 单选，每 tab 自带输入控件）

### 2.1 三方案对照

- **方案 α（顶部 tab 4 选 1，每 tab 自带输入控件）**：页面顶部一行 tab：海运 FCL / 海运 NGB / 空运周价 / 拼箱 LCL。切换 tab 时，输入区与对比表同时切换（海运系用 POL/POD 下拉选港口 id，空运用「起运地」「目的地」字符串输入）。
- **方案 β（入口 radio 单选运价类型后再出输入区）**：顶部一排 radio「请选择运价类型」，选中后下方一次性渲染输入区 + 对比表。
- **方案 γ（二级路由：/compare/ocean、/compare/air 等）**：每类比价独占一个 URL，左侧菜单二级展开。

### 2.2 决策：采用方案 α

- **用户心智流畅度**：与「运价管理」列表页保持一致的 tab 模型，楢崎看到两个页面都是「先选类型再看数据」这一个动作模式，心智负担最低；不需要在「比价」页再学一套 radio 或二级菜单。
- **开发复杂度**：与前端现状改造幅度最小——页面骨架保留，tab 组件复用列表页同一套，输入区按 tab 切换为 2 套控件（港口下拉 / 机场字符串输入）；无需新增路由、无需拆页面。
- **视觉一致性**：比价和列表并排挂在左菜单下「运价管理」模块，同一模块内的 tab 控件风格统一、避免一页 tab 一页 radio 的不协调。

### 2.3 方案 α 与运价管理列表页 tab 方案的异同

| 维度 | 运价管理列表（列表页） | 运价比价（本页） |
|---|---|---|
| tab 数 | 5 个（Ocean / NGB / Air 周价 / Air 附加费 / LCL） | 4 个（Ocean / NGB / Air 周价 / LCL；无 Air 附加费） |
| tab 切换后的输入区 | 无主输入区（筛选器按类隐藏/禁用） | 必有主输入区；Ocean/NGB/LCL 用港口 id 下拉，Air 用字符串输入 |
| 默认 tab | 海运 FCL | 海运 FCL |
| 0 条结果 | 表格空态 + 指引去「运价导入」 | 表格空态 + 指引「该航线暂无候选报价，请换一组起讫地或检查费率是否已激活」 |
| 分页 | 每页 15 | 比价结果**不分页**（一次给全）；见 3.5 |
| 排序 | 不可点 | 不可点；按各类默认规则（见 3.4） |

### 2.4 输入控件差异

- **Ocean / NGB / LCL**：沿用现状 2 个港口下拉（基于 `ports` 表 id），先选 POL 再选 POD。NGB 的下拉仍使用同一个 `ports` 接口，不强制过滤到 NGB 范畴港口（业务原因：营业可能输入 CNSHA 发现没有 NGB 批次，也是一条合法信息，让他自己换）。
- **Air**：2 个**字符串输入框**（起运地 / 目的地），对应 `AirFreightRate.origin` 与 `destination`；不做下拉（当前 `air_freight_rates` 表里的值来源于导入、未做字典归一）。输入匹配规则用「前缀匹配 + 大小写不敏感」，业务含义是「输入 HKG 能命中 HKG / HKG- 等变体」；空输入不触发比价。
- **切 tab 时**：清空全部输入与上一次比价结果，避免「把海运的 POL 带到空运 tab 上去查」这类语义混乱。

## 需求 4：筛选 / 排序 / 结果数量规则

### 4.1 筛选

- **Ocean / NGB**：除 POL / POD 主输入外，**不再提供二级筛选**（船司多选、状态多选、日期范围均不做）。理由：比价本质是「同航线看谁家便宜」，筛子加多了遮蔽候选。v1.0 保持极简。
- **Air**：除起运地 / 目的地主输入外，**不提供二级筛选**；生效周自动「以今天所在周为默认过滤基准」由后端处理（见 4.3），前端不出控件。
- **LCL**：同 Ocean，只有 POL/POD 两个输入。

### 4.2 默认排序

- **Ocean / NGB**：按 `container_20gp` 升序（空值排末尾）—— 与现状 `compare_rates` 行为一致，业务含义「20GP 最便宜的排在第一行」。
- **Air**：按 `price_day1` 升序（空值排末尾）—— 即以「当周周一价」为锚进行排序。若楢崎现场要求换锚点，记录为待确认项，不临场改。
- **LCL**：按 `freight_per_cbm` 升序（空值排末尾）。
- 以上均不让用户点列头改排序，保持 v1.0 简洁。

### 4.3 时间范围过滤（业务规则）

- **Ocean / NGB**：只返回 `status in (active, draft)` 的记录；`valid_to < 今天` 的记录不返回（等同于过期不参与比价）。现状 `compare_rates` 已查 `active/draft` 但未过滤 `valid_to`，需要补足，否则过期 rate 会混进对比表。
- **Air**：只返回 `effective_week_end >= 今天` 的记录（即当前或未来生效周）；若今天不在任何 effective_week 内，返回 0 条（合法空态）。
- **LCL**：只返回 `valid_to >= 今天 或 valid_to is null` 的记录。

### 4.4 结果数量

- 不分页。一次返回全部候选（前端已有 `table-scroll` 容器，纵向滚动即可）。
- 若某类命中 > 50 条（极端情况），在表格上方显示一行三语提示：「候选较多，请缩小航线范围」，不截断数据。

## 需求 5：空态 / 单结果 / 多结果的展示

### 5.1 空态（0 条）

- 表格区居中显示三语文案：「该航线暂无候选报价 / 当該航路に候補運賃がありません / No rates available for this lane」。
- 下一行小字提示：「请检查起讫地是否填写正确；Ocean/NGB 需在『运价管理』中确认该批次是否已激活 / 起終点をご確認ください。Ocean/NGB は『運賃管理』にて当該バッチが有効化されているかご確認ください / Please check origin/destination. For Ocean/NGB, confirm the batch is activated in Rate Management.」
- 空态不算 bug；不抛错 toast。

### 5.2 单结果（1 条）

- 仍以表格形式显示该 1 行（不切换成卡片视图）。
- 不做「唯一候选」特殊提示（楢崎能自己数）。
- 「最低价高亮」逻辑仍然生效（对 1 行而言该行就是最低）。

### 5.3 多结果（≥ 2 条）

- 按 4.2 默认排序展示。
- 最低价按列高亮：Ocean 的 20'DC / 40'DC 两列，Air 的 Day1~Day7 七列分别独立取最低，LCL 的 CBM / TON 两列。其他列不做高亮。
- 结果区 header 保留现状的「共 N 条 / 全{N}件 / {N} quotes」tag（沿用 `compare.totalQuotes` i18n）。

## 需求 8：验收清单（UI 可观察事实，7 条）

1. 进入「运价比价」页，页面顶部能看到一行 tab，包含 4 个选项：海运 FCL / 海运 NGB / 空运周价 / 拼箱 LCL（**不**含 Air 附加费）；三语随系统语言切换正确。
2. 默认选中「海运 FCL」tab，输入区显示 2 个港口下拉框（POL / POD），表格列头是「20' DC / 40' DC / 40' HC / BAF 20/40」等海运字段。
3. 切到「空运周价」tab：输入区切换为 2 个**字符串输入框**（起运地 / 目的地），提交比价后表格列头展开为 Day1 ~ Day7 七列 + 生效周起/止 + 航司 + 服务描述。
4. 切到「海运 NGB」tab：列头与 Ocean 一致，结果区右上角多一枚「NGB 半月更新」tag。
5. 切到「拼箱 LCL」tab：输入区仍是 2 个港口下拉，表格列头为「每 CBM / 每 TON / LSS / EBS / CIC / AMS·ACI·ENS / 开船日 / 中转港 / 航程 / 有效期」；无承运商列。
6. 任何 tab 下点「比价」按钮，若命中 0 条，表格区显示三语空态文案 + 小字提示，**不**出红色报错 toast。
7. 切换 tab 时，上一次的输入与比价结果全部被清空，不会把「Ocean 的 POL id」带到「空运」tab 上。

（第 8 条备选，若需要：切换系统语言到日文/英文，tab 文案 + 列头 + 空态 + tag 三语同步切换，无中文残留。）
