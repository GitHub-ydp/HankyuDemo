## 3. 接口契约

### 3.1 `activate_rate_batch` 签名（不变）

```python
def activate_rate_batch(
    batch_id: str,
    db: Session,
    *,
    dry_run: bool = True,
    force: bool = False,
    selected_row_indices: list[int] | None = None,
) -> dict[str, Any] | None:
    """
    返回 None ⇒ 批次不存在（FastAPI 层转 404）
    返回 dict  ⇒ 命中 RateBatchActivateResponse schema（含成功 / 业务失败 两种）
    抛异常    ⇒ FastAPI 层转 500
    """
```

### 3.2 响应 schema（扩）

```python
# backend/app/schemas/rate_batch.py
class ActivationErrorItem(BaseModel):
    code: str   # F-ACT-04 / F-ACT-05 / DB_UNIQUE / PORT_NOT_FOUND / CARRIER_NOT_FOUND
    row_index: int | None = None
    record_kind: str | None = None
    detail: str

class RateBatchActivateResponse(BaseModel):
    # —— 既有字段（不变）——
    batch_id: str
    batch_status: str            # "active" / "draft"（失败）/ "draft"（dry_run）
    activation_status: str       # "activated" / "dry_run" / "failed" / "already_active" / "empty_batch"
    activated: bool
    imported_rows: int
    skipped_rows: int
    generated_at: datetime
    selected_rows: int
    diff_summary: RateBatchDiffSummary
    is_stub: bool = False        # 本轮改 False（stub 已退场）
    message: str | None = None

    # —— 新增字段 ——
    file_type: str | None = None                       # "air" / "ocean" / "ocean_ngb"
    effective_from: date | None = None
    effective_to: date | None = None
    imported_detail: dict[str, int] = {}               # {"air_freight_rates": 84, "air_surcharges": 63}
    superseded_batch_ids: list[str] = []               # 被降级为 superseded 的老批次 UUID 列表
    warnings: list[str] = []                           # 例："LCL 暂不入库（v2.0）"
```

### 3.3 错误码表（与业务需求 §7 对齐）

| code | 触发条件 | HTTP | activation_status | batch_status | activated |
|---|---|---|---|---|---|
| F-ACT-01 | batch_id 无效 | 404 | — | — | — |
| F-ACT-02 | 批次已 active（幂等保护） | 200 | `already_active` | `active` | `false` |
| F-ACT-03 | records 为空 | 200 | `empty_batch` | `draft` | `false` |
| F-ACT-04 | port / carrier 字典查不到（仅 Ocean/NGB 路径） | 200 | `failed` | `draft` | `false` |
| F-ACT-05 | DB 约束 / FK / SQLAlchemy 异常 | 200 | `failed` | `draft` | `false` |
| F-ACT-06 | 服务重启导致 draft 丢失 | 404 | — | — | — |

## 4. 数据流

### 4.1 端到端流程图

```
FastAPI: POST /rate-batches/{id}/activate
 └→ rate_batch_service.activate_rate_batch(batch_id, db, dry_run, force, selected)
     ├─ 校验 A: draft = _draft_batches.get(batch_id)
     │       └ None → return None（FastAPI → 404 F-ACT-01）
     ├─ 校验 B: draft.batch_status == "active"
     │       └ Yes → return resp(activation_status="already_active")（F-ACT-02）
     ├─ 校验 C: len(draft.parse_records) == 0
     │       └ Yes → return resp(activation_status="empty_batch")（F-ACT-03）
     │
     ├─ dry_run 分支:
     │   └ 复用现有 diff 摘要 → return resp(activation_status="dry_run",
     │                                   imported_rows=0, activated=False)
     │
     └─ real 分支: activator.activate(draft, db, force=force)
         │
         ├─ file_type = draft.legacy_payload["file_type"]  # "air" / "ocean" / "ocean_ngb"
         ├─ effective_from/to = draft.legacy_payload.get("effective_from/to")
         │
         └─ with db.begin():  # 单事务包裹
             ├─ Step 1: 查该 file_type 所有 status=active 的老批次
             │          → UPDATE import_batches SET status='superseded'
             │          → superseded_batch_ids = [...]
             ├─ Step 2: INSERT import_batches (
             │            batch_id=draft.batch_id, file_type=X,
             │            status='active', source_file, sheet_name,
             │            effective_from, effective_to, row_count=0 占位,
             │            imported_by='step1_activator')
             ├─ Step 3: for record in draft.parse_records:
             │          ├─ record.record_kind == "air_weekly"   → mappers.to_air_freight_rate(record, batch_id)
             │          ├─ record.record_kind == "air_surcharge" → mappers.to_air_surcharge(record, batch_id)
             │          ├─ record.record_kind == "fcl"          → mappers.to_freight_rate_from_ocean(record, batch_id, db)  # db 用于查 carrier
             │          ├─ record.record_kind == "ocean_ngb_fcl"→ mappers.to_freight_rate_from_ngb(record, batch_id, db)
             │          └─ record.record_kind in {"lcl","ocean_ngb_lcl"} → 跳过，skipped+1，warnings.append("LCL 暂不入库（v2.0）")
             │          （单条失败抛 ActivationError → 整事务 rollback，见 §4.2 D-4）
             ├─ Step 4: session.add_all(orm_objects)
             ├─ Step 5: UPDATE import_batches SET row_count = imported_rows
             │          WHERE batch_id = draft.batch_id
             └─ commit
         │
         ├─ draft.batch_status = "active"（内存状态同步）
         └─ return ActivationResult(
                 imported_rows=N, skipped_rows=S,
                 imported_detail={"air_freight_rates":84, "air_surcharges":63},
                 superseded_batch_ids=[...], warnings=[...], errors=[])
```

### 4.2 关键决策（D-1..D-9）

**D-1 · draft → records 怎么拿**
- 方案 A：从 `DraftRateBatch.legacy_payload` 反序列化回 `ParsedRateRecord`
- 方案 B：激活时重跑 `parse_excel_file(draft.file_path, db)`
- **方案 C（选这个）**：上传时 `_build_draft_batch` 直接持 `parse_result` 引用（或至少 `parse_result.parsed_rows` 列表），激活时直接用。理由：
  - 避免重复解析（Air 文件 ~2s 可忽略但属无意义浪费）
  - 避免反序列化的字段脱漏（`ParsedRateRecord` 有 `date / Decimal / extras` 嵌套，legacy_payload 都被 str 化过）
  - 内存成本 = 147 条 × 数百字节 = 几十 KB，可忽略
  - **代价**：重启后 draft 连带 records 一起丢，但这本就是 R-02 已知约束

**D-2 · DraftRateBatch 的数据结构扩展**
- `DraftRateBatch` 新增 `parse_records: list[ParsedRateRecord]`（`rate_batch_service.py:28-48`）
- `_build_draft_batch` 里多一行：`parse_records = list(parse_result.parsed_rows) if parse_result.parsed_rows else [r for s in parse_result.sheets for r in s.rows]`

**D-3 · import_batches 记录什么时候插**
- 方案 A：激活时才 INSERT（status=active）
- 方案 B：draft 创建时就 INSERT（status=draft），激活时 UPDATE status=active
- **选 A**。理由：
  - draft 本来就是内存 stub（R-02），DB 里不留半成品最干净
  - 失败回滚简单：事务里一起 rollback，DB 不留任何痕迹
  - 代价：不能在 DB 里查历史 draft——但业务需求 §9.3 明确不做审批/撤销，没价值

**D-4 · 原子性策略**
- 业务需求 §4.4 "整体回滚"
- 实装：整个激活流程包在 `with db.begin():` 里，任一 record 映射失败 / INSERT 失败 → 抛 `ActivationError(code, row_index, detail)` → SQLAlchemy 自动 rollback → 上层捕获后组装 errors 数组返回 200
- **不做部分成功**

**D-5 · 老 active 降级的顺序**
- 同事务内：先 `UPDATE ... SET status='superseded' WHERE file_type=? AND status='active'` → 再 INSERT 新 `import_batches(status='active')`
- 理由：同事务 FK 约束天然保护；不会出现"短暂两条 active"
- 注意：`air_freight_rates.batch_id` 是 FK 到 `import_batches.batch_id`，UPDATE import_batches 不影响子表（子表不存 status）

**D-6 · 同批次内 record 去重 / 冲突**
- 扫 `AirFreightRate`：只有 `ix_air_freight_rate_origin_dest_week` 非唯一索引 + `ix_air_freight_rate_batch` 非唯一索引 → **无 unique 约束**
- 扫 `AirSurcharge`：同上，无 unique 约束
- 扫 `FreightRate`：无 unique 约束
- **结论**：同批次两条同目的地 record 都写进去，保留 parser 的决策（目前 Air parser 每个目的地一行，不会重复）
- 若将来冲突，由业务决定加 unique，本轮不动

**D-7 · Ocean/NGB 的 carrier_id 解析**
- Ocean parser 已在 adapter 内用 `_resolve_port` 查过 port_id（`ocean.py:136, 197, 285, 295`），records 里 `origin_port_id / destination_port_id` 已填
- Ocean-NGB parser 只填 `carrier_name / origin_port_name / destination_port_name`（`ocean_ngb.py:384-386`），**未查字典**
- **映射器职责**：
  - `to_freight_rate_from_ocean(record, batch_id, db)`: 直接用 record 的 `origin_port_id / destination_port_id`；查 carrier：`db.query(Carrier).filter(Carrier.name.ilike(f"%{record.carrier_name}%")).first()`（复用 `import_service._find_carrier` 的策略，`backend/app/services/import_service.py:125`）。找不到 → 抛 `ActivationError("CARRIER_NOT_FOUND", ...)`
  - `to_freight_rate_from_ngb(record, batch_id, db)`: 额外调 `_resolve_port` 查 port_id（复用 `backend/app/services/rate_parser.py:150` 已有函数）+ 查 carrier。找不到 → 抛 `ActivationError("PORT_NOT_FOUND", ...)` 或 `CARRIER_NOT_FOUND`

**D-8 · LCL 跳过的处理**
- `record_kind in {"lcl", "ocean_ngb_lcl"}`: 不 INSERT 任何表，`skipped_rows += 1`，`warnings.append("LCL 暂不入库（v2.0）: row_index=N origin=X dest=Y")`
- 不阻断激活

**D-9 · Air weekly 的 airline_code 字段**
- parser 的 `ParsedRateRecord.airline_code = None`（Air weekly 一行多航司），但 `extras["airline_codes"] = [...]`
- `AirFreightRate.airline_code` 字段是 `nullable=True`
- **映射**：`airline_code=None`，保留 `service_desc` 原文（含所有航司代码），Step2 的 `_extract_airline_codes(service_desc)` 已经能从 service_desc 里抽出航司列表（`rate_repository.py:192-199`）

## 6. R-01 Step2 effective_on 默认值修复

### 6.1 问题复述

`backend/app/services/step2_bidding/bidding_orchestrator.py:100`：
```python
eff_date = effective_on or datetime.utcnow().date()
```
Demo 2026-05-14 调用，而 Air 批次周窗口 2026-04-20~04-26 → `AirFreightRate.effective_week_start <= eff_date <= effective_week_end` 永远 False → `filled=0`。

### 6.2 修复方案（决策）

- **不改 `RateMatcher.match` 的 `effective_on` 参数**（它是内部 API，正交）
- **不改 `RateRepository.query_air_weekly` 的签名**（T-B3 已稳定，pytest 已过）
- **只改两处**：
  1. `rate_repository.py` 加方法 `infer_default_effective_on(file_type=air) -> date`
  2. `bidding_orchestrator.py:100` 把 `datetime.utcnow().date()` 替换为 `repo.infer_default_effective_on(file_type=ImportBatchFileType.air)`

### 6.3 推断策略

```python
def infer_default_effective_on(
    self,
    *,
    file_type: ImportBatchFileType = ImportBatchFileType.air,
) -> date:
    today = datetime.utcnow().date()
    stmt = (
        select(ImportBatch.effective_from, ImportBatch.effective_to)
        .where(ImportBatch.status == ImportBatchStatus.active)
        .where(ImportBatch.file_type == file_type)
        .order_by(ImportBatch.effective_to.desc())
        .limit(1)
    )
    row = self._db.execute(stmt).first()
    if row is None:
        return today
    eff_from, eff_to = row
    if eff_from and eff_to and eff_from <= today <= eff_to:
        return today      # 窗口内：用真今天
    if eff_to is not None:
        return eff_to     # 窗口外：用最新批次的 effective_to（Demo 走这条）
    return today
```

### 6.4 对既有 pytest 的影响评估

- `test_rate_repository.py`（117+ 行）：调的是 `query_air_weekly` / `query_air_surcharges`，不调 `infer_default_effective_on` → 不影响。
- `test_rate_matcher.py`（T-B5 32 条用例）：直接传 `effective_on=date(2026, 4, 23)` → 不走默认值路径 → 不影响。
- Orchestrator 层无直接 pytest（T-B10 是前后端联调）→ 不影响。
- **新增 pytest**：`test_infer_default_effective_on.py`，覆盖：
  - active 批次存在 + today 在窗口内 → 返回 today
  - active 批次存在 + today 在窗口外 → 返回 effective_to
  - 无 active 批次 → 返回 utcnow 今天
  - 多个 active 批次（跨 file_type）→ 只看 air（filter 生效）

### 6.5 业务效果验证

Demo 5/14 当天：

## 9. 开放问题（给监工决策）

### Q-ARCH-01 · `imported_rows` 是否含 skipped？

- 业务需求 §3 观察点 2："`imported_rows` 对 Apr 20 Air 文件应为 147"
- 但 §4.2 "Ocean LCL 不入库"时 skipped 另外统计
- Air 无 LCL → imported_rows=147 自然成立
- Ocean 若 parser 产 10 FCL + 5 LCL → imported_rows=10, skipped_rows=5
- **架构建议**：`imported_rows = 真落库条数`，`skipped_rows = LCL 跳过数`；imported_rows 不含 skipped
- 需监工确认是否与业务大师对齐

### Q-ARCH-02 · dry_run 要不要也算降级提示？

- 业务需求 §4.3 dry_run 要返回 `diff_summary` + `selected_rows`
- 没说要返回 "将降级 Y 个老批次"
- 前端 §6.5 话术 "试运行完成，将导入 X 条，**降级 Y 个历史批次**"
- **矛盾**：业务需求 §4.3 没要 superseded 计数，§6.5 要了
- **架构建议**：dry_run 也计算 `superseded_batch_ids`（只查不改），返回字段一致；UI 按业务 §6.5 展示
- 需监工拍板

### Q-ARCH-03 · R-01 推断策略冲突

- 业务大师在 §9.1 R-01 建议 (a) "改 effective_on 默认值为最新 active 批次的 effective_to"
- 架构实装偏进一步：窗口内用 today、窗口外用 effective_to
- **理由**：避免 5/14 激活 5/11-5/17 批次时误用 effective_to=5/17 （用 today 更准确）
- 需监工决定：严格按业务 (a)（永远用 effective_to）还是架构扩展（窗口感知）

### Q-ARCH-04 · Ocean LCL 跳过的 warning 文案

- 业务需求 §4.2 "LCL 暂不入库（v2.0）"
- 单条 record 跳过时要不要带 row_index / origin / dest 细节？
- **架构建议**：warnings 是列表但去重后只保一条"LCL 暂不入库（v2.0）: 跳过 N 条"；细节写 `skipped_rows_detail: list[dict]` 供调试
- 或：每条一个 warning，前端只展示前 3 条 + "共 N 条"
- 需监工选一个
