## 8. 不通过项 / 建议回炉项

### 8.1 P2 风险项（不阻塞 Demo）：LCL Lv.3 fallback 公式与真实数据不符

**位置**：`backend/app/services/step1_rates/adapters/ocean_ngb.py:465-474`（`_compute_lv_fallback`）

**问题**：函数对 LCL Lv.3 用 `factor = Decimal("1.2")` 公式（与 FCL 一致）；但真实 Excel 的 LCL Lv.3 行 AO/AP **是手输硬编码 20**，并非 ROUNDUP(Lv.1×1.2,-1)=10 的公式结果。

**证据**（独立验证脚本 `/tmp/check_lcl_formula.py`）：
```
row 84: B='Lv.1' AO=5 AP=5
row 85: B='Lv.2' AO='=ROUNDUP(AO84*1.1,-1)' AP='=ROUNDUP(AP84*1.1,-1)'   ← 公式
row 86: B='Lv.3' AO=20 AP=20                                              ← 硬编码！
row 87: B='Lv.1' AO=5 AP=5
row 88: B='Lv.2' AO='=ROUNDUP(AO87*1.1,-1)' AP='=ROUNDUP(AP87*1.1,-1)'
row 89: B='Lv.3' AO=20 AP=20
... （共 7 组 LCL，Lv.3 全部硬编码 20）
```

**当前为什么不踩雷**：真实文件 metadata.formula_fallback_count=0，说明 Lv.2 LCL 公式都有缓存值，data_only 一次拿对，根本不进 fallback 分支。所以**当前 Demo 不会暴露**。

**未来踩雷场景**：客户某次保存 Excel 没缓存公式结果（业务需求 §10 BR-NGB-6 列出的真实风险），LCL Lv.2 走 fallback 还行（实测 ROUNDUP(5×1.1,-1)=10 与原公式一致）；但 LCL Lv.3 走 fallback 会算成 ROUNDUP(5×1.2,-1)=10，**与真值 20 差 1 倍**。

**修复建议**：业务大师向楢崎确认 LCL Lv.3 的真实业务规则（推断是"档位级的对外报价档"，与 FCL 的 ROUNDUP 公式无关）。短期可在 `_compute_lv_fallback` 添加 `mode == "LCL" and level == "Lv.3"` 时打额外 warning"LCL Lv.3 fallback 与 Excel 实际硬编码值不一定一致"，避免 Demo 当天误导楢崎。

**不阻塞 Demo 的理由**：
- formula_fallback_count=0，不触发分支
- 业务需求 §3.5 末尾原文明示"fallback 仅作用于 R/S/T/AO/AP 主运费列"——架构本来就允许两侧不严格匹配
- 业务需求 §0 第 33 行的"Lv.3 = ROUNDUP(Lv.1×1.2,-1)"原本是基于 FCL 段 R3/R4 公式实证写的，业务侧本来也没断言 LCL 用同公式
