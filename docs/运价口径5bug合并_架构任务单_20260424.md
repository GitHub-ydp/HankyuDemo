#### 0.2.1 运价类型枚举 `RateType`（唯一权威定义）

在 `backend/app/schemas/freight_rate.py` 文件顶部（现有 import 之后）新增一个与持久层 enum 解耦的 **Schema 层枚举**：

```python
class RateType(str, enum.Enum):
    ocean_fcl = "ocean_fcl"      # FreightRate + ImportBatch.file_type=ocean
    ocean_ngb = "ocean_ngb"      # FreightRate + ImportBatch.file_type=ocean_ngb
    air_weekly = "air_weekly"    # AirFreightRate
    air_surcharge = "air_surcharge"  # AirSurcharge
    lcl = "lcl"                  # LclRate
```

**为什么新造一个枚举而不是复用 `ImportBatchFileType`（`backend/app/models/import_batch.py:12-15`）？**
现有 `ImportBatchFileType` 只有 `air / ocean / ocean_ngb` 三值，**没有** `air_surcharge` 和 `lcl` — 而 `air_freight_rates` 与 `air_surcharges` 是同一 `file_type=air` 下的两张子表，`lcl` 目前无批次主表枚举。本轮 RateList 必须拆 5 tab，复用三值枚举不够。

**前端 i18n key 命名规则（强制）**：
- `rates.tabs.{value}` = RateList tab 文案，例：`rates.tabs.ocean_fcl` / `rates.tabs.air_weekly`
- `compare.tabs.{value}` = RateCompare tab 文案（仅 ocean_fcl / ocean_ngb / air_weekly / lcl 四项，**不含 air_surcharge**）
- `rates.cols.{value}.{colKey}` = 按类型命名的列头，例：`rates.cols.air_weekly.day1`

注意 value 使用下划线 snake 形式（和后端枚举 value 一致），前端 i18n 显示值由翻译决定，避免出现"camelCase key 对 snake_case enum"的割裂。

#### 0.2.2 跨表查询抽象：**加一层按类型分派的 service 函数，不拆多个 service 文件**

本轮采用方案 **A：在 `freight_rate_service.py` 内加多态分派**，拒绝方案 B（拆 `ocean_fcl_service.py` / `air_weekly_service.py` / ... 5 个文件）。

理由（符合 CLAUDE.md "三份相似代码好过早抽象"）：
1. 每类查询逻辑 ≤ 50 行（简单 `db.query(Model).filter(...)`），抽 5 个文件会让 `freight_rate_service.py` 瘦成空壳，给 step2 `rate_repository.py`（见 `backend/app/services/step2_bidding/rate_repository.py:29`）下游产生"去哪找海运查询"的困惑
2. 列表页的 5 tab + 比价页的 4 tab 存在查询复用（Ocean 和 NGB 共用 FreightRate 表，仅 `ImportBatch.file_type` 过滤器不同），抽 5 文件后 Ocean/NGB 会冗余 80% 代码
3. 三个 if 分支 + 5 个独立函数 `_list_ocean_fcl / _list_ocean_ngb / _list_air_weekly / _list_air_surcharge / _list_lcl` 够用了，暴露一个总入口 `list_rates_by_type(db, rate_type: RateType, ...)` 让 API 层一致调用

**具体落到代码的函数签名（v1）**：

```python
# backend/app/services/freight_rate_service.py 新增：

def list_rates_by_type(
    db: Session,
    rate_type: RateType,
    *,
    # 海运族（ocean_fcl / ocean_ngb）用
    origin_port_id: int | None = None,
    destination_port_id: int | None = None,
    carrier_id: int | None = None,
    origin_keyword: str | None = None,
    destination_keyword: str | None = None,
    carrier_keyword: str | None = None,
    status: str | None = None,
    # 空运族（air_weekly / air_surcharge）用
    origin_text: str | None = None,
    destination_text: str | None = None,
    airline_code: str | None = None,
    # 通用
    page: int = 1,
    page_size: int = 20,
) -> tuple[list[Any], int]:
    """按类型分派查询。返回的 items 为对应 Model 实例列表，API 层再按 schema 序列化。"""
    if rate_type in (RateType.ocean_fcl, RateType.ocean_ngb):
        return _list_ocean_like(db, rate_type, origin_port_id=..., ..., page=page, page_size=page_size)
    elif rate_type == RateType.air_weekly:
        return _list_air_weekly(db, origin_text=origin_text, destination_text=destination_text,
                                 airline_code=airline_code, page=page, page_size=page_size)
    elif rate_type == RateType.air_surcharge:
        return _list_air_surcharge(db, airline_code=airline_code, page=page, page_size=page_size)
    elif rate_type == RateType.lcl:
        return _list_lcl(db, origin_port_id=origin_port_id, destination_port_id=destination_port_id,
                         page=page, page_size=page_size)
    raise ValueError(f"Unknown rate_type: {rate_type}")
```

**`_list_ocean_like` 内部 Ocean vs NGB 的区分**：

```python
# 关键：通过 import_batches.file_type 关联过滤
q = db.query(FreightRate).outerjoin(
    ImportBatch, FreightRate.batch_id == ImportBatch.batch_id
)
if rate_type == RateType.ocean_fcl:
    # Ocean：排除 NGB，兼容老数据（老数据 FreightRate 可能无 batch_id，视为 ocean）
    q = q.filter(
        or_(
            ImportBatch.file_type == ImportBatchFileType.ocean,
            FreightRate.batch_id.is_(None),  # 老导入链路，无批次 uuid，兜底为 ocean
        )
    )
elif rate_type == RateType.ocean_ngb:
    q = q.filter(ImportBatch.file_type == ImportBatchFileType.ocean_ngb)
```

**为什么老 FreightRate 行兜底为 Ocean 而不是抛错？**
看 `backend/app/models/freight_rate.py:96-101`：`batch_id` 是 `Mapped[uuid.UUID | None]`（**可空**）；老 `import_parsed_rates` 流程（`backend/app/api/v1/rates.py:204-219`）写入的 `freight_rates` 行没有 `batch_id`。若强制 NGB 必须 join 到 batch，会把老数据全丢进 Ocean tab；但把老数据都扔进 Ocean tab 反而是对的 — 因为那是老导入链路，业务上就应归到「海运 FCL」tab。

#### 0.2.5 airline_code 补齐策略（P1-4 专用）

采取 **"两条腿走路"**：

1. **前向修复**：改 adapter + mapper，让新导入的 Air 批次 `airline_code` 正确落表
2. **历史回填**：写一个一次性 SQL 脚本（不走 Alembic migration，因为没有 schema 变更），从 `service_desc` 用正则 `[A-Z]{2}` 提取航司码写回 `airline_code` 字段

**为什么不用 Alembic？**
- 没有 schema 变更（列本来就存在）
- Alembic 的 `upgrade/downgrade` 语义是 DDL，用它跑一次性 DML 反模式
- 一次性脚本更可控（可以 dry-run 统计命中率再正式跑）

**脚本位置**：`backend/scripts/backfill_airline_code_20260424.py`（新建 `backend/scripts/` 目录；项目根目录有 `scripts/` 但那是数据导入目录，避免混淆）

**脚本逻辑（非代码，伪规格）**：

```
对 air_freight_rates 表：
  WHERE airline_code IS NULL AND service_desc IS NOT NULL
  从 service_desc 提取 '[A-Z]{2}' (借用 air.py:37 的 _AIRLINE_CODE_RE)
  若提取到 >= 1 个：
    - 若恰好 1 个：airline_code = 该码
    - 若 >= 2 个：airline_code = "NH/OZ" 斜杠拼接（按出现顺序）
  若 0 个：保持 NULL，加 dry-run 统计里"无航司码"分桶

输出：
  - dry-run: 打印"命中 X 行 / 未命中 Y 行 / 跳过 Z 行"
  - 正式跑：UPDATE air_freight_rates SET airline_code = :code WHERE id = :id
```

**正则来源复用**：直接 import `AirAdapter._AIRLINE_CODE_RE`（`backend/app/services/step1_rates/adapters/air.py:37`），保证前后一致。

## 7. 风险清单

1. **`freight_rates` 老表向后兼容**：`list_rates_by_type` 的 ocean_fcl 分支用 `outerjoin(ImportBatch)` + `batch_id IS NULL 兜底到 ocean`；若写成 inner join 会吃掉老数据。测试时必须验证 ocean_fcl tab 总数 = FreightRate 总数 - NGB 批次关联的行数。

2. **`airline_code` 回填正则误匹配**：`[A-Z]{2}` 会捕到 "EU" / "US" / "CN" 等非航司码（业务需求附录的 `CZ/CK/CA` 格式语料中暂未见）。回填脚本 dry-run 时必须人工扫一遍 "would_update" 的样例输出；若出现异常地域码，脚本跳过该行保持 NULL。

3. **`_draft_batches` 内存 + `import_batches` DB 合并时的 duplicate key**：同一 batch_id（被激活的批次）两头都有；去重策略是**内存版本优先**（含 preview，能再次 diff）。若监工认为"已激活批次不应再次 diff 修改"，可改为 DB 版本优先——但本轮保留内存优先，Demo 场景下新激活批次刚刚产生，前端页面连续操作更顺。

4. **Schema 拆分对 Step2 `Step1RateRepository` 的副作用**：`backend/app/services/step2_bidding/rate_repository.py:29` 返回的是 `Step1RateRow`（不走 schemas 层），**与本轮 schema 新增无冲突**。Step2 下游只读 Step1 物理表（`AirFreightRate` / `AirSurcharge`），继续工作。唯一风险点：若 `to_air_freight_rate` 改动（T-AC-01）影响了 `airline_code` 存储格式为 "NH/OZ" 斜杠拼接，Step2 `rate_repository.py:69` 的 `airline_code.in_(["NH"])` 精确匹配会错过多航司行——**已记录为 Step2 已知 gap，本轮不动**。

5. **前端 i18n key 改名影响现有测试/截图**：业务需求要求新增大量 key（`rates.tabs.*` / `compare.tabs.*` / `rates.cols.*` / `compare.col.*`），不删任何旧 key，理论上截图回归无影响。但若楢崎在之前 Demo 已截图"老版 RateList 顶部无 tab"，新版截图会与之不符——**这是业务正向变更，不算 bug**。

6. **P0-1 `response_model` 动态化的 OpenAPI 精度下降**：API `/rates` 去掉 `response_model=ApiResponse[PaginatedData[FreightRateDetail]]` 静态注解。若监工后续接入自动生成前端 TS client 的工具链（如 openapi-typescript-codegen），生成精度下降。备选 1：拆 5 个子路由（见 §2 决策）；备选 2：`response_model` 写成 `Any`。本轮接受精度下降，因为前端 TS 类型手写维护在 `types/index.ts`。

7. **Air compare 排序锚点 price_day1 的合理性**：业务需求 §4.2 明确用 `price_day1`，但营业部门可能更关心"一周均价"。本轮先用 Day1；若彩排时楢崎提出异议，记录为 v1.1 待确认项，不现场改。

8. **`GET /rate-batches` 合并 DB 后的字段不一致**：DB 批次无 preview_rows / warnings / parse_records，前端表格渲染时这些列需要防御式 `|| []` / `|| '—'`。若前端没防御，切到 BatchesPanel 第二页看到 DB 批次会崩溃。测试必须覆盖"翻页到 DB 批次 + 点击查看详情"场景（后者合理返回 404，前端提示"该批次已持久化，无预览数据"）。

9. **T-RL-04 前端表格组件拆分颗粒度**：`<AirWeeklyTable>` / `<AirSurchargeTable>` / `<LclTable>` 是否在 RateList.tsx 内联定义还是独立文件？**决策：内联在 `RateList.tsx` 里**（避免新建 3 个小组件文件污染 `components/` 目录）；若未来其他页面（如 Dashboard）也需要 air weekly 表格再抽出。"三份相似代码"原则。

10. **T-RB-01 去重 key 字段类型**：`_draft_batches` 的 `batch_id` 是 str（`serialize_summary` 第 295 行 `"batch_id": draft.batch_id`），DB `ImportBatch.batch_id` 是 uuid.UUID。比较时必须 `str(row.batch_id) in memory_ids`，**不能** `row.batch_id in memory_ids` — 否则 UUID vs str 永不相等，去重失效。
