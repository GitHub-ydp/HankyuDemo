# 运价管理页面三类分离 业务需求 v1.0

- 日期：2026-04-24
- 作者：业务大师
- 出处：用户决策（正解 B），现状代码 `frontend/src/pages/RateList.tsx` + `backend/app/services/freight_rate_service.py:11`（`get_rates`）
- 相关前序：Step1 Air / NGB / Ocean 三解析器 04.21~04.23 已入库；04.24 ebc2573 闭环 5 已把 Dashboard 统计口径合并到 FreightRate + AirFreightRate 两张表

---

## 需求 1：运价管理列表页必须与 Dashboard 显示同一口径的运价全集

- **出处**：`backend/app/services/freight_rate_service.py:197-219` `get_rate_stats` 已把 FreightRate + AirFreightRate 两张表加总（ebc2573 闭环 5）；同文件 `:11` `get_rates` 只查 FreightRate。`frontend/src/pages/RateList.tsx:212-217` 调 `rateApi.list`，所以列表页仅能看到 FreightRate 表。
- **业务目标**：楢崎在 Demo 现场点完「Dashboard 显示运价 200 条」后，紧接着点「运价管理」菜单，列表页必须能看到同样 200 条的全集，不能出现「首页写 200、列表页只有 116」这种数字对不上的事故。
- **业务规则**：
  1. 当前系统里客观存在五类运价（来源于四张物理表 + Ocean/NGB 在同表内的业务区分）：
     - 海运 Ocean（`freight_rates` 表，`import_batch.file_type = ocean`）
     - 海运 NGB（`freight_rates` 表，`import_batch.file_type = ocean_ngb`）
     - 空运周报价 Air（`air_freight_rates` 表）
     - 空运附加费（`air_surcharges` 表）
     - 拼箱 LCL（`lcl_rates` 表）
  2. 「运价管理」列表必须能覆盖到这五类中已导入有数据的全部类别。
  3. Ocean 与 NGB 虽然共用一张物理表，但业务意义上要让楢崎在 UI 上分别看到，而不是混成一堆。
- **验收条件**：楢崎切到「运价管理」页、把五个 tab 依次点一遍，每个 tab 的「共 N 条」加起来等于 Dashboard 首页的「总运价」数字。
- **边界**：本轮只做「让数字对得上 + 列头对得上」；不做跨类型的合计/导出/比价。
- **业务风险**：如果 Demo 只展示 Ocean 116 条，楢崎会追问「Air 那 84 条去哪了」，直接暴露后台与前端口径不一致，整个运价中台的可信度受损。

## 需求 2：五类运价的业务语义必须在 UI 上表达清楚

### 2.1 海运 Ocean（FCL 普通航线）

- **出处**：`backend/app/models/freight_rate.py:40-113`，配合 `import_batch.file_type = ocean`
- **业务字段**（用户查列表时关心的）：航线（起运港 → 目的港）、承运商、航线代码、20GP/40GP/40HQ 三箱型运费、币种、有效期（`valid_from` → `valid_to`）、航程天数、费率等级（Lv.1/2/3）、状态（draft/active/expired）、来源。
- **查询维度**：POL、POD、承运商、状态、有效期。
- **用户看这个列表的目的**：Demo 里验证「常规 FCL 航线」这一类最主流业务的运价齐不齐，演示 Step2 PKG 自动填写时这是主要命中源。

### 2.2 海运 NGB（宁波 / 江南内河港 FCL）

- **出处**：同 `freight_rates` 表，`import_batch.file_type = ocean_ngb`；源文件为 `资料/2026.04.21/.../NGB 2026.4.16-4.30.xlsx`
- **业务字段**：与 Ocean 同张表同一套字段，但起运港集中在 CNNGB / CNZAP / CNZOS 等宁波/乍浦/舟山港口。
- **查询维度**：与 Ocean 相同。
- **用户看这个列表的目的**：NGB 是宁波分公司单独维护、半月更新节奏，业务上由独立团队出费率表，营业人员希望单独查看这批数据，确认半月更新是否按时入库。
- **边界说明**：材料未明 Ocean 和 NGB 在列头上是否要完全一致。本轮保守处理——字段相同，列头一致；仅通过 tab 切换区分。

### 2.3 空运周报价 Air

- **出处**：`backend/app/models/air_freight_rate.py`
- **业务字段**：起运地（`origin`，机场代码/城市字符串）、目的地（`destination`，字符串）、航司（`airline_code`）、服务描述、生效周（`effective_week_start` → `effective_week_end`）、7 列日价（`price_day1..7`）、币种。
- **查询维度**：起运地关键字、目的地关键字、航司、生效周。
- **用户看这个列表的目的**：空运是周更新周期，营业人员要一眼看出「这周的某条航线到底是周一价 XX、周四价 YY」，7 列日价必须在列表上展开、不能只给一个「平均价」。
- **字段差异的业务含义**：与海运不同，空运不分箱型而是按天波动，这是 Demo 时必须向楢崎强调的重点差异。

### 2.4 空运附加费

- **出处**：`backend/app/models/air_surcharge.py`
- **业务字段**：航司、起运区域（`from_region`）、区域（`area`）、目的地范围（`destination_scope`）、MYC 最低收费 + 每公斤费率、MSC 最低收费 + 每公斤费率、生效日期、币种。
- **查询维度**：航司、起运区域、生效日期。
- **用户看这个列表的目的**：核对 MYC/MSC 这两类燃油/安全附加费的基准价和阶梯，是报价时必须叠加的项；单独分出来是因为它的「起算单位」不是一票运价、是按 kg 阶梯。
- **字段差异的业务含义**：没有「航线」概念、没有具体的 POL/POD 两个机场码，只有区域性描述；列头不能硬套海运的「起运港 → 目的港」。

### 2.5 拼箱 LCL

- **出处**：`backend/app/models/lcl_rate.py`
- **业务字段**：起运港、目的港、每 CBM 运费、每 TON 运费、币种、LSS/EBS/CIC/AMS/ACI/ENS 几项附加费（以字符串存储）、开船日、中转港、航程文本、有效期。
- **查询维度**：POL、POD、有效期。
- **用户看这个列表的目的**：LCL 是散货拼箱，按体积/重量二选一计价（W/M），单位是 CBM 不是 TEU，列表必须展示「每 CBM 价 / 每 TON 价」两列，不能套 Ocean 的箱型列。
- **边界说明**：LCL 当前是否已导入实数据、有多少条，本轮业务需求以「结构已就绪」为前提；若实际为空则列表显示空态即可，不影响本轮验收。

## 需求 3：用户交互设计

### 3.1 tab 单选 / 多选

- **决策**：**单选**。
- **理由**：五类的列头字段不可兼容（海运看箱型、空运看日价、附加费看 MYC/MSC 阶梯、LCL 看 CBM/TON），多选时表头无法合理合并。单选能保证每次列表的语义是一致的、列头是明确的。

### 3.2 默认选中

- **决策**：默认选中「海运 Ocean」。
- **理由**：
  1. Ocean 是目前数据量最大（116 中的大多数）、Demo Step2 主要命中的类型；
  2. 与现状 `get_rates` 只查 FreightRate 的行为一致，不破坏既有 Demo 话术；
  3. 楢崎在 PPT 里的主场景也是 FCL 航线。

### 3.5 空态

- 某 tab 下无数据时，表格区居中显示三语文案：「暂无数据 / データなし / No data」，并补一行业务提示：「该类别尚未导入费率，可前往『运价导入』上传源文件」，三语对应：
  - zh: 该类别尚未导入费率，可前往「运价导入」上传源文件
  - ja: 当該種別はまだ取込されていません。「運賃取込」からアップロードしてください
  - en: No rates imported for this type yet. Upload from Rate Upload page.
- 空态**不**算 bug，属于合法状态。

### 3.6 分页

- 沿用现状每页 15 条（`PAGE_SIZE = 15`），无需改动。
- 切换 tab 时**必须重置到第 1 页**，避免「Air 只有 84 条却停在第 9 页」的空白尴尬。

### 3.7 排序

- 本轮**不做用户可点的排序**（列头不可点）。
- 默认排序：Ocean / NGB / LCL 按 `id desc`（最新导入优先）；Air 周价按 `effective_week_start desc`；Air 附加费按 `effective_date desc`。业务含义是「最新的在上面」。

### 3.8 筛选器与 tab 的联动

- POL / POD / 承运商 三个现状筛选框在切换到「Air 附加费」时**必须隐藏或禁用**，因为该类无航线、无 POL/POD 概念，硬留着会误导用户。
- 切换 tab 时，与当前类不兼容的筛选条件一律清空，避免「带着海运的 POD 去查空运」。
- 状态 chip（全部 / 有效 / 待确认 / 已过期）仅在 Ocean / NGB 下出现，其余三类无 `status` 字段，隐藏。

## 需求 4：不做范围（本轮明确不做）

1. **跨类型合计统计** — 列表页不显示「五类合计」横条；合计由 Dashboard 负责。
2. **跨类型导出** — 不提供「一键导出全部 200 条」；导出由各自导入批次管理。
3. **跨类型比价** — 不在列表页做 Ocean 与 Air 的对比（业务不可比）。
4. **可点列头排序** — 不做。
5. **列头字段自定义 / 列显隐** — 不做。
6. **Ocean 与 NGB 合并展示** — 明确拆两个 tab，不做「All Ocean」合并 tab。
7. **LCL 列表的写操作** — 本轮只看，不做状态切换（LCL 目前没有 status 字段）。
8. **Air 周价日价的可编辑** — 本轮只展示 7 列日价，不提供行内编辑。

## 验收清单（UI 可观察事实）

1. 进入「运价管理」页，页面顶部能看到一行 tab 组件，至少包含 5 个选项：海运 FCL / 海运 NGB / 空运周价 / 空运附加费 / 拼箱 LCL（三语随 i18n 切换正确）。
2. 默认进入页面时，tab 选中「海运 FCL」，且表格列头是「20' DC / 40' DC / 40' HC」三箱型。
3. 点击「空运周价」tab，列头切换为「Day1 … Day7」7 列日价，且 POL/POD 筛选框隐藏或禁用。
4. 点击「空运附加费」tab，列头切换为「MYC 最低/每 kg、MSC 最低/每 kg」，状态 chip 组消失。
5. 五个 tab 各自的「共 N 条」相加，等于 Dashboard 首页的「总运价」数字（当前环境下约 200，允许随导入批次变动）。
6. 切换 tab 后页码自动重置为第 1 页。
7. 某 tab 下无数据时，表格区显示空态提示文案并指引去「运价导入」页，且不抛错。
8. 切换系统语言到日文/英文，tab 文案与列头文案同步切换，无中文残留（三语 i18n 覆盖完整）。
