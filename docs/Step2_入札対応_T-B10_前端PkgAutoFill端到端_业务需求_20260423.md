## 1. 任务定位与 T-B10 v0.1 边界

- **出处**：架构任务单 §9 第 715-800 行（T-B10 原计划 7 个 endpoint）+ §17 第 1035 行（T-B10 依赖 T-B9；T-B9 是 service.py 5 步编排 + 状态机，后者未交付）。
- **业务目标**：把已交付的 T-B4 parse / T-B5 match / T-B7 fill / T-B8 identify **四条后端能力**拼成一个**端到端浏览器可演**的闭环，让 5/14 Demo 当天楢崎能在前端一次拖拽跑通「上传 → 识别 → 解析 → 填入 → 下载两版 xlsx」。
- **业务规则**（v0.1 相对 7-endpoint 原计划的取舍）：
  1. **v0.1 只做 7 个 endpoint 里的 2 个**：上传触发一次性自动填入、一次性 token 下载。
  2. **砍掉**的 5 个：列表查询、单条查询、按行 PATCH 改价、审核 submit、候选列表（这 5 个依赖 T-B2 持久化或 T-B9 状态机）。
  3. **服务端不留状态**：一次请求内完成 identify → parse → match → fill → 写盘 → 返回两个下载 token。后续请求无法"读回上一次的 bid"。
  4. 这是 **T-B10 v0.1**，v0.2+ 在 T-B2/T-B9 交付后补齐持久化与审核流。
- **验收条件**：Demo 现场楢崎按 `docs/Demo_兜底方案与演示话术_20260424.md` §1.3 第 56-57 行"Step2 自动填入 PVG 段 → 审核界面 → 下载"这段流程跑完；**v0.1 审核界面简化为"识别 + 解析 + 填入 结果的只读面板 + 两个下载按钮"**，不提供"改价 + 留痕"交互（改价交互由 T-B10 v0.2 在 T-B2/T-B9 交付后补）。
- **边界**：
  - 本轮**不**新增前端路由（沿用 `/pkg`），旧 PkgAutoFill.tsx 页面内部切换到新管线。
  - 本轮**不**删除旧 `pkg_parser.py / pkg_filler.py`（T-B11 的事）。
- **业务风险**：
  - Demo 话术 §3.1 第 124 行承诺"审核界面支持改价 + 留痕"。v0.1 没有改价交互，一旦楢崎现场要求改价，需走 Demo 兜底话术。**此风险必须在 Demo rehearsal 前由张东旭与演示人对齐**。
  - "无状态"导致同一份 xlsx 刷新页面即丢识别/填入结果；Demo 当天演示人必须一次性跑完再讲解，不能中途切页。

## 2. 业务目标（为什么做 T-B10 v0.1）

- **出处**：Step2 整体业务需求 §3 端到端流程 + Demo 兜底 §0（Demo 定位表第 19-25 行）。
- **业务目标（三条，缺一不可）**：
  1. **让楢崎看到端到端闭环**：上传一份空白 2-①.xlsx，**不做任何额外操作**，几秒后看到"识别 Customer A / 解析 5 段 21 行 / 填入 5 行 / 下载 cost 与 S/R 两版 xlsx"。Demo 话术 §0 第 21 行明确要求"操作流程顺畅度极高"。
  2. **让 Demo 话术里"AI 确实做了处理"这句话有证据**（§0 第 22 行）：UI 必须看得到"识别中→识别 Customer A（high 置信）/ 解析中→5 段 / 填入中→成功填 5 条 / 生成 cost 版 + S/R 版"的阶段性反馈（不要求真实的进度条精度，但必须有阶段切换）。
  3. **让楢崎看到"系统知道自己不会什么"**：上传 Customer B 的 LCL xlsx 时，前端必须**明确告知"当前仅支持 Customer A，此文件识别为 unknown：<原因>"**并**阻断**后续按钮，不得沉默失败或产出错误结果（Demo 话术 §1.3 第 58 行"演示支持多客户 — 拖 Customer B 进去看识别结果"）。
- **验收条件**：
  - Demo rehearsal 当天三个场景全绿：正向（Customer A）/ 拒绝（Customer B）/ 友好报错（损坏 xlsx）。
  - 张东旭用秒表计时：从点击上传到两个下载按钮出现，**<= 15 秒**（不是硬性 SLA，是 Demo 节奏硬底线；超过 30 秒必须排查）。
- **边界**：v0.1 不做 Demo 呈现顺序第 6 步"拖 Customer B 进去看结构识别" —— 本轮 Customer B 只看到"识别为 unknown"的拒绝提示，**不做结构识别展示**（那是 v2.0）。

## 3. 输入契约（上传 xlsx 的限制）

- **出处**：architect task §3.3 第 140-145 行（POST /bidding/pkg 的 multipart/form-data 原设计）+ T-B8 §需求 1-1（输入必须是 .xlsx 文件）。
- **业务目标**：把"什么文件能进系统"定死，避免 Demo 现场被拖一个 PDF/图片/100MB 大文件把浏览器卡死。
- **业务规则**：
  1. **格式白名单**：`.xlsx` 唯一允许。
     - `.xls`（老 Excel 97-2003）**拒绝**：T-B8 identifier 用 openpyxl，openpyxl 不支持 .xls。
     - `.xlsm`（带宏）**v0.1 暂不接受**：黄金样本 Customer A 是 .xlsx，不带宏；Nitori 才有 .xlsm（v2.0 范围，见 Demo 兜底 §2.3 P2-2）。
  2. **大小上限**：业务建议 **10 MB**（实证 2-①.xlsx = 20185 B ≈ 20 KB，三份样本全部 <= 20 KB；设 10 MB 给未来 Customer B 132 列留足余量）。超限前端直接拒绝，不走网络。
  3. **单文件**：v0.1 一次只能上传一个 xlsx；批量上传（多文件队列）留给 v0.2+。
  4. **上传动作**：沿用旧页面的"点击选择 + 拖拽"两种入口。
- **验收条件**：
  - 拖一个 .xls → 前端直接报错"仅支持 .xlsx 格式"，不发请求。
  - 拖一个 .xlsm → 前端报错"当前版本不支持 .xlsm（含宏文件），计划 v2.0 支持"，不发请求。
  - 拖一个 20MB 的 .xlsx → 前端报错"文件超过 10 MB 上限"，不发请求。
  - 拖一个 .pdf / .jpg → 前端报错"仅支持 .xlsx 格式"，不发请求。
- **边界**：不做病毒扫描 / 压缩炸弹检测（v0.1 信任上传者）。
- **业务风险**：openpyxl 对"名称为 .xlsx 但内容其实是 .xls 或损坏"的文件会抛 `BadZipFile / InvalidFileException`（identifier.py 已处理为 `WBOPEN_FAIL`）。**前端扩展名校验只是第一道防线，真正的兜底在 §6 失败路径 F1**。

## 4. 接口契约（业务语义，不是技术字段）

v0.1 暴露两个业务动作。以下只写"请求里业务上带什么 / 响应里业务上给什么"，字段名/HTTP 方法/路径完全由架构大师定。

### 4.1 动作 A：自动填入（一次性）

- **出处**：架构任务单 §9.1 第 717-750 行（原 POST /bidding/pkg 的 request/response 契约）。
- **业务语义**：前端交一份 xlsx，后端**同步返回**四块业务信息 + 两个下载凭证。
- **请求携带**：一份 xlsx 文件。**无其它参数**（v0.1 不接受"指定加价系数 / 指定期间 / 指定客户代码"等可选参数；加价系数固定 1.15，客户由 identifier 自动判，期间由 T-B4 parse 自动读）。
- **响应必须包含的 4 块业务信息**：
  1. **识别结果块**（来自 T-B8 IdentifierResult）：
     - 识别出的客户（`customer_a` 或 `unknown` 二选一）。
     - 命中了哪几个维度（来自 matched_dimensions，v1.0 只可能是 B/D 两维中的一或二）。
     - 置信度（`high` / `medium` / `low` 三档文本，**不给数字百分比**，因为 identifier v1.0 返回的是分档字符串；业务风险见 §8 风险 3）。
     - unknown 原因（`unmatched_reason`，只在 unknown 时非空；让审核者知道为什么被拒）。
     - 告警信息（warnings，例如 "WBOPEN_FAIL: BadZipFile" / "MULTI_SHEET: ..."）。
  2. **解析结果摘要**（来自 T-B4 ParsedPkg）：
     - 识别出的入札期间（例如 `"2026-01 ~ 2026-01"` 或 parse 给的原文字符串）。
     - Sheet 名（Customer A 必为 `見積りシート`，让楢崎目视核对）。
     - 段数（Customer A 黄金样本应为 5）。
     - 行数（Customer A 黄金样本应为 21，含 `記入例` 示范 2 行）。
     - parse 过程中的 warnings（例如"PVG 段未找到发地行"等；v1.0 黄金样本应为 0 条）。
     - **样例行（最多 5 条）**：用于 UI 预览的行级摘要（section 代码、目的地文本、cost_type）。不需要返回全部 21 行——Demo 不看明细，只看"解析到了这些地方"。
  3. **填入结果块**（来自 T-B7 FillReport）：
     - 计数三元组（filled_count / no_rate_count / skipped_count）。
     - 全局警告（FillReport.global_warnings；例如"加价系数使用默认值 1.15，待楢崎确认"）。
     - 每行状态摘要（row_idx + section_code + destination_code + status + cost_price + sell_price + markup_ratio + 置信度 + 命中的 Step1 批次号）—— 让审核者看到"哪行填了什么价，来自哪条周表"。
  4. **下载凭证块**：两个一次性 token（cost 版 + S/R 版），每个 token 能换回 xlsx。**禁止**直接在响应里暴露服务器绝对路径（T-B7 `cost_file_path / sr_file_path` 是服务端内部字段，不得透传给前端）。
- **响应文件名建议**（非强制，由架构大师最终定；给出业务偏好）：
  - cost 版下载名 = `cost_<原上传文件基名>_<bid_id>.xlsx`（T-B7 §需求 6 要求 `cost_` 前缀）。
  - S/R 版下载名 = `sr_<原上传文件基名>_<bid_id>.xlsx`。
  - `bid_id` 建议 `yyyymmdd_HHMMSS_<uuid4[:8]>`（20 字符，可读可排序、唯一，即使同秒多次上传也不冲突）。
- **识别为 unknown 的行为**：
  - 后端**不执行** parse / match / fill，直接返回"识别结果块 = unknown + unmatched_reason"，解析结果块 / 填入结果块 / 下载凭证块**全部为空**。
  - 前端据此进入阻断态（见 §5 UI 流程）。
- **验收条件**：一次 Customer A 上传调用，响应能让前端完整渲染 §5 描述的 5 个面板，不需要额外调用。
- **边界**：
  - 不支持异步模式（v0.1 同步返回；如果实测超过 15 秒会堵塞 UI，架构大师可改用流式或轮询，但业务接口语义不变）。
  - 不返回"原始 xlsx 的 base64"（前端不需要显示源文件预览）。
- **业务风险**：
  - 如果 match/fill 阶段抛异常，业务要求**降级返回"识别成功 + 解析成功 + 填入失败"**，而不是整个请求 500。（Demo 现场至少能看到"系统识别对了、解析对了，只是填入有问题"）—— 具体失败路径见 §6 F4。

### 4.2 动作 B：一次性 token 下载

- **出处**：架构任务单 §9.4 第 774-777 行（原 GET /bidding/{bid_id}/download?variant=cost|sr）。
- **业务语义**：前端拿动作 A 返回的 token，换回一份 xlsx。
- **请求携带**：一个 token（string）。
- **响应**：一份 xlsx 二进制流 + 合理的下载文件名（浏览器"另存为"时显示 §4.1 中定义的命名）。
- **Token 生命周期**（业务建议）：
  - 有效期 **1 小时**（60 分钟）：Demo 现场楢崎可能边看边讨论，1 小时够用。
  - 下载一次后 token 失效（一次性）：避免 Demo 视频被分享后外部人员反复下载（虽然 v0.1 数据是黄金样本无敏感价格，仍建议保留此约束，为 v0.2 真实费率准备）。
  - 过期后返回"凭证已失效，请重新上传"的可读错误。
- **验收条件**：
  - 点击 cost 下载按钮 → 浏览器弹出"另存为"，默认文件名 `cost_2-①_20260424_HHMMSS_xxxxxxxx.xlsx`。
  - 连续点第二次 → 前端给"已下载过，请重新上传或刷新"的提示（不崩页）。
- **边界**：
  - Token 无 JWT / 签名要求，内部用 uuid 即可（v0.1 不涉及跨域共享）。
  - 不支持断点续传 / 分片下载（xlsx 极小）。
- **业务风险**：
  - 服务端临时文件清理策略：**业务要求 1 小时后清理**，和 token 有效期一致。清理不及时不是 Demo 级问题，但上线前必须解决（v0.2 要求）。

## 5. UI 流程与状态机

- **出处**：旧 PkgAutoFill.tsx 三步式 StepBar（upload → review → download）+ Demo 兜底 §1.3 呈现顺序 + T-B7 §9.1 加价系数提示条。
- **业务目标**：让 Demo 演示人按顺序指屏讲解"现在在做什么"，每个阶段都有视觉锚点。
- **v0.1 状态机（4 个状态 + 1 个异常态）**：

```
[空闲] ──上传──▶ [识别+解析+填入 同步进行中] ──成功──▶ [完成] ──刷新──▶ [空闲]
                                                └──unknown──▶ [拒绝] ──重新上传──▶ [空闲]
                                                └──异常─────▶ [失败] ──重试/重传──▶ [空闲]
```

**v0.1 砍掉了旧页面的"review 中间态"**（旧流程 upload → 看 parse 预览 → 点击 fill → 看结果）。原因：
- 后端新管线是"一次性"的（动作 A 一次返回识别+解析+填入），不再有独立的 "parse 只预览不填" 模式。
- Demo 话术 §1.3 要求节奏紧凑，中间"点击 fill"按钮是多余一步。
- 代价：用户没机会"先看 parse 对不对再决定要不要 fill"—— 但 parse 本身是纯读不改源，失败也不会污染文件，所以一次性跑通更符合 Demo 诉求。

### 5.1 各状态的 UX 描述（业务层）

**状态 1 · 空闲 / 上传区**
- 页面主区：大号拖拽框 + "选择文件" 按钮 + 格式/大小限制文字提示。
- 文案：三语切换（参照旧 PkgAutoFill，保留 `pkg.uploadDragText` / `pkg.uploadHint` 等 key 的语义，仅改文本配合新管线）。
- 步骤条：高亮"第 1 步 · 上传"。

**状态 2 · 进行中**
- 上传进度条（0-100%，由浏览器 XHR upload 事件驱动）。
- 上传完成后切换为"识别中... / 解析中... / 填入中..."文案，允许不精确（Demo 现场视觉层面需要"AI 在工作"的感觉，精度由架构大师决定是否真实拆段）。
- 步骤条：高亮"第 2 步 · 处理"。

**状态 3 · 完成（识别为 customer_a）**
- 自顶向下四个面板（业务要求的呈现顺序）：
  1. **识别结果卡片**（最显眼，顶部）：
     - 大字显示"客户：Customer A（Mimaki 样式）"（Customer A 的业务落地名日后再由楢崎定；v0.1 显示 `Customer A`）。
     - 置信度徽章（`high` 绿色 / `medium` 黄色 / `low` 红色；v0.1 Customer A 黄金样本应恒为 `high`）。
     - 命中维度（chip 形式，例如 `sheet_name 匹配` + `表头结构 匹配`）。
     - 三语话术提示（Demo 话术 §3.1 第 124 行"自动识别低置信要求营业审核"风格）。
  2. **解析结果面板**：
     - 一行摘要文案：`入札期间 2026-01 · Sheet 見積りシート · 5 段 21 行`。
     - 样例行列表（最多 5 条，展示段代码/目的地/类型），折叠可展开。
     - parse warnings（v1.0 黄金样本应为 0 条；若有，红色小字显示在面板右侧）。
  3. **填入结果面板**：
     - 三个计数块：filled / no_rate / skipped（沿用旧页面的 stat-tile 风格）。
     - **加价系数提示条（硬约束）**：面板底部灰色小字 —— `加价系数：1.15（默认值，待楢崎确认）`。三语必须全翻译（zh/ja/en）。出处 T-B7 §9.1 第 277-278 行。
     - 每行状态摘要表格（行号/段/目的地/status/cost/sell/来自批次），折叠可展开。**Demo 展开讲解"这行为什么填 45"时指给楢崎看的证据就在这里**。
  4. **下载面板**：
     - 两个并排的大按钮：`下载成本价版（社内決裁用）` / `下载销售价版（客户送付用）`（三语文案见 T-B7 §需求 6-3 第 205-206 行）。
     - 按钮下方灰色小字标注"凭证将在 1 小时后失效 / 每个文件仅可下载一次"（v0.1 的诚实标注）。
     - "重新上传" 按钮（回到状态 1）。
- 步骤条：高亮"第 3 步 · 完成"。

**状态 4 · 拒绝（识别为 unknown）**
- 顶部**红色/橙色**醒目警告区：
  - 标题：`当前版本仅支持 Customer A`（日文：`現バージョンは Customer A のみ対応`；英文：`This version only supports Customer A`）。
  - 副文本显示 `unmatched_reason`（例如 `sheet 名非 '見積りシート'：发现 ['2025 LCL RATE(1001-0331)']`）。
  - warnings 列出（可折叠，主要给技术排障用，不是 Demo 重点）。
- **所有下游按钮禁用 / 不显示**：没有下载按钮、没有"强制继续"按钮。
- 只有一个"重新上传"按钮回到状态 1。
- 步骤条：停留在"第 2 步"，不点亮"第 3 步"。
- **业务硬要求**：不允许"强制把 unknown 当 Customer A 处理"的任何入口（出处 T-B8 业务需求 §需求 1 第 46 行"禁止返回 customer_b/e/nitori"；Demo 兜底 §1.3 第 58 行"看识别结果"明确 Customer B 只演识别不演填入）。

**状态 5 · 失败**（见 §6 F1-F4 各失败路径）
- 红色错误横幅 + 具体错误文案 + "重新上传"按钮。

### 5.2 三语 i18n 清单（本轮新增 / 修改的 key 集合）

业务层列出必须有三语的**文案分组**；具体 i18n key 命名由开发大师定，必须覆盖下列分组：

1. 上传区：拖拽提示 / 格式限制 / 大小限制 / 上传中。
2. 识别结果卡片：标题 / 置信度三档 / 命中维度前缀 / 拒绝标题 / 拒绝副文本（参数化 reason）。
3. 解析摘要：期间 / sheet / 段数 / 行数 / warnings 小标题 / 样例行展开按钮。
4. 填入结果：filled / no_rate / skipped 三个计数标签 / 加价系数提示条 / 行级表头（行号/段/目的地/状态/成本/销售/来源批次）/ 每种 RowStatus 的显示名（8 种：filled / no_rate / already_filled / example / non_local_leg / local_delivery_manual / constraint_block / overridden）。
5. 下载面板：cost 按钮 / sr 按钮 / 凭证说明 / 失效提示 / 重新上传 / "仅下载一次"提示。
6. 失败态：§6 F1-F5 的 5 条错误文案各需三语版本。

- **验收**：三语切换后上述 6 类文案全部有翻译，**不得出现英文或中文"漏翻译"** 的兜底回退（检测方法：把 key 列清单交给张东旭人工通读）。

## 6. 失败路径与文案

每条失败路径必须在前端有**明确、友好、不吓人**的中文/日文/英文错误文案，**不得让浏览器崩页 / 白屏 / 只显示 HTTP 500**。

| 代号 | 场景 | 后端信号 | 前端文案（中文，日英等价翻译） | 用户下一步 |
|---|---|---|---|---|
| F1 | 上传损坏 xlsx（如把 .xls 改名为 .xlsx） | identifier 返回 WBOPEN_FAIL warning | `文件无法打开，可能已损坏或不是有效的 xlsx 文件。请确认后重新上传。` | 重新上传 |
| F2 | 上传 Customer B/E/Nitori 的 xlsx | identifier 返回 matched_customer=unknown + unmatched_reason 非空 | `当前版本仅支持 Customer A。识别结果：<reason>` | 重新上传 Customer A 的文件 |
| F3 | T-B4 parse 抛异常（例如 sheet 为空 / 表头行找不到） | 后端捕获异常，降级响应 | `文件识别为 Customer A，但解析失败：<异常摘要>。请联系技术支持或上传样本文件。` | 重新上传 / 联系支持 |
| F4 | T-B5 match 0 行 / T-B7 fill 抛异常 | 后端捕获异常，降级响应，返回识别+解析成功但填入失败 | `解析成功，但自动填入失败：<异常摘要>。您仍可下载未填入的源文件（如有需要）。` | 本轮不提供"下载未填入源文件"兜底（v0.1 直接报错）；用户只能重新上传 |
| F5 | token 过期 / 已用 | 后端 410 Gone | `下载凭证已失效（超过 1 小时或已下载过）。请重新上传文件。` | 重新上传 |
| F6 | 文件大小超 10 MB | 前端本地校验不发请求 | `文件大小超过 10 MB 限制。请压缩或拆分后重试。` | 重新选择文件 |
| F7 | 文件扩展名非 .xlsx | 前端本地校验不发请求 | `仅支持 .xlsx 格式。.xls / .xlsm / PDF / 图片等不被支持。` | 重新选择文件 |
| F8 | 网络异常 / 请求超时 | fetch 失败 | `网络异常，请检查网络后重试。` | 重试 |

**业务风险**：
- 开发图省事把所有后端异常都挂到 F3/F4 的"异常摘要"直接显示栈——**业务不接受**。栈内容必须归类到上面 8 个代号之一，或在界面显示"系统异常，请联系技术支持"并在浏览器 console 打完整信息供排障。**Demo 现场不得出现 Python traceback / Pydantic error 字样**。
- F4 的业务定位：这是最可能发生的 Demo 翻车点（match/fill 对数据敏感，真实环境若 Step1 费率库为空会直接 0 行）。Demo rehearsal 必须把 Step1 费率预先导入好，避免现场命中 F4。

## 8. 风险与开放问题

### 8.1 风险 1：v0.1 无状态导致的"刷新即丢"（中）
- **事实**：Demo 演示人如果不小心刷新页面或切走再回来，所有结果丢失，必须重跑。
- **缓解**：Demo rehearsal 时演示人被反复告知"不要刷新"；前端可加 `beforeunload` 二次确认（业务层不强制，架构大师视工作量决定）。
- **v0.2 解法**：T-B2 迁移后 bid_id 可 GET 读回。

### 8.2 风险 2：加价系数 1.15 与 Demo 话术 "+10%" 不一致（高）
- **事实**：
  - T-B7 代码固定 1.15。
  - Demo 兜底 §2.1 P0-1 第 74 行话术写 "+10%"。
  - T-B7 §9.1 又写 "1.15（默认值，待楢崎确认）"。
- **业务影响**：前端显示"1.15"、Demo 演示人口述"10%"、Nakamura 心算用"1.09"—— 三个数字打架，现场楢崎质疑。
- **Q-T-B10-01（必问监工 / 赵总 / 张东旭）**：前端提示条的文案**取哪个值**？业务建议统一用 T-B7 代码的 1.15（因为是真实计算值），Demo 话术统一改口为 "+15%"，并保留"默认值待楢崎确认"话术。**本需求默认按 1.15 实现**，等张东旭确认后再改。

### 8.3 风险 3：IdentifierResult.confidence 是字符串而非数字（中）
- **事实**：旧 PkgAutoFill 里的 `confidence` 是 0-1 的 float（展示 `Math.round(confidence * 100)%`）；T-B8 identifier 返回的是 `"high" / "medium" / "low"` 字符串分档。
- **业务影响**：UI 显示方式必须切换。业务偏好：**显示文字分档 + 彩色徽章**，不显示百分比。理由：identifier v1.0 的分档来自"维度 B/D 命中几个"的离散逻辑（high=2 维命中，medium=1 维命中），强行转百分比是误导。
- **缓解**：§5.1 状态 3 的识别卡片已明确"不给数字百分比"。
- **保留问**：**Q-T-B10-02（问张东旭）**：matcher 阶段的 PerRowReport.confidence 是 0-1 float（和旧 UI 一致），行级表格的"置信度"列是否继续展示百分比？业务建议**是**（这是行级匹配的真实置信度，和 identifier 的全局分档不同含义）。

### 8.4 风险 4：Customer B 拒绝文案的日文措辞（低）
- **事实**：§5.1 状态 4 的日文文案 `現バージョンは Customer A のみ対応` 是业务大师草拟，未经楢崎本人审阅。
- **缓解**：**Q-T-B10-03（可问可不问）**：日文"当前仅支持 Customer A" 的客户友好措辞偏好？v0.1 按现草拟版实施。

### 8.5 风险 5：warnings 全都往 UI 推会显得系统有问题（中）
- **事实**：identifier / parse / fill 都可能产生 warnings。例如 identifier 即使成功也可能带 MULTI_SHEET 告警；fill 会有 global_warnings 说"加价系数使用默认值"。Demo 现场楢崎看到满屏黄色警告，观感差。
- **业务决策**：
  - 识别/解析 的 warnings 在正常路径 **默认折叠**（只显示数量徽章 `2 warnings`，点击展开）。
  - 填入的 global_warnings 中**仅"加价系数"那条**提升为一等公民（§5.1 状态 3 的加价提示条），其余 warnings 与上面一样默认折叠。
  - 对应英文界面同等处理。

### 8.6 风险 6：V4 "用 Excel Desktop 打开不报损坏"属人工抽检（中）
- **事实**：T-B7 测试报告已将此点标为"监工人工抽检"，无法自动化。
- **缓解**：**Demo 前 checklist 必须包含此条**。测试大师在浏览器跑完 V1 后，开发大师本机用 Excel for Mac + Windows Excel（若可用）各打开一次两个 xlsx，不报"文件已修复"弹窗。

### 8.7 风险 7：下载按钮文案的 S/R 术语 vs 销售价术语（低）
- **事实**：T-B7 §需求 6 已定日文 `S/R 版ダウンロード（顧客送付用）` 为默认；中文是"销售价版"。业务上两个术语指同一事物（S/R = Sell Rate = 销售价）。
- **缓解**：v0.1 按 T-B7 §需求 6 第 205-206 行给的三语文案实施。Q-T-B7-03 合并到本轮一起问。

## 10. 5 条最关键业务决策（落在本需求上）

1. **v0.1 只做 2 个 endpoint，无状态**。原计划的列表/查询/改价/审核/候选 5 个 endpoint 推迟到 v0.2（等 T-B2 / T-B9）。
2. **unknown 立刻阻断，禁止强制继续**。Customer B 拖进来只看到拒绝提示，没有任何"继续"入口。
3. **加价系数提示条是 UI 硬约束**。面板底部灰色小字"1.15（默认值，待楢崎确认）"，三语必须都有。口径与 Demo 话术的 "+10%" 冲突问题记为 Q-T-B10-01 交张东旭/监工决策，**默认按 1.15 实现**。
4. **失败路径全归口到 F1-F8 八个代号**，禁止 Demo 现场出现 Python traceback / Pydantic error 字样。F4（match/fill 失败）是 Demo 最可能翻车点，rehearsal 必须把 Step1 费率预导入。
5. **下载走一次性 token + 1 小时有效期**，不暴露服务器绝对路径。文件名规范 `cost_<基名>_<bid_id>.xlsx` / `sr_<基名>_<bid_id>.xlsx`，`bid_id = yyyymmdd_HHMMSS_<uuid4[:8]>`。
