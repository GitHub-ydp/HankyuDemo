## 0. 本轮范围定义

### 0.1 用户目标

> 营业人员把运价数据（任何来源）丢进系统，系统就应该想办法让它变成一个"批次草稿"，进而走差异对比、激活、串到 Step2。

### 0.2 本轮要做到的形态（口径 A）

- Excel 上传：**三段式决策树**（固定 adapter → AI 兜底 → 不支持提示），固定命中和 AI 兜底**都真正入库**为批次草稿
- 邮件 / 微信图片 / 粘贴文本 三个来源：**沿用现有 `ai_parse.py` 能力真跑**，不做任何退化或占位
- 所有来源产出的批次草稿，**走同一套**批次列表 / 预览 / diff / 激活流程
- 两个原本分离的前端页面（`/batches` + `/upload`）合成**一个入口**：主入口 `/upload`；`/batches` 从左侧菜单移除，路由保留重定向到 `/upload`

### 0.3 Demo 当天承诺范围

- 剧本仍然以 Excel 三份固定模板（Ocean / Ocean-NGB / Air）为主流程演示
- 邮件/图片/文本三 tab **不主动讲**。楢崎追问则按需求 6 话术回答；必要时张东旭**可以**当场演一次（因为真跑真入库）
- Customer B / E / Nitori 本轮**不留 hook**，完全锁 Customer A

## 需求 1：页面入口合并到 `/upload`

- **出处**：
  - 现状：`frontend/src/App.tsx` 第 59-60 行，`/batches` 和 `/upload` 是两个独立路由
  - 现状：`frontend/src/components/Layout/index.tsx` 第 20-21 行，侧边栏菜单项"运价批次"(`/batches`) 和"运价导入"(`/upload`) 并列存在
  - 现状：`frontend/src/pages/Dashboard.tsx` 第 173 行，"新建导入"快捷按钮目标地址是 `/upload`
  - 用户决策（2026-04-24）："`/batches` 页面不是隐藏了吗？我建议都在 `/upload`，前端样式没必要就不变化"
- **业务目标**：
  - 营业在日常工作中，"看有哪些批次 / 看某批次差异 / 把新一批运价传进来"是**同一件事的三个动作**。当前菜单把"导入"和"批次"分两个菜单项，营业每次半月更新费率要点两次菜单，还要在两个页面间切换，认知割裂。
  - Dashboard 快捷按钮已经把用户导向 `/upload`。本轮**顺势把 `/upload` 做成主入口**，把 `/batches` 的能力全部搬进来。
- **业务规则**：
  1. **主入口锁定为 `/upload`**。左侧菜单"运价导入"条目保留（`menu.upload`），展示合并后的完整页面
  2. **左侧菜单"运价批次"(`/batches`) 这一行删除**（即 `Layout/index.tsx:20` 的 NavEntry 去掉）
  3. `/batches` 路由**保留可访问**（浏览器历史 / 旧书签 / 旧文档截图都可能点到），行为是**重定向到 `/upload`**（前端 Navigate 即可）
  4. 合并后 `/upload` 页面必须包含下列能力，**缺一不可**：
     - 现有 4-tab 上传交互（Excel / 邮件 / 微信图片 / 粘贴文本）—— **保留同页**、**不改**
     - 批次列表（从 `RateBatches.tsx` 搬过来）
     - 批次激活动作
     - 差异（diff）对比面板
     - 原格式回填下载入口
  5. **前端样式最小改动**：批次列表、diff 面板、激活按钮等从 `RateBatches` 搬进来时**直接沿用它现有的视觉**，不做任何"统一视觉语言 / 重新设计"的尝试。用户原话："前端样式没必要就不变化"
  6. 上传完成后，**自动定位到新建的批次**（滚动到列表中这一行并高亮 / 自动展开详情），营业不需要手动切换或刷新
- **验收条件**：
  1. 楢崎从左侧菜单只点一次"运价导入" → 同屏能看到：4-tab 上传区 + 批次列表 + 单个批次详情的入口
  2. 左侧菜单里**没有**"运价批次"这一项
  3. 浏览器里直接敲 `/batches` → 无感跳到 `/upload`，不报 404
  4. 上传一个新 Excel 成功后，**不用手动回列表刷新**就能看到新批次并能立刻点进去看 diff / 激活
  5. 批次列表、diff 面板、激活按钮的视觉与合并前 `/batches` 页面**一致**（对比截图肉眼无明显改动）
- **边界**：
  - 不改后端 `/rate-batches/*` 系列请求契约，只是前端调用点搬到同一个页面
  - 不改批次详情面板（预览 / diff / 激活）的内部布局
  - 不做"统一设计语言 / 组件规范化"一类视觉改动
- **业务风险**：
  - 如果架构大师为了"美观"顺手把两块（上传 tab + 批次列表）的视觉对齐做了重构，违反用户"样式没必要就不变化"的指示，Demo 前出视觉 bug 的风险陡增
  - 如果 `/batches` 重定向只做了路由层但菜单还留着那一项，演员或楢崎点菜单后会看到跳转一下，观感差。**菜单项必须删掉**
  - 如果"上传后自动定位新批次"没做好，楢崎会问"传完了去哪看"，合并的价值损失一半

## 需求 2：Excel 三段式决策树 + 其它三 tab 真跑不退化

### 2.1 Excel 通道（三段式决策树）

- **出处**：
  - 用户决策（2026-04-24）："导入 Excel 文件的时候，先判断是不是固定模板，如果不是，就 AI 提取运价信息，如果再提取不到，就提示用户该文件没有相关的运价信息！"
  - 现状：`backend/app/services/step1_rates/registry.py` 固定 adapter 注册表，Ocean / Ocean-NGB / Air 三份模板命中逻辑已上线
  - 现状：`ai_parse.py` 已具备对非结构化内容做运价提取的能力（邮件 / 图片 / 文本三 tab 当前就在用它）
- **业务目标**：营业把任意一份 Excel 丢进来，系统应当**尽最大努力**把它变成批次草稿；只有**确实提取不到任何运价信息**时，才诚实地告诉用户"这份文件没有运价数据"。不让用户在"为什么不认识我的文件"上做无意义的排查。
- **业务规则（本轮核心，架构大师重点读）**：

  **第 1 段 — 固定 adapter 命中**
  - 营业拖入 `.xlsx / .xls`，后端先尝试用 registry 里的 Ocean / Ocean-NGB / Air 三份模板命中
  - 命中 → 进入现有批次草稿流程（预览 → diff → 激活），行为与今天 `/rate-batches/upload` **完全一致**
  - 未命中 → **自动进入第 2 段**，**前端不提示用户"没命中"**，对营业是无感降级
  - 【业务硬约束】本轮**不改**固定 adapter 的识别规则、**不新增**模板

  **第 2 段 — AI 兜底解析（自动进入，对用户无感）**
  - 触发条件：第 1 段未命中任何固定 adapter
  - 行为：后端**调用 `ai_parse.py` 对 Excel 内容做运价提取**
  - AI 提取到了运价数据 → 进入批次草稿流程（与第 1 段一致），批次记录中标记"来源：AI 解析"
  - AI 提取不到 → **进入第 3 段**
  - 【业务硬约束】整个第 2 段对用户**只看到一个 loading**，不单独弹"已降级到 AI"之类的中间态提示；避免楢崎现场问"为什么要 AI，你们的模板识别不行吗"
  - 【业务硬约束】AI 解析产出的批次草稿，必须能走同一套 diff 面板（结构对齐现有固定 adapter 产出）

  **第 3 段 — 确认无运价信息，明确提示**
  - 触发条件：固定 adapter 没命中 + AI 也提不到运价
  - 行为：**不创建空批次**、**不往批次列表插占位行**
  - 前端提示文案推荐（三语必须）：
    > **中文**："该文件未识别到运价信息，请确认文件内容或改用固定模板（Ocean / Ocean-NGB / Air）。"
    > **日文**：「このファイルから運賃情報は検出されませんでした。内容をご確認いただくか、固定テンプレート（Ocean / Ocean-NGB / Air）をご利用ください。」
    > **英文**："No rate information detected in this file. Please verify the content or use a fixed template (Ocean / Ocean-NGB / Air)."

### 2.2 邮件 / 微信图片 / 粘贴文本 三 tab

- **出处**：
  - 现状：`frontend/src/pages/RateUpload.tsx` 四个 tab 中，后三个（inbox / email / wechat / paste）已对接 `aiParseApi`
  - 用户决策（2026-04-24）："D-1=c（真跑真入库，不退化）"
- **业务目标**：保留并沿用已跑通的 AI 解析能力。Demo 剧本不主动演，但楢崎追问或要求演示时**可以真跑**给他看。
- **业务规则**：
  1. 这三个 tab **本轮不改**。保持当前 `ai_parse.py` + `_parse_cache` + `/ai/confirm` 直写 `freight_rates` 表的调用链路不动
  2. **本轮已知 gap**：架构大师读码发现这三 tab 直写 `freight_rates`，**不经过** `_draft_batches`（批次列表数据源），因此产出的数据**不会**出现在批次列表里、也不会激活/diff。v2.0 版本业务需求曾写"走同一条后续路径"是字面失准，这一项正式作废。**2026-04-24 用户拍板：接受现状不打通**（打通需改 `/ai/confirm` 产出 Draft 链路，工程量 ≥ 1 天，Demo 前动 AI 链路风险过高）。打通任务移入 Demo 后技术债清单
  3. Demo 剧本**不主动**切到这三个 tab；如楢崎当场追问 / 要求演示，可真跑（D-1=c）
- **验收条件**（2.1 + 2.2 合并）：
  1. 营业拖入 `资料/2026.04.21/RE_ 今後の進め方に関するご提案/` 下三份任一 → 第 1 段命中 → 产出批次草稿 → 与 Demo 彩排脚本行为一致
  2. 营业拖入**同目录外的任一合法运价 Excel**（举例：一份没注册过模板但内容是航线-费率表的 Excel）→ 第 2 段 AI 接住 → 产出批次草稿，标记来源为 AI
  3. 营业拖入 Customer A 的 PKG Excel（里面没有运价，是报价需求单）→ 第 2 段 AI 找不到 → 第 3 段提示"未识别到运价信息"
  4. 营业拖入 `.pdf / .docx` 等扩展名 → 前端在选择文件时就拦住（"当前 Excel 通道仅支持 .xlsx / .xls / .csv"）；**不**进入后端流程
  5. 营业切换到"邮件 / 图片 / 文本" tab → 能真实使用 AI 解析，结果走现有 `/ai/confirm` 直写 `freight_rates`（**不进批次列表**，已知 gap）
- **边界**：
  - 不做固定 adapter 新模板添加
  - 不做邮件/图片/文本 tab 的 UI 改版
  - 不做 AI 置信度在前端的可视化（不是 Demo 范围）
- **业务风险**：
  1. 第 2 段"对用户无感降级"需要确保**只有一个 loading 态**，别出现"固定 adapter 解析失败"之类的中间报错闪现。否则楢崎看到"模板没识别出来"的错误消息会追问
  2. 第 3 段的提示措辞必须是"该文件没有运价信息"而不是"解析失败"。前者是对文件的判断，后者让用户怀疑系统能力
  3. Demo 剧本**不主动**切到邮件/图片/文本 tab，但这些 tab **真能跑**——如果楢崎当场要求演一下，张东旭需要预先在彩排里走一遍，别现场翻车

## 需求 3：Demo 当天脚本（从 `/upload` 演起）

**这一段是给 Demo 演员（张东旭）和架构大师用的"行为对账表"**。架构实现完成后，对这条路径逐步走通就算过。

