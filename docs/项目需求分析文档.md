# 阪急阪神 —— 入札（招标）业务效率化项目需求分析

## 一、项目背景

**客户公司**：阪急阪神（Hankyu Hanshin）—— 日本货运代理（Forwarder）公司

**核心诉求**：利用 AI 技术提升招标（入札）业务的效率，减少人工操作，降低人力成本，提高投标成功率。

**项目语言**：文档包含日文、中文、英文三个版本，面向国际化团队协作。

**关键对接人**：

## 二、业务现状与痛点分析

### 2.1 当前业务流程（现行投标响应流程）

1. 客户（新規/既存）定期发送 **投标包（PKG）**，频率为3个月/半年/1年
2. 投标包格式由客户指定，基本固定，内容包含：航线、海运/空运运费、有效期间、港杂费
3. 营业担当确认内容后，向各地购买担当发送邮件请求报价
4. 购买担当确认当地运输费用（有时也通过 WeChat 等渠道获取）
5. 购买担当通过邮件回复营业担当
6. 营业担当将费率手动填入客户指定格式的投标文件，完成投标

### 2.2 运价获取子流程（购买部门 — 核心瓶颈）

> 来源：福山提供的《運賃表ワークフローについて》PPT

购买部门是运价数据的核心枢纽，其日常工作流程如下：

1. **供应商数量**：约 **15~20 家**船司(Shipping Line)、联运商(Co-loader)、代理店(Agent)
2. **获取渠道**：邮件（Excel/PDF 附件）、WeChat 消息、QQ 群、邮件正文纯文本
3. **更新频率**：约每 **2 周**更新一次运价
4. **人工操作**：收到运价后，逐条**手动录入** Excel 运价管理表
5. **痛点**：供应商多、渠道杂、格式不统一，手工录入耗时且易出错

### 2.3 福山明确的 AI 需求（PPT Slide 3）

> **原文**：
> - メールやWeChatで入手している運賃を、そのままAIに読み込ませることにより、手打ち入力の手間を省くことはできないか
> - AIがレーン（例：NHAVA SHEVA）や、運賃等の文言から必要情報を取り、表を埋めることは可能であるか

**翻译**：
1. 能否让 AI 直接读取通过邮件和 WeChat 获得的运价，省去手动录入的麻烦？
2. AI 能否从航线名（如 NHAVA SHEVA）和运价文本中提取必要信息，自动填写表格？

### 2.4 核心痛点汇总

| 序号 | 痛点 | 详细说明 |
|------|------|----------|
| 1 | **运价来源分散** | 15~20家供应商，通过邮件、WeChat、QQ等多渠道发送，格式各异 |
| 2 | **格式不统一** | Excel表格、PDF合约、邮件纯文本、聊天截图，四种以上格式 |
| 3 | **手动录入耗时** | 每2周更新一次，每次需人工逐条录入Excel，耗费大量时间 |
| 4 | **费率频繁变动** | 运费随季节和市场波动，部分报价有效期仅7天 |
| 5 | **投入产出比极低** | 投标成功率仅 0%~10%，投入与回报严重失衡 |
| 6 | **人工操作易出错** | 手工录入可能导致费率数值错误 |
| 7 | **信息维度多** | 不仅要填费率，还需提供服务方案（航司、中转时间、附加费等） |

## 四、AI 解决方案设计

### 4.1 目标架构（AI 导入后的业务流程）

1. 购买部门收到运价（各渠道）→ 上传至系统
2. **AI 自动解析**各种格式的运价数据（Excel/PDF/文本/图片）
3. AI 提取标准化字段：起运港、目的港、柜型、费率、附加费、有效期、航线代码、备注
4. 数据存入**统一费率数据库**
5. 客户发来投标包（PKG）→ AI 自动匹配费率 → 自动填入
6. 营业人员审核 → 提交投标

### 4.2 AI 解析策略（按数据源类型）

| 数据源 | 解析技术 | 输出 |
|--------|---------|------|
| **结构化 Excel** | pandas + openpyxl 直接读取 | 标准化费率记录 |
| **PDF 合约** | pdfplumber 文本提取 + Claude 结构化 | 标准化费率记录 |
| **邮件纯文本** | Claude NLP 实体提取 | 标准化费率记录 |
| **WeChat 截图** | Claude Vision（多模态识别） | 标准化费率记录 |

### 4.3 标准化费率数据模型

```
freight_rate:
  id               -- 主键
  source_type      -- 来源类型：excel/pdf/email_text/wechat_image
  source_file      -- 来源文件名
  carrier          -- 船司/联运商：KMTC, ONE, Evergreen, OOCL 等
  origin           -- 起运港（UN/LOCODE）：CNSHA, CNNGB 等
  destination      -- 目的港（UN/LOCODE）：KRPUS, USLAX 等
  service_code     -- 航线代码：KTX, CHT, ANX 等
  container_20gp   -- 20尺柜运费 (USD)
  container_40gp   -- 40尺柜运费 (USD)
  container_40hq   -- 40尺高柜运费 (USD)
  container_45     -- 45尺柜运费 (USD)
  baf_20           -- 燃油附加费 20'
  baf_40           -- 燃油附加费 40'
  lss_20           -- 低硫附加费 20'
  lss_40           -- 低硫附加费 40'
  valid_from       -- 有效起始日
  valid_to         -- 有效截止日
  transit_days     -- 中转天数
  remarks          -- 备注（直达/中转/特殊条件等）
  created_at       -- 导入时间
```

| 技能名称 | 功能说明 |
|----------|----------|

## 五、定价团队（Pricing Team / 購買部署）的角色

定价团队（即购买部门）是本方案的关键数据源，其职责包括：

- 从 **15~20 家供应商**（船司、联运商、代理店）获取最新运价
- 通过**邮件、WeChat、QQ** 等多渠道收集费率信息
- 维护 **中国主要港口**（上海、宁波、青岛、厦门、盐田等）至全球各目的地的运价
- 每 **2 周**更新一次运价表
- **当前方式**：手动录入 Excel → **目标方式**：上传至AI系统自动解析入库

## 六、技术难点分析

### 6.1 数据层面

| 难点 | 说明 | 实际案例 |
|------|------|---------|
| **格式多样** | 4种以上数据格式需统一处理 | Excel、PDF(372页)、邮件文本、WeChat截图 |
| **列结构差异** | 不同供应商的 Excel 列名和结构不同 | KMTC 有BAF/LSS列，NVO FAK 没有 |
| **数据量大** | 单个费率表可达数千行 | NVO FAK 约4000+行 |
| **有效期短** | 部分运价有效期仅7-10天 | NVO FAK 有效期 3/20-3/31 |
| **非结构化** | 邮件文本和截图需 AI 理解 | 长荣报价用中文口语描述多条航线 |

### 6.2 文档处理层面

| 难点 | 说明 |
|------|------|
| **投标包格式不统一** | 不同客户的投标包格式各异，需适配多种模板 |
| **多语言混合** | 同一文档中混合中文、日文、英文（如 KMTC 表头中文+英文港口名） |
| **数据精准映射** | 需将航线信息精确对应到投标包中正确的栏位 |

### 6.3 AI/系统层面

| 难点 | 说明 |
|------|------|
| **视觉识别准确率** | WeChat 截图中的数字识别需高精度 |
| **NLP 实体提取** | 从非结构化中文文本中准确提取港口名、费率、有效期 |
| **PDF 表格提取** | 372页 PDF 中定位费率表区域并结构化提取 |
| **港口名标准化** | 同一港口可能有多种写法（巴生港/Port Kelang/MYPKG） |
| **费率校验** | 需要合理性检查，防止解析错误导致报价异常 |

## 七、建议技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| **大模型（LLM）** | Claude API（多模态） | 文本理解、图片识别、实体提取、多语言翻译 |
| **RAG** | 向量数据库 + Embedding | 从费率数据库精准检索航线费率 |
| **数据库** | PostgreSQL | 核心费率数据存储 |
| **缓存** | Redis（可选） | 高频查询的热门航线费率缓存 |
| **后端** | Python + FastAPI + Celery | API服务、AI调度、异步处理 |
| **前端** | React + TypeScript + i18n | 管理界面、费率展示、三语切换 |
| **文档处理** | pandas + openpyxl + pdfplumber | Excel/PDF 读写与解析 |
| **部署** | Docker 容器化 | 统一部署环境 |

## 十、风险与注意事项

1. **数据安全**：费率数据属于商业敏感信息，AI 系统需做好数据隔离和权限控制
2. **准确性验证**：AI 自动解析的数据必须经过人工审核后才能使用
3. **有效期管理**：运价有效期短（7~31天），需要清晰标注过期数据
4. **港口名标准化**：需建立港口名称映射表（中文/日文/英文/UN LOCODE）
5. **供应商格式变化**：同一供应商的费率表格式可能随时间变化，解析器需具备容错能力
6. **Demo 数据真实性**：使用福山提供的真实数据构建Demo，增强说服力
