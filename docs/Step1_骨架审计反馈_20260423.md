## 三、🔴 P0 — 阻塞，必修（预计 30 分钟）

### P0-1 `rate_batch_service._build_draft_batch` 会 KeyError

**位置**：`backend/app/services/rate_batch_service.py:250`

```python
batch_id=legacy_payload["batch_id"],
```

但 `ParsedRateBatch.to_legacy_dict()`（`entities.py:254-269`）**没有 `batch_id` 字段**。

**实测确认**：

```
keys: ['file_type', 'source_file', 'effective_from', 'effective_to',
       'total_rows', 'warnings', 'records', 'parsed_rows']
has batch_id: False
```

**后果**：任何 `/rate-batches/upload` 调用都会抛 `KeyError: 'batch_id'` → 500。

**修复建议**：在 `ParsedRateBatch` 增加 `batch_id` 字段（与 `Step1ParseResult` 保持一致）：

```python
# backend/app/services/step1_rates/entities.py

import uuid

@dataclass(slots=True)
class ParsedRateBatch:
    file_type: Step1FileType
    source_file: str
    batch_id: str = field(default_factory=lambda: str(uuid.uuid4()))
    effective_from: date | None = None
    effective_to: date | None = None
    records: list[ParsedRateRecord] = field(default_factory=list)
    warnings: list[str] = field(default_factory=list)
    adapter_key: str | None = None
    metadata: dict[str, Any] = field(default_factory=dict)

    def to_legacy_dict(self) -> dict[str, Any]:
        parsed_rows = [record.to_legacy_dict() for record in self.records]
        payload = {
            "batch_id": self.batch_id,          # <-- 新增
            "file_type": self.file_type.value,
            "source_file": self.source_file,
            "file_name": self.source_file,      # <-- 顺便补一下，rate_batch_service 会 .get() 它
            "effective_from": self.effective_from,
            "effective_to": self.effective_to,
            "total_rows": len(self.records),
            "warnings": list(self.warnings),
            "records": parsed_rows,
            "parsed_rows": parsed_rows,
        }
        if self.adapter_key is not None:
            payload["adapter_key"] = self.adapter_key
        payload.update(self.metadata)
        return payload
```

**验收标准**：
1. `rate_batches.py` upload 端口能在三份真实文件上跑通（即使 records=[] 返回空 preview 也算通过）
2. 返回的 `batch_id` 可用于后续 `/rate-batches/{batch_id}/diff` 查询

## 四、🟡 P1 — 应修，不堵里程碑（但别拖过 4/28）

### P1-1 `app/models/__init__.py` 丢掉了 Phase 1 模型的导出

**位置**：`backend/app/models/__init__.py`

现状只导出 Port / Carrier / ImportBatch / Air\* / FreightRate / LclRate / UploadLog。Phase 1 的 `Lane / Surcharge / Tariff / DomesticRate` **没有**从 `__init__.py` 重新导出。

**当前影响**：
- 业务代码都走 `from app.models.lane import Lane` 直接子模块导入，运行时没问题 ✅

- `Base.metadata.create_all()` 对冷启动环境会漏建这 4 张表

**修复建议**：`app/models/__init__.py` 补回导出：

```python
from app.models.lane import Lane, TransportMode
from app.models.surcharge import Surcharge, CalculationType
from app.models.tariff import Tariff
from app.models.domestic_rate import DomesticRate
```

并更新 `__all__`。

## 七、4/28 里程碑审计我会用的方法

1. **抽样对数**：从每份文件随机抽 10 条入库记录，人工打开 Excel 对价格、币种、生效期、合并单元格配对关系
2. **Ocean JP sheet 的 2 行配对**重点核对：同一 Destination + Carrier 下 20FT 价格和 40FT 价格是否都写到了同一条记录的 `container_20gp / container_40gp`
3. **LCL 双单位拆分**重点核对：源文件 `"5/CBM, 10/TON"` 是否正确拆成 `freight_per_cbm=5, freight_per_ton=10`
4. **Ocean-NGB Lv.1/2/3**重点核对：同一航线是否真的产生 3 条记录、`rate_level` 字段是否正确
5. **Air 周表 7 列**：`price_day1 ~ price_day7` 和源文件 7 个日期列对应关系是否正确
6. **Surcharges 合并单元格**：`AREA / FROM` 纵向合并的值是否正确展开到下方所有行
