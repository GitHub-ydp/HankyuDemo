## 三、🔴 P0 — 数据正确性问题（4/28 前必修）

### P0-A  Other Ports sheet 全部 `20GP` 行被丢弃 —— 数据丢失

**实测表现**：14 条 "`encountered 40FT/40HQ before 20FT pair`" 警告，实际是**14 条 20GP 行数据全部丢失**。

**根因**：Other Ports sheet 的容器类型标识与 JP sheet **不一致**：

| sheet | 20 尺标识 | 40 尺标识 |
|---|---|---|
| JP N RATE FCL & LCL | `20FT` | `40FT/40HQ` |
| FCL N RATE OF OTHER PORTS | **`20GP`** | **`40HQ`** |

**实例**（源文件 row 29-30）：
```
row29: BANGKOK | KMTC | 20GP | 850 | ...
row30: None    | None | 40HQ | 1700 | ...
```

`_normalize_container_type` 现有逻辑：
```python
if upper == "20FT": return "20ft"
if "40FT" in upper or "40HQ" in upper: return "40ft_40hq"
```

→ `20GP` 命中不了任何分支，返回 None → 被 138-139 行 `continue` 跳过 → 850 USD 的 20GP 报价**完全丢失**。
→ 后面的 `40HQ` 行因为 pending=None，触发 "`encountered 40FT/40HQ before 20FT pair`" 警告，以孤儿身份保存（`container_20gp=None`）。

**影响范围**：Other Ports sheet 57 条 FCL 中约 14 对配对破碎，涉及 BANGKOK / LAEM CHABANG / HONG KONG 等非日本港。**这些港的 20GP 价格在库里全部查不到**。

**修复建议**：扩展容器类型识别：

```python
def _normalize_container_type(self, value: Any) -> str | None:
    text = self._normalize_text(value)
    if not text:
        return None
    upper = text.upper().replace(" ", "")
    if upper in {"20FT", "20GP", "20'"} or upper.startswith("20"):
        return "20ft"
    if "40FT" in upper or "40HQ" in upper or "40GP" in upper or upper.startswith("40"):
        return "40ft_40hq"
    return None
```

**验收**：Other Ports sheet 14 条 "orphan 40FT" 警告消失，BANGKOK/KMTC/20GP=850 能被正确查到。

### P0-B  目的地名称内部换行未标准化 —— 下游全错

**实测表现**：JP sheet 的首条记录 `destination_port_name = 'TOKYO\nYOKOHAMA'`（带换行符）。

**根因**：源文件 `A9` 单元格实际值是 `'TOKYO\nYOKOHAMA'`（一格内换行显示为两行），`_normalize_text` 只做 `strip()`，不规整内部空白。

**影响**：
- 前端展示成两行
- Diff 匹配 / 港口 lookup / SQL 查询全部对不上（`TOKYO YOKOHAMA` ≠ `TOKYO\nYOKOHAMA`）
- 后续 Step2 入札对接时，客户模板里是 `TOKYO YOKOHAMA`（单空格），直接对不上

**修复建议**：`_normalize_text` 增加内部空白规整：

```python
import re
_WHITESPACE_RE = re.compile(r"\s+")

def _normalize_text(self, value: Any) -> str | None:
    if value is None:
        return None
    text = _WHITESPACE_RE.sub(" ", str(value)).strip()
    return text or None
```

**验收**：`destination_port_name == 'TOKYO YOKOHAMA'`（单空格，无换行）。

### P0-C  warning 噪音爆炸 —— 合法业务文本被误报

**实测表现**：124 条记录产生 **166 条 warning**（去重后），其中 **72 条是 `AT COST (COLLECT)` 误报**。

```
WARN: JP N RATE FCL & LCL row 9: surcharge column 1 contains non-numeric text 'AT COST (COLLECT)'.
WARN: JP N RATE FCL & LCL row 10: ... (重复 72 行)
```

**根因**：源文件 `LSS+CIC` 列的 `AT COST (COLLECT)` 是**合法业务语义**——意思是"按到付实报实销"，不是数据错误。同类合法文本还有 `Collect` / `subject to Dest Charge` / `Included`。

**影响**：楢崎亲手上传时会看到一片 warning，**以为文件格式有问题**——实际上是解析器误报。信任直接受损。

**修复建议**：建立"已知合法非数字值"清单，命中时写入 `extras` 但**不 emit warning**：

```python
KNOWN_NON_NUMERIC_TOKENS = {
    "at cost (collect)", "at cost", "collect", "included",
    "subject to dest charge", "subject to destination", "dest collect",
}

def _classify_surcharge_text(self, raw: str | None) -> tuple[Decimal | None, str | None, bool]:
    """返回 (decimal_value, raw_text, is_known_non_numeric)。"""
    if not raw:
        return None, None, False
    decimal_value = _safe_decimal(raw)
    if decimal_value is not None:
        return decimal_value, None, False
    normalized = raw.strip().lower()
    is_known = normalized in KNOWN_NON_NUMERIC_TOKENS or "at cost" in normalized
    return None, raw, is_known
```

调用点改为只对**未知的**非数字文本报 warning。

**验收**：warning 总数 <50 条（可接受的异常信号），`AT COST (COLLECT)` 不再出现在 warning 列表，但保留在 `extras.lss_cic_raw` 等字段。

## 四、🟡 P1 — 设计层问题（4/30 里程碑前敲定）

### P1-A  `baf` / `ebs` / `yas_caf` 单字段语义矛盾

**现象**：
- 源文件 BAF 按集装箱类型分价：20FT=200, 40FT=400

**问题**：`baf` 这个字段到底表示什么？如果是"代表值"——为什么取 20FT？如果是"汇总"——为什么不是总和或最大值？查询和回填时语义不清。

# 取所有 sheet 的 max(effective_from) 和 min(effective_to)，作为"全 sheet 都生效"的窗口
valid_froms = [m["effective_from"] for m in sheet_summaries if m.get("effective_from")]

## 五、🟢 P2 — 小优化

### P2-A  `record_kind='fcl'` / `'lcl'` 是字符串，非枚举
建议定义 `RateRecordKind` 枚举，避免散落的字符串拼写错误。

### P2-B  LCL BUSAN `0/RT` 的语义
实测：`BUSAN: freight_raw='0/RT' → freight_per_cbm=0, freight_per_ton=0`

"0/RT" 在业务上可能是"免费"或"到付（按托）"。当前入库成 0 会让"最低价筛选"把它当最便宜，有误导。

**建议**：`value == 0` 时把 `freight_per_cbm/ton` 置 `None` 并 `warnings.append("BUSAN LCL freight is 0/RT — treat as collect/unquoted, needs 楢崎 confirmation")`。

### P2-C  `_build_fcl_row_payload` 在 20FT 行就预填了 `container_40gp=freight` / `container_40hq=freight`
如果某 20FT 行没有 40FT 配对（孤儿场景），会错误地把 20FT 价格写进 40gp/40hq。建议初始化为 `None`，只让 merge 步骤填 40。
