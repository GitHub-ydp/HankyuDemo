## 需求 2：为什么要产出 cost 版和 S/R 版两个文件

- **出处**：
  - 邮件 `1-②-1.msg`（楢崎 → 购买部杨杰，2025-12-10 18:05）原文："Please advise **estimated cost**, main carrier and route(Column G) within this Friday"——**cost 版是购买部给上海日籍管理层 (Nakamura) 审批用的内部稿**。
  - 邮件 `1-②-2.msg`（杨杰 → 楢崎）回信附件即 `2-②.xlsx`（cost 版实物）。
  - 邮件 `1-②-3.msg`（Nakamura → 楢崎，2025-12-11）原文："**S/R is as below** … ATL 49 MIA 54 AMS 41 SYD 25 TPE 13 **Please send to HEX**"——Nakamura **在 cost 版基础上**手工计算 S/R（销售价），指示楢崎转发给 HEX（阪急阪神日本本部关根）。
  - 邮件 `1-③.msg`（楢崎 → 关根，日本本部）原文："PVG発分を送付致します。添付ご査収のほどお願い致します"——楢崎把 S/R 版（`2-④.xlsx`）附件发回日本，日本本部再转客户 ミマキエンジニアリング。
- **业务目标**：系统必须同时满足**两个读者的不同需求**，用"一份 xlsx 写两遍"替代 Nakamura 当前的手工加价与"另存为"操作。
- **业务规则**（两版差异由下列表规定，任何字段偏离都是 bug）：

| 差异维度 | cost 版 | S/R 版 |
|---|---|---|
| 业务读者 | 上海购买部门内部审批（原流程由 Nakamura 审）| 日本本部关根 → 最终客户 ミマキエンジニアリング |
| E 列（单价） | **matcher 返回的成本价**（由 T-B5 产出，未加价） | **成本价 × 加价系数**（本轮系数固定 = 1.15；业务期望 ≈ 1.08-1.09） |
| F 列（lead time） | **两版相同**（实证见 §0.2 / §0.3 F 列完全一致） | 同 cost 版 |
| G 列（carrier/route）| **两版相同**（实证见 §0.2 / §0.3 G 列完全一致） | 同 cost 版 |
| H 列（備考） | **空**（实证 2-②.xlsx H 列 PVG 段全为 None）| **填 `'ALL-in'`**（实证 2-④.xlsx 的 5 条 AIR_FREIGHT 行均为 `'ALL-in'`；2 条 LOCAL_DELIVERY 保持空）|
| 文件命名 | `cost_` 前缀（架构任务单 §11.2 第 839 行；见 §需求 6）| `sr_` 前缀 |

- **验收条件**：楢崎把 `2-①.xlsx` 拖进系统 → 得到两个文件。两份 xlsx 打开后 E 列在 PVG 段 5 条 AIR_FREIGHT 行（R13/R15/R16/R18/R19）数值**不相等**（S/R = cost × 1.15），且 H 列在同样 5 行呈现 `空 vs 'ALL-in'` 的差异。
- **边界**：
  - 两个 xlsx 是**各自独立的 .xlsx 文件**，不是一个文件多 sheet、不是压缩包。
  - 命名规则 / 下载入口由 §需求 6 单独定义。
- **业务风险**：Nakamura 在原流程中**手工**把 cost 加到 S/R（邮件 1-②-3 即为证据）。如果系统产出的 S/R 数值和他心算不一致（差 1-2 块人民币），他会在 Demo 现场质疑"机器算的不对"——解法见 §9 风险 1。

## 需求 3：T-B7 收到的输入契约（上游给了什么）

- **出处**：T-B4 实现 `backend/app/services/step2_bidding/customer_profiles/customer_a.py`（第 140-176 行 parse 方法）、T-B5 架构任务单 §7.1 match 签名 + 测试报告 32 条验收。
- **业务目标**：把"T-B4 + T-B5 产出的结构化数据"当成**黑盒输入**，T-B7 自己不得再去碰源 xlsx 的解析、也不得再去碰费率库。
- **业务规则**（T-B7 业务可用的信息分 3 层）：
  1. **段级信息（ParsedPkg.sections）**：5 个段的发地代码 / 币种 / 表头所在行 / 是否为本地段（`is_local_section=True` 仅 PVG）。**T-B7 只对 `is_local_section=True` 的段回写**。
  2. **行级信息（ParsedPkg.rows）**：每行的目的地代码、原文、行号（Excel 物理行）、cost_type（AIR_FREIGHT / LOCAL_DELIVERY）、是否为"記入例"行、客户硬约束文本、客户已填 E 列值。**T-B7 消费的关键是"行号 → 去哪一行"** 以及"is_example / cost_type 决定是否写"。
  3. **行级匹配结果（matcher 返回的 PerRowReport 列表，与 rows 按 row_idx 对齐）**：每行一个 status（9 种枚举之一）+ 成本价 + 销售价 + 加价比例 + lead_time 文本 + carrier 文本 + 备注文本 + validator 警告。**T-B7 必须按 status 分支决定"写什么、不写什么"**（见 §需求 4）。
- **验收条件**：架构大师拆任务时必须把"上述三层数据的消费顺序"在代码里体现。T-B7 不得打开源 xlsx 再解析一次表头、不得重新调 matcher。
- **边界**：
  - T-B7 **允许**读源 xlsx 的**单元格格式**（字体、合并、列宽、comments、workbook properties），但**不允许**再解析业务字段（已由 T-B4 完成）。
  - T-B7 **不接收** Step1 费率库连接、**不接收** RateRepository 实例。
- **业务风险**：若 parse / matcher 出的 row_idx 与源 xlsx 实际行号不一致（例如 parse 把 R13 登成 R12），T-B7 回写会全错位。**验收标准见 §需求 7 V1**。

## 需求 4：回写规则（逐列 × 逐 status 业务分支）

- **出处**：架构任务单 §11.1 第 823-833 行"查不到费率时的回填规则"表 + §11.2 第 835-843 行"cost 版 vs S/R 版差异"表 + 黄金样本 §0.1 / §0.2 / §0.3 实测。
- **业务目标**：把"一行匹配结果"翻译成"Excel 那一行上 E/F/G/H 四格的具体值"，并保证对任何 status 都有明确答案（不能留"材料未明"的灰区）。
- **业务规则**：下表为业务结论的完整决策表。行是 matcher 给的 status，列是 E/F/G/H 四个目标单元格。"保持原值"= 不触碰、维持 2-①.xlsx 的原始内容。

| matcher 给的 status | 业务含义（为什么） | E（单价） | F（lead time） | G（carrier） | H（備考） | 两版是否差异 |
|---|---|---|---|---|---|---|
| FILLED | 成功匹配到运价 | cost 版：成本价 / S/R 版：成本价 × 1.15 | lead_time 文本 | carrier 文本 | cost 版：空 / S/R 版：`'ALL-in'` | **是**（E、H）|
| NON_LOCAL_LEG | 非 PVG 段（NRT / AMS / TPE / ICN 所在段）| 保持原值 | 保持原值 | 保持原值 | 保持原值 | 否 |
| EXAMPLE | "※記入例" 示范行（黄金样本 R4/R5）| 保持原值 | 保持原值 | 保持原值 | 保持原值 | 否 |
| LOCAL_DELIVERY_MANUAL | PVG 段内 R14/R17 本地配送行；客户原文要求"由 HEX 日本在各自当地填" | 保持原值（实测 = 0）| 保持原值（实测 = `'－'`）| 保持原值（实测 = `'－'`）| 保持原值（空）| 否 |
| ALREADY_FILLED | 客户或上游已在 E 列填了非零值，不覆盖 | 保持原值 | 保持原值 | 保持原值 | 保持原值 | 否 |
| NO_RATE | 费率库没命中 | 留空（置 None） | 留空 | 留空 | 留空 | 否 |
| CONSTRAINT_BLOCK | 客户硬约束把所有候选拦掉（如只接受 NH 但周表没 NH）| 留空 | 留空 | 留空 | 写回 `client_constraint_text`（让审核者看到是哪条约束拦的）| 否 |
| OVERRIDDEN | 营业在审核页改过价（本轮 T-B10 才交付，T-B7 暂不主动处理此 status）| 按营业改后的值 | 同 | 同 | cost 版：空 / S/R 版：`'ALL-in'` | 是 |

- **LOCAL_DELIVERY 行的特别业务背景**（出处 `1-②-1.msg` 第 5-6 行）：楢崎对杨杰说 "add 'LOCAL DELIVERY COST' Colum to AMS & ATL, but **you can ignore it**"——中国上海只负责空运段；本地配送由日本/各地 HEX 自己报，因此 R14 / R17 **不得被 T-B7 填入任何费率**，保留客户原模板里的 `'－'`。
- **H 列"ALL-in"业务含义**（出处 Nakamura 邮件 `1-②-3.msg` + 2-④.xlsx 实证）：ALL-in = "此 AIR FREIGHT 单价已含所有附加费（燃油费 / 安检费），无后续加费"。这是**给客户**看的标记；cost 版给内部，不需要此标记，因此 cost 版 H 列不写。**v1.0 常量 = `'ALL-in'` 五个字符**；未来若客户硬约束含文本需求（见 §需求 5），再讨论是否追加。
- **验收条件**：见 §需求 7 V1-V5 五条硬测。
- **边界**：
  - H 列在本轮**不合并** `client_constraint_text`（架构任务单 §8.2 第 690 行提过"有 client_constraint 则追加"，但黄金样本 PVG 段 5 行的 client_constraint_text 全为空，**v1.0 不处理追加**，保持 `'ALL-in'` 单值写入；若未来出现 PVG 行的客户约束非空，先抛 warning 给审核页，再由人工决定是否追加）。
  - 不为 OVERRIDDEN 状态做特殊分支（T-B7 本轮只处理"一次性从 matcher 产出"的场景）。
- **业务风险**：
  - 若架构大师把"NO_RATE + H 列 = 空" 和 "CONSTRAINT_BLOCK + H 列 = client_constraint_text"写反了，审核页会看不到"为什么没填"，营业无法追问购买部。**测试必须覆盖 CONSTRAINT_BLOCK 的 H 列写入**。
  - 若开发把"LOCAL_DELIVERY_MANUAL 行"错写成"FILLED"分支，会把 `'－'` 覆盖成数字，Demo 现场 Nakamura 一眼发现。

  3. **B1 单元格（入札期间文本）** 原样保留。
  4. **記入例行（R4 / R5）** 原样保留。
  5. **段级硬约束行（R38 `'※韓国→日本→ブラジル…'`）** 原样保留。
  6. **单元格格式**：字体、字号、颜色、背景色、边框、对齐、换行 —— 原样保留。
  7. **合并单元格（merged_cells）**：原文件有几个合并区、合并到哪里，产出文件必须完全一致（经实测 2-①.xlsx 无合并，但开发实现不得因此省去"保留合并"的代码路径）。
  8. **列宽 / 行高（column_dimensions / row_dimensions）**：原样保留。
  9. **公式（formula cells）**：原文件若存在公式，公式原文必须保留；不得将公式计算结果写死成数字。
  10. **批注（comments）**：如有批注，保留；不得被清除。
  11. **Document Properties**：允许追加 `batch_id / exported_at / writer_version` 等元数据（参照 Step1 `stamp_document_properties` 做法），**但**原文件已有的 title / author 等不得被清除。
  12. **Sheet 名**：原文件为 `'見積りシート'`，产出必须保留此名；禁止改名、禁止新增 Sheet。
- **验收条件**：
  - 用 openpyxl 打开产出 xlsx，对非 PVG 段逐单元格与原文件比对，**除允许改动的 5 行 × 4 列 = 20 个单元格外，所有其他单元格的 value 与格式完全一致**（见 §需求 7 V4 / V5）。
  - Windows Excel 打开产出 xlsx 不弹任何"文件已损坏、已修复"警告。
- **边界**：
  - v1.0 **不**保证"VBA 宏"保真（Customer A 黄金样本无宏；Nitori 才有宏，属 v2.0 范围）。
  - v1.0 **不**保证"数据验证（data validation）/ 条件格式"的完全保真，但**不得主动删除**（若 openpyxl 默认保留则无需额外编码）。若实测发现黄金样本存在条件格式且被破坏，需在本文件 §9 追加风险项。
- **业务风险**：
  - openpyxl 对"图片 / 冻结窗格 / 超链接"可能不保真；Demo 前测试大师必须在黄金样本上单独验证一次（V7 边界用例）。
  - 如果开发图省事用 pandas 读写 xlsx，格式会全丢。**业务层硬要求：必须沿用 openpyxl、以 copy2 + load/save 骨架实现**（架构任务单 §1.2 第 89 行、§6.4 第 568 行已强调）。

## 需求 6：两个输出文件的命名与下载

- **出处**：架构任务单 §9.4 第 774-776 行下载接口、§11.2 第 839 行文件名表、§13.1 第 873 行 DB 字段 `cost_file_path / sr_file_path`。
- **业务目标**：楢崎 / Nakamura / 前端审核者都能从文件名一眼分辨"这是内部 cost 版还是客户 S/R 版"，不靠打开内容判断，避免误发。
- **业务规则**：
  1. **命名规范**：
     - cost 版文件名 = `cost_` 前缀 + 原上传文件名（保留原文件的 `.xlsx` 扩展名）。例：原文件 `2-①.xlsx` → cost 版 `cost_2-①.xlsx`。
     - S/R 版文件名 = `sr_` 前缀 + 原上传文件名。例：`sr_2-①.xlsx`。
  2. **下载渠道**：v1.0 两个文件都要能被审核页"下载 cost 版 / 下载 S/R 版"按钮分别下载；具体 API 与前端实现由 T-B10 交付，T-B7 只需保证两文件物理存在、路径记录在 BiddingRequest 上。
  3. **命名的跨语言展示**：前端按钮文案需要中 / 日 / 英三语（业务文档 §2.2 / CLAUDE.md 多语要求）：
     - 中文：`下载成本价版（内部审批用）` / `下载销售价版（寄客户用）`
     - 日文：`コスト版ダウンロード（社内決裁用）` / `S/R 版ダウンロード（顧客送付用）`
     - 英文：`Download cost version (internal)` / `Download S/R version (to client)`
  4. **禁止**把 cost 版和 S/R 版合并成一个文件（例如用两个 sheet）。原始业务流的两封邮件（`1-②-2.msg` 附件 = cost 版独立 xlsx、`1-③.msg` 附件 = S/R 版独立 xlsx）证明**分文件**是客户与日本本部的既有工作习惯。
- **验收条件**：Demo 当天楢崎在审核页看到两个清楚的下载按钮；本地下载后文件名符合规则 1；两个文件均可独立打开，互不依赖。
- **边界**：
  - 不做 zip 打包下载（v1.0）。
  - 不做"审核通过后才允许下载 S/R 版"的权限分级（v1.0；v1.5 审核页功能交付时再议）。
- **业务风险**：
  - 若同期间（例如 2026-01）同客户多次上传（楢崎重跑 matcher），文件可能重名覆盖。**v1.0 兜底策略：按 `bid_id` 为目录层级隔离**（具体实现由架构大师定；业务只要求"不会覆盖"）。
  - 中 / 日 / 英三语按钮文案需要张东旭向楢崎确认日文措辞是否贴切（见 §9 Q-T-B7-03）。

## 需求 7：业务级验收标准（给测试大师）

测试大师需基于 §0 黄金样本三份文件的实测数据写 pytest。以下 5 条"必过"+ 2 条"边界"是**业务级硬底线**，不达标 = T-B7 不交付。所有验收用例必须能独立运行（不依赖 Step1 入库真实数据；允许 mock matcher 返回固定 PerRowReport）。

### 7.1 业务验收 5 条

- **V1 · 双版本差分**：给定 PVG 段 5 条 AIR_FREIGHT 的 matcher 结果（假设 cost_price 分别为 45 / 50 / 38 / 22 / 12），T-B7 产出 `cost_*.xlsx` 与 `sr_*.xlsx`。读取两文件的 E13 / E15 / E16 / E18 / E19：
  - cost 版 = `[45, 50, 38, 22, 12]`（数字类型，不是字符串）
  - S/R 版 = `[52, 58, 44, 26, 14]`（即每条成本 × 1.15 并向上取整；具体取整策略由架构任务单 §8.1 `ceil_int` 规定）
  - 五条 S/R 均 ≠ 对应 cost（不得意外相等）
- **V2 · H 列两版差异**：同上产出文件，读取 H13 / H15 / H16 / H18 / H19：
  - cost 版全部为 None 或空字符串
  - S/R 版全部等于字符串 `'ALL-in'`
  - H14 / H17（LOCAL_DELIVERY 行）两版均保持原值（空）
- **V3 · F / G 列两版一致**：F13 / F15 / F16 / F18 / F19 的 value 在两版之间逐格相等；G 列同理。（业务依据：Nakamura 的 S/R 邮件 1-②-3 只改价、不改 lead time 和 carrier）
- **V4 · 非 PVG 段零改动**：对 NRT（R3-R10）、AMS（R21-R23）、TPE（R25-R32）、ICN（R34-R41）四个段的**每一个单元格**，产出两版与原 2-①.xlsx 逐格 value 相等。R38 的 ICN 段级硬约束 `'※韓国→日本→ブラジル…'` 原样保留。
- **V5 · PVG 段 LOCAL_DELIVERY 行零改动**：R14 / R17 的 E / F / G / H 四列在两版中与 2-①.xlsx 逐格 value 相等（E = 0、F = G = `'－'`、H 空）。

### 7.2 边界验收 2 条

- **V6 · 全未匹配（NO_RATE 五条）**：mock matcher 对 PVG 段 5 条 AIR_FREIGHT 行全返回 NO_RATE。T-B7 产出两版文件：
  - 两版 E / F / G / H 列在 R13 / R15 / R16 / R18 / R19 均留空（None）
  - 非 PVG 段仍然零改动（同 V4）
  - 产出的 FillReport 记录 5 条 no_rate 状态 + 0 条 filled 状态
  - 文件仍能打开，且 Sheet 名仍为 `'見積りシート'`
- **V7 · 部分未匹配（3 FILLED + 2 CONSTRAINT_BLOCK）**：例如 R13 / R15 / R16 FILLED，R18 / R19 CONSTRAINT_BLOCK（且 `client_constraint_text = '※某约束原文'`）：
  - R13 / R15 / R16 按 V1 / V2 规则填入
  - R18 / R19：E / F / G 留空；H 列写回该行的 `client_constraint_text`（两版一致，都写约束文本 —— 因为 S/R 版 H 列本来就是 `'ALL-in'`，被约束覆盖后写约束；此处的 H 列两版策略由 §需求 4 表末行规定）
  - 非 PVG 段仍零改动

### 7.3 补充要求（给测试大师但不计入 V1-V7）

- 所有生成的 xlsx 用 Excel Desktop 打开不得弹出"文件已损坏"警告（人工抽检，无法自动化；列入 Demo 前 checklist）。

## 需求 8：可观察性（Demo 与审核用）

- **出处**：业务文档 §3.3 / §5"审核与反馈"；架构任务单 §12 异常人工介入点。
- **业务目标**：当 Demo 现场楢崎问"这行为什么没填 / 为什么填成这个价"时，系统能在审核页**指给他看源头**（来自哪条周表记录、是哪一天的价、加价系数是多少）。
- **业务规则**（T-B7 范围内的最小集）：
  1. T-B7 必须把每行匹配结果**序列化记录**在 FillReport 中（架构任务单 §4 已定义结构）；Demo 级审核页（v1.5）据此渲染。
  2. 对于 FILLED 行，记录：cost_price、sell_price、markup_ratio（本轮固定 1.15）、命中的 Step1 批次号 / 周表记录号 / 附加费记录号。
  3. 对于非 FILLED 行（NO_RATE / CONSTRAINT_BLOCK / LOCAL_DELIVERY_MANUAL / NON_LOCAL_LEG / EXAMPLE / ALREADY_FILLED），记录 status 原文 + matcher 给的 warnings（让营业能答得出"为什么没填"）。
  4. Demo 当天不做"一键导出追溯报告"（v2.0）；但 **FillReport 本身必须是结构化可读的**（JSON 或 dataclass），前端能渲染、测试能 assert。
- **验收条件**：测试用例 V1 / V2 / V6 / V7 都必须额外断言 FillReport 的 filled_count / no_rate_count / skipped_count 三个统计字段符合预期。
- **边界**：审核页 UI、追溯报告导出、"重跑 matcher"按钮——都不在 T-B7 范围。
- **业务风险**：如果 FillReport 只记"成本/销售"不记"来自哪条 Step1 批次"，Demo 现场被追问时答不出，项目可信度受损。**此点由 T-B5 matcher 的 QuoteCandidate 传递，T-B7 只需透传，不能丢字段**。

## 需求 9：风险清单与待楢崎确认问题（Q-T-B7-XX）

### 9.1 风险 1：1.15 加价系数与实证偏差 5-7%（高）

- **事实**：§0.4 实证 cost→S/R 平均加价 ≈ 1.09（SYD 甚至 1.136）；架构任务单 §18.2 第 1055 行承认"×1.15 会比真实 S/R 高 5 块钱，客户或楢崎一眼看出"。
- **业务影响**：Demo 现场 Nakamura 若按自己经验心算 45 × 1.1 ≈ 50，看到系统算出 52，可能不接受。
- **缓解**：
  - 前端 S/R 版展示页 **必须**显示"加价系数 = 1.15（默认值，待楢崎确认）"提示条。
  - **Q-T-B7-01（必问）**：楢崎 5/14 Demo 前若能给出真实加价规则（固定 / 按客户 / 按航线 / 查表），T-B6 回来改 markup 策略；否则 Demo 兜底话术见 `docs/Demo_兜底方案与演示话术_20260424.md`。

### 9.2 风险 2：T-B7 先接 mock markup，后续接 T-B6 是否要改 T-B7 代码（中）

- **事实**：本轮 T-B7 使用固定常量 1.15；T-B6 交付后应改为"从 MarkupApplier 取"。
- **业务影响**：若架构设计时把 1.15 写死在 T-B7 内部，T-B6 交付后需改 T-B7，违反"扩展点前置"。
- **缓解**：架构任务单 §8.1 第 682 行已要求 `_MARKUP_RATIO` 常量放在 markup.py；**T-B7 必须通过依赖注入接受加价函数，不得自己写死 1.15**。业务侧不接受"接入 T-B6 还要改 T-B7 几十行"的架构方案。

### 9.3 风险 3：H 列若存在客户原约束是否需要和 `'ALL-in'` 合并（中）

- **事实**：架构任务单 §8.2 第 690 行提过"有 client_constraint 则追加"；但黄金样本 PVG 段 5 行 client_constraint_text **全空**，无法用实证验证合并格式。
- **业务影响**：若未来某月某条 PVG 行带客户约束，系统要不要写 `'ALL-in / <约束原文>'`？
- **缓解**：**Q-T-B7-02（待问楢崎/Nakamura）**：H 列"ALL-in"出现时，是否允许追加客户约束原文？分隔符用 `' / '` 还是另起一行？**v1.0 默认不追加，仅写 `'ALL-in'`，若检测到 PVG 行 client_constraint_text 非空则在 FillReport 加 warning，审核页人工处理**。

### 9.4 风险 4：双版本文件名日文按钮文案（低）

- **事实**：§需求 6 给出的日文按钮 `S/R 版ダウンロード（顧客送付用）` 属业务大师草拟，未经楢崎本人确认。
- **缓解**：**Q-T-B7-03（可问可不问）**：楢崎对"成本价版 / 销售价版"的日语术语偏好是 `コスト / S/R` 还是 `コスト / 売単価`？v1.0 按 Nakamura 邮件原文 `S/R` 作为默认。

### 9.5 风险 5：openpyxl 对冻结窗格 / 超链接 / 条件格式保真未全面验证（中）

- **事实**：黄金样本 2-①.xlsx 的冻结窗格 / 条件格式未经业务大师逐项测试；架构任务单 §16.3 V-B22 只测了 merged_cells。
- **缓解**：测试大师在交付前用 Excel Desktop 人工抽检一次（Demo 前 checklist）；若发现丢失，升级为 P0。
