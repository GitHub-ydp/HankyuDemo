## 1. 核心设计决策

### 决策 A：DB 表设计 — 选方案 α1 单行表每字段一列

**采用**：**α1**（单行表 `app_settings`，每列一个字段，共 16 业务列 + `id` + `updated_at` + `updated_by`）。

理由（四点全部成立才选 α1）：

1. **字段集在本轮冻结**。业务需求 §0.3 明确排除其他系统参数，§7 "本轮不做"列了 11 项，下一轮即使加也只是 AI 相关 or 用户管理，用户管理会自带 `users` 表不影响本表。三个月内本表列数变化可能为 0。"schema 变化频率"这条 α2 的核心卖点不成立。
2. **查询复杂度**。`config_service.get_ai_config()` 每次 AI 调用都会命中（§2），α2 要 16 次 `JOIN`/`GROUP_BY` 或 python 侧聚合才能拼出一个 dataclass，而 α1 一个 `SELECT * FROM app_settings LIMIT 1` 搞定。性能和可读性都 α1 赢。
3. **类型安全**。α2 每行 `value TEXT`，`int`/`bool`/`str` 在 Python 层反序列化，容易把"85"和 85 混淆；α3 JSON blob 同样弱类型。α1 直接映射 SQLAlchemy 列类型，pydantic schema 校验直接走 ORM → BaseModel 自动转换。
4. **migration 维护成本**。α1 未来加字段就是 `op.add_column`，一行；α2 要改 value_type 枚举；α3 要写 JSON 迁移脚本。在字段频率低的前提下 α1 最省心。

**最容易被反驳的点**："万一下一轮真加用户管理、加系统参数、加 20 个字段怎么办？" 反驳：用户管理会有独立 `users` 表，不塞进 `app_settings`；系统参数若真要做是业务需求 §7 的明确 TODO，下一轮任务单决策时可以新开 `system_settings` 表或再讨论是否转 α2。YAGNI 原则：本轮按冻结字段集做，下一轮再优化结构不晚。

**字段明细**见 §4。

### 决策 B：配置读取层 — 选方案 β2 TTL 缓存 + DB 覆盖 env

**采用**：**β2**（抽 `config_service.get_ai_config() -> AIConfig`，内部 30 秒 TTL 缓存 + 写时失效）。

理由：

1. **β1（每次 DB 读）性能不可接受**。Step2 PKG 自动填表 1 次调用产生 ~8 次 `chat` 调用（每条航线 1 次 AI 识别），每次多 1 次 SQLite 读虽然单次 <1ms，但加起来会放大 8 倍 logging / connection 开销，demo 现场可能出现莫名其妙的卡顿。
2. **β3（内存单例 pub/sub）需要跨 worker 消息总线**。业务需求 §3 明确"保存即时生效"，但本 Demo 是单 worker uvicorn（见 `CLAUDE.md` "常用命令" 段 `uvicorn app.main:app --reload --port 8000`，默认单进程）。Redis pub/sub 属于过度设计。
3. **β2 TTL 的语义**：
   - `get_ai_config()` 首次调用：读 DB + env fallback，放入 module-level 单例 + `loaded_at` 时间戳
   - 后续调用：若 `now - loaded_at < 30s` 直接返缓存；超时重新读 DB
   - `PATCH /admin/settings/ai` 成功后主动调 `invalidate_cache()` 清零 `loaded_at`
   - 多 worker 场景下即使有 1 个 worker 没收到 invalidate，30 秒内自动收敛（风险 R6 的上限）
4. **与现有 `settings` 共存**：`config_service` 启动时从 `settings.*` 建立 env 默认值基线（`EnvDefaults` 数据类），DB 里读出的非空字段覆盖 env，DB 里 `NULL` 或空串的字段回落到 env。这样 `.env` 修改后重启仍然是"改默认值"的入口，不变的语义。

**最容易被反驳的点**："30s TTL 让 Demo 现场'改完立刻测试'看不到效果怎么办？" 反驳：TTL 在写入时主动 invalidate（见第 3 点），所以走 UI 保存路径的用户**下一次请求**就拿新值，不用等 30 秒；30 秒只是兜底多 worker 漂移，**不影响业务需求 §3 的"保存即生效"验收**。

### 决策 C：API 端点 — 独立 router `admin_settings.py`

**采用**：独立 router 文件 `backend/app/api/v1/admin_settings.py`，挂载到 `router.py`，**不**合到 `admin.py`。

- `GET /api/v1/admin/settings/ai` → 返回 `AIConfigResponse`（见 §3），敏感字段脱敏：`vllm_api_key` / `anthropic_api_key` 只返回 `{"masked": "••••VwH", "is_set": true}` 这种结构体，不返回明文。**返回值里额外带 `source: "db" | "env"` 告诉前端"这个字段当前是 DB 里的值还是 env fallback"**，满足决策 E 的 UI 提示需求。
- `PATCH /api/v1/admin/settings/ai` → body 是 `AIConfigPatch`（所有字段 `Optional`）。敏感字段语义走决策 D（下）。调用 `config_service.update_ai_config(patch, updated_by=user_id)`，成功后 invalidate 缓存，返回最新脱敏版配置。
- `POST /api/v1/admin/settings/ai/test-connection` → body 可选 `TestConnectionRequest`（未来若要"测未保存的配置"准备字段，本轮 body 为空，**仅测当前已保存的配置**，业务需求 §1.3 明确"未保存的编辑不参与测试"）。调 `ai_client.health_check(provider=当前生效 provider, timeout=5.0)`，返回 `TestConnectionResponse{ok, provider, latency_ms, detail}`。

**不合到 `admin.py`** 的理由：`admin.py:33` 当前只有 `reset_rates` 一个端点，语义是"清数据"。设置页的语义是"读写配置"，耦合到一起会让后续用户管理、系统参数再新增端点时混乱。独立 router 让权限门槛（未来上鉴权时）可以整体加在 prefix 上。

**router 挂载位置**：`backend/app/api/v1/router.py:12` 后新加一行 `from app.api.v1.admin_settings import router as admin_settings_router`，并在 `:23` 后 `router.include_router(admin_settings_router)`。

### 决策 D：敏感字段 UI 语义 — "空串=不改 / 特殊标记=清空"

**采用**：**二选一方案 + 明确 sentinel**。

规则：

| 前端传的值 | 后端语义 | UI 实现 |
|---|---|---|
| 字段**不出现**在 PATCH body 里 | 保持 DB 现有值（或 env fallback） | 表单未进入"编辑模式"时，不 include 到 payload |
| 字段 = `null` | 显式清空（DB 里存空串） | 用户点"清空"按钮主动触发，UI 弹二次确认 |
| 字段 = 非空字符串 | 覆盖 DB | 用户点"修改"切到明文输入后输入新值 |
| 字段 = `""`（空串） | **等同 null，清空** | UI 层应阻止用户把"修改"模式下空串直接保存，提示"请输入新值或点清空" |

UI 交互（对应业务需求 §1.2.3）：

1. 默认态：右侧显示 `••••VwH`（后 4 位），旁边按钮"修改"
2. 点"修改" → 输入框清空 + placeholder=`"请输入新 API Key"` + 两个按钮"保存此字段"+"取消"
3. 点"取消" → 回默认态，payload 里此字段不出现
4. 输入非空串点"保存此字段" → 进入草稿状态（不真正提交，等底部"保存"按钮统一 PATCH）
5. "清空"按钮独立（业务需求 §1.2.3 "允许保存为空"），点后弹 Modal "确定清空 API Key？清空后该 Provider 下一次调用会失败"，确认后 payload 里此字段显式 `null`

后端约束：`AIConfigPatch` 的敏感字段类型必须用 pydantic 的 `tristate` 风格——即 `vllm_api_key: str | None = SENTINEL_UNCHANGED`（架构大师定义一个 class sentinel），`None` 显式清空，`SENTINEL_UNCHANGED` 表示不改。开发大师可用 pydantic v2 的 `model_fields_set`（调用方 `model.model_fields_set` 返回实际设置过的字段集合）实现——**不出现在 set 里就是"不改"**，这比自定义 sentinel 优雅且 pydantic 原生。

**最容易被反驳的点**："空串和 null 在 HTTP JSON 里前端要显式构造 `null`，RESTful 风格更常见的是 DELETE 子资源来清空。" 反驳：为一个字段走 DELETE 多设计一个 `DELETE /admin/settings/ai/vllm_api_key` 端点不划算；`null` 语义在 OpenAPI 里清晰，pydantic v2 的 `Optional[str]` 原生区分"未传"vs"传 null"。本设计已经明确 UI 点"清空"才会发 `null`，不会误触。

### 决策 E：env → DB 迁移 — 选方案 ε1 空表 fallback（不预填）

**采用**：**ε1**（migration 只建表，不插默认行；代码层 `get_ai_config()` 发现表空时回落 env）。

理由：

1. **ε2 预插行**的坏处：env 的真实值（含 API Key）会被 migration 写到 DB 里。既要处理"migration 执行时 env 可能还没配置真 key"，又要承担 DB 里一上来就落盘敏感数据的风险。而且 `.env` 改了以后 DB 里的值不会自动跟随，语义倒置。
2. **ε3 启动时同步**的坏处：每次启动都 upsert 一遍，用户在 UI 改了 DB 后重启又会被 env 覆盖回去，荒谬。
3. **ε1 的语义清晰**：
   - DB 里没有任何字段 → `get_ai_config()` 返回纯 env 值，`source="env"`
   - 用户首次点"保存"（PATCH 一个字段）→ `update_ai_config` 在 DB 插入单行，只填用户改的那 1 列，其他列 `NULL`
   - `get_ai_config()` 读出这行 → 用户改过的列用 DB 值，其他列读 `NULL` 回落 env
   - UI 上 `source: "db" | "env"` 逐字段标注，用户能看到"哪些是我改过的、哪些还是默认值"

**单行唯一性**：表加 `id INTEGER PRIMARY KEY CHECK (id = 1)`（单行约束），或更简单——`update_ai_config` 里 `db.query(AppSettings).first()` 不存在就 insert，存在就 update，代码侧保证单行。架构大师推荐**代码侧保证**（CHECK 约束 SQLite/PostgreSQL 语法略有差异，避免 migration 绊脚）。

## 2. 数据流与时序

### 2.1 保存流程

```
┌─ 用户点"保存" ─────────────────────────────────────────────┐
│                                                            │
│ 前端 (Settings.tsx)                                        │
│   ↓ 收集 dirty fields（只 PATCH 改过的字段）                │
│   ↓ 敏感字段按决策 D：未进入编辑模式的不带上；点过"清空"的带 null │
│   ↓ settingsApi.updateAIConfig(patch)                      │
│                                                            │
│ 后端 (admin_settings.py)                                   │
│   ↓ PATCH /api/v1/admin/settings/ai                        │
│   ↓ pydantic 验证 AIConfigPatch                             │
│   ↓ config_service.update_ai_config(patch, updated_by)     │
│                                                            │
│ service (config_service.py)                                │
│   ↓ 读/建单行 AppSettings                                   │
│   ↓ 对 patch.model_fields_set 中每个字段 setattr            │
│   ↓ 敏感字段加密处理（base64 编码，见决策 A 反驳点）          │
│   ↓ db.commit()                                            │
│   ↓ invalidate_cache() 清 module-level singleton            │
│   ↓ 返回最新 AIConfig（去掉 encrypted，解码后 dataclass）    │
│                                                            │
│ router 返回                                                 │
│   ↓ AIConfigResponse（脱敏后，Key 只返回 "••••后4位"）       │
│                                                            │
│ 前端                                                        │
│   ↓ message.success("保存成功")                             │
│   ↓ 重新 setState，dirty 清空，表单回到"已保存"状态          │
└────────────────────────────────────────────────────────────┘
```

### 2.2 AI 调用读配置

```

  ↓
_resolve_provider(None)
  ↓ → get_ai_config().ai_provider           (原: settings.ai_provider)
_resolve_model("vllm", None)
  ↓ → get_ai_config().vllm_model             (原: settings.vllm_model)
```

### 4.3 敏感字段加密策略

**本轮采用 base64 编码（混淆级别，非真加密）**。理由：

1. 业务需求 §2.2 "API Key 在 DB 里不要明文落盘。最低要求：base64 或 AES-Sym 加密"——业务大师接受 base64 做 Demo
2. Demo 单机 SQLite，DB 文件本身已被 `.gitignore`；再引入 `cryptography` 库 + 密钥管理 + IV 存储 + 加解密失败处理会让任务单膨胀 2~3 倍，不值当
3. 列名加 `_encoded` 后缀强提醒这不是明文：`vllm_api_key_encoded` / `anthropic_api_key_encoded`
4. 升级到 AES-GCM 的路径清晰：只需替换 `config_service._encode_secret` / `_decode_secret` 两函数 + migration 改列名/加 IV 列，public API 不变
5. 业务需求 §0.3 附件第 2 问明确了"base64 混淆就够，还是必须 AES-GCM"——业务大师取前者，对齐决策

**实现约束**：
- `config_service._encode_secret(plaintext: str) -> str`：`base64.b64encode(plaintext.encode()).decode()`
- `config_service._decode_secret(encoded: str) -> str`：反向；解码失败 raise + fallback 空串 + log warning
- 空串编码后仍是空串（特殊处理）

### 6.2 兜底策略（`config_service.get_ai_config()` DB 失败时）

```python
def get_ai_config(db=None) -> AIConfig:
    try:
        row = db.query(AppSettings).filter_by(id=1).first()
    except Exception as e:
        logger.warning("app_settings 读失败，回落 env: %s", e)
        row = None
    return _merge_db_over_env(row, _env_defaults())
```

**业务语义**：SQLite 锁 / 表缺失 / 字段缺失 → 全部回落 env，**ai_client 不阻塞业务**，前端显示的就是 env 默认值 + source="env"。这是风险 R5 的缓解。

### 6.3 回滚步骤

**回滚前端**：不依赖后端端点的前端代码先撤。`Settings.tsx` / 菜单齿轮 / 路由注册全部 revert。用户看不到设置页，业务不受影响。

**回滚 API**：`router.py` 移除 `admin_settings_router` include，`admin_settings.py` 删除。

**回滚 ai_client**（T-ST-05 reversal）：把 18 处 `get_ai_config().xxx` 改回 `settings.xxx`。**或者**不 revert 代码，只在 `config_service.get_ai_config()` 里强制 `return AIConfig(**_env_defaults())`，等价于禁用 DB 层。后者更快，适合紧急回滚。

**回滚 migration**：`alembic downgrade 20260421_0001`，`app_settings` 表被 drop。**注意**：回滚前务必确认 DB 里没有用户真改的配置值没备份，否则 drop 会丢数据。

## 7. 风险清单

| # | 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|---|
| R1 | 用户保存错误的 `vllm_base_url` 后，下一次 AI 调用立刻失败 | 中 | Step2 PKG 填表整链路挂 | (a) 前端保存按钮**禁用** "未点过测试连通性" 的保存——**过度严苛，不采纳**；(b) 采纳：保存后在 toast 旁边显示 "已保存；建议点'测试连通性'验证"；(c) 错误调用时 `ProviderUnavailableError` 会被业务层 try/except 吞掉并写 warnings，前端已显示"AI 解析失败"友好提示（`email_text_parser.py:76-85` / `wechat_image_parser.py:89-98`）。Demo 现场张东旭知道"改错了就再改回来"即可 |
| R2 | 多 worker uvicorn 下，一个 worker PATCH 了 DB，另一个 worker 缓存 30s 内仍旧值 | 中（Demo 单 worker 概率为 0） | 某些 AI 调用用旧值 | 决策 B β2 已说明：30s TTL 兜底；生产环境 `--workers 4` 时最多错 30s。若不可接受，后续加 Redis pub/sub（范围 C，本轮不做）。Demo 单 worker 不触发 |
| R3 | 敏感字段空值的语义歧义（空串 vs null） | 高 | 用户不小心清空了 Key | 决策 D 已明确三态：**不出现=不改**，**null=清空**，**非空 str=覆盖**。UI 强制走"修改"按钮 + "清空"按钮两条路径，不允许直接在输入框里输空串保存（业务需求 §1.2.3）。后端 `AIConfigPatch` 用 `model_fields_set` 精确区分"传了"vs"没传" |
| R4 | Alembic migration 失败（字段名拼错 / SQLite 语法问题） | 低 | 部署阻塞 | migration 在本地先 `alembic upgrade head → downgrade → upgrade` 往返一次验证；`downgrade` 脚本简单 `drop_table`，不可能失败 |
| R5 | `config_service` 读 DB 失败（SQLite WAL 锁 / 连接池耗尽） | 中 | ai_client 阻塞 | §6.2 已兜底：try/except 包住 DB 读，任何异常回落 env 默认值，log warning，**绝不 raise 到 ai_client**。业务不中断 |
| R6 | 前端 PATCH 部分更新与敏感字段"不输入=不改"语义 | 高 | 用户以为空值=保持，实际是改成空 | 决策 D + T-ST-07 的 `ApiKeyField` 子组件：UI 层强制走"默认态显示 ••••"→ 点"修改"→ 输入明文→ 点"保存此字段"进草稿的流程，**不存在"输入框留空"这个状态**。"清空"按钮独立且二次确认 |
| R7 | 测试连通性时输入的 Key 泄漏到日志 / URL / query | 中 | 安全事故 | 测试连通性端点 **body 为空**，不接受前端传 Key；只用 DB 里已保存的配置测。`ai_client.health_check` 内部 log 时脱敏（T-ST-05 审计一遍 `logger.*` 调用，确保不打 `vllm_api_key` 明文） |
| R8 | 用户从 vllm 切 anthropic 但 `anthropic_api_key` 是空 | 中 | 下次 AI 调用抛 `ProviderUnavailableError` | (a) 保存时**不**强制校验（业务需求 §8 明确"保存成功；下一次 AI 调用时 ProviderUnavailableError"）；(b) 前端在 provider 切换的 Modal.confirm 里加一行"目标 provider 的 API Key 当前为空，切换后下一次调用会失败，确认切换？"——**架构大师推荐加这一行**，业务需求 §1.2.1 没明写但逻辑上应有，开发大师可在 T-ST-07 实装时加 |
| R9 | 缓存 invalidate 在 PATCH 失败（比如 DB commit 失败）时被错误调用 | 低 | 下次读到旧数据当成新数据 | `update_ai_config` 确保 `db.commit()` 先成功才调 `invalidate_cache()`，顺序绝对不能反；try/except 包住 commit，commit 抛异常时不触发 invalidate |
| R10 | base64 编码被视为真加密导致安全审计不过 | 中（非 Demo 阶段） | 安全合规 | 列名 `xxx_key_encoded` 明示非加密；`config_service` 文件头注释明示"base64 混淆非加密，生产环境需升级 AES-GCM"；§7 列入 TODO |
