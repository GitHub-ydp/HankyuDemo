## 0. Demo 定位（核心共识，所有决议围绕这条展开）

**这是一次"解决方案演示"，不是"生产系统上线"。**

### 0.1 客户真正在意的

| 维度 | 客户关注度 | 举例 |
|---|---|---|
| **操作流程顺畅度** | 🔴 极高 | 上传 → 解析 → 检索 → 填入 → 下载，一气呵成无卡顿 |
| **AI 确实做了处理** | 🔴 极高 | 看得到"解析中"、看得到"差异对比"、看得到"自动填入 X 条" |
| **数据真实性** | 🟡 中 | 是真的 Excel、真的字段、不是 Hello World |
| **Demo 演员的熟练度** | 🟡 中 | 演示人对系统熟、能讲清楚每一步在做什么 |
| **具体数字的精确性** | 🟢 低 | 加价比例、汇率、单日价 vs 周均价 — **大概率不较真** |
| **业务规则完备性** | 🟢 低 | "Case by case"、"Must go NN"、"已有数据不覆盖"这种边角 |

### 0.2 翻译成工程动作

- 把"看得见的流程环节"做**扎实**（Step1 上传/解析/入库/导出 + Step2 解析/检索/填入/审核）
- 把"看不见的精确规则"用**兜底值 + 话术**盖过去
- 把"楢崎可能问但我们没定的事"提前**准备标准回答**

## 1. Demo 范围冻结

### 1.1 做

- ✅ **Step1** — Air / Ocean / Ocean-NGB 三份文件全流程（上传 → 解析 → 入库 → 差异对比 → 原格式回填下载）
- ✅ **Step2** — Customer A 端到端（上传 PKG → 解析 → 检索 → 填入 → 审核 → 下载），另外 Customer B / E / Nitori 演"能识别结构"的程度，不保证填入 100% 正确

### 1.2 不做（明确砍掉）

- ❌ **Step3 个别报价制作** — 客户可能都不要，等 Step1+2 Demo 完再讨论
- ❌ 邮件自动抓取 PKG / 日本本部多段合并 / 自动发回邮件 — v2.0 范围
- ❌ Customer E / Nitori 的完整填入（132 列 / 带宏 / 内置校验）— 结构识别到即可
- ❌ 换汇自动化、加价表逐客户维度细化 — Demo 用默认兜底

### 1.3 Demo 当天呈现顺序建议

```
1. Step1 Air 上传 → 解析 → 看预览（3 min）
2. Step1 Ocean 上传 → 差异对比 → 采用（3 min）
3. Step1 原格式回填下载 → 打开 Excel 对比（2 min）
4. Step2 Customer A PKG 上传 → 解析 → 看 5 段识别（3 min）
5. Step2 自动填入 PVG 段 → 审核界面 → 改一条价 → 下载（4 min）
6. Step2 演示"支持多客户" — 拖 Customer B 进去看识别结果（2 min）
7. 答问（5-10 min）
```

总时长 ~25 分钟 + 答问。**楢崎亲自操作 2-3 个环节**（Step1 上传 + Step2 下载是最有记忆点的）。

## 3. 演示话术库

**原则**：客户问到精确值的时候，**不慌、不解释过度、引导回流程价值**。

### 3.1 客户常问 + 标准回答

| 客户可能问 | 标准回答 |
|---|---|
| "这个加价 10% 是怎么定的？" | "Demo 默认 +10% 做演示。实际上线会接阪急内部加价表，按客户 × 航司 × 目的地多维度维护，规则由阪急营业/副总最终决定。**今天重点展示的是流程**。" |
| "这个价格和我们实际报的差了 X 元？" | "Demo 数据是您 4/21 发来的样本，不是最新价。我们验证的是**系统能把数据流转跑通**，具体数值以阪急实际入库的运价为准。" |
| "Customer B 这几列为什么没填？" | "Customer B 有 132 列，Demo 里我们做了**结构识别**和主干字段填入。全量 132 列的业务含义需要您这边提供字段说明书，我们在 v2.0 完整覆盖。" |
| "Nitori 的 .xlsm 宏还能跑吗？" | "能。我们的原则是**保持文件完整性**——宏、公式、校验规则全部原样保留，只改数据格子。您打开后 Bid Lane Check 该跑还是会跑。" |
| "Air 报价为什么用首日价不用周均？" | "Demo 固定取入札期间首日。实际上线可以按**客户期望起飞日**取对应那天的价——这个规则您这边定下来我们接进去。" |
| "换汇怎么处理？" | "Demo 保留原币种输出。换汇规则（实时汇率 vs 月度结算汇率 vs 人工）属于阪急内部结算规则，上线接进来即可。" |
| "这套系统上线要多久？" | "Demo 覆盖的 Customer A 主干场景已经能跑通。从 Demo 到生产上线主要是**补齐其它客户的字段适配 + 接阪急内部字典（加价表、汇率、区域映射）**，按客户反馈的优先级排期，预计 1-2 个月。" |
| "AI 做错了怎么办？" | "所有自动填入的值都有**置信度标识**，低置信度的自动标红，要求营业审核。审核界面支持改价 + 留痕，系统不会绕过人。" |
| "数据安全吗？" | "运价数据存本地数据库，不上传任何第三方。AI 调用（Claude API）只处理解析结构，不传送业务数值明文；这个细节上线可做 on-prem 部署进一步加固。" |
| "这个 Demo 的测试数据是哪里来的？" | "您 4/21 邮件附的 3 份 Excel + 4/2 福山先生转给我们的 4 家客户 PKG 样本。" |
