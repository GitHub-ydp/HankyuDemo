## 2. pytest 本地运行状况（环境问题，记录备案）

### 2.1 直接 pytest

```
cd backend && python3 -m pytest tests/services/step1_rates/test_air_adapter.py -v
```

输出：
```
ERROR collecting tests/services/step1_rates/test_air_adapter.py
E   TypeError: dataclass() got an unexpected keyword argument 'slots'
```

原因：`entities.py` 用 `@dataclass(slots=True)`，这是 Python 3.10+ 特性。

### 2.2 用 sitecustomize monkey-patch 绕 slots

```
cd backend && PYTHONPATH=/tmp/air_test python3 -m pytest tests/services/step1_rates/test_air_adapter.py -v
```

输出：
```
E   TypeError: unsupported operand type(s) for |: 'type' and 'NoneType'
File: app/models/port.py:16  name_cn: Mapped[str | None] = ...
```

原因：测试文件从 `app.services.step1_rates.adapters.air import AirAdapter`，触发 `app/services/step1_rates/__init__.py` → `service.py` → `adapters/__init__.py` → `ocean.py` → `rate_parser.py` → `models/__init__.py` → `port.py` 整条 import 链。最末端 `port.py` 用 PEP 604 联合类型 `str | None`，要求 3.10+。

### 2.3 结论

- 本机 Python 3.9.6 下 pytest 无法 collect。属于**环境问题，不是 adapter 逻辑问题**。
- 若要在本机跑 pytest，至少需要 Python 3.10（并装齐 SQLAlchemy 等后端依赖）；当前开发机只有 Xcode 自带 3.9。
- 团队已有 `docs/windows_py310_测试复核清单.md` 约定在 Windows py3.10 环境下做最终复核；本测试采用**独立加载 + 断言脚本**替代 pytest，保证被测代码本身被真实执行。

### 7.6 单元级探针（辅助方法）

- `_to_decimal_or_dash`: `"-" → (None,True)`；`None / "" → (None,False)`；`1.52 → (Decimal,False)`；`"-1" → (Decimal(-1),False)`（注意：负数字符串不被当成 dash，这是对的 —— dash 必须是独立的 `-`，不是负号）。

- `_to_date_lenient`: `datetime/date/'YYYY-MM-DD'/'YYYY/MM/DD'/'YYYY.MM.DD'` 均能转 `date`；非法字符串返回 `(None, <原文>)` 供告警。

## 8. P2 级观察（不阻塞交付，仅备案）

### 8.1 P2a — 架构文档 §3.7 伪代码与代码实现口径不一致（文档错，代码对）

- 架构 §3.7 写 `remarks = _normalize_text(J 列)`。
- `_normalize_text` 的实现是 `re.sub(r"\s+", " ", str(value)).strip()`，会把换行替换为空格。
- 代码实际用的是 `_normalize_service_text`（`[\ \t]+` 只折叠空格+tab，保留 `\n`）。
- **结果正确**（R15 remarks `'TG663 D2457\nTG665 DAILY'` 换行保留，见表 #6），但文档措辞有误导性。建议架构大师在下一版文档中把 §3.7 的伪代码同步成 `_normalize_service_text`，或明确说明"保留换行"需要哪种 normalize。

### 8.2 P2b — 业务文档 §5.2 遗漏 VS 行四字段全 `-`

- 业务文档 §5.2 只提到 BA（第 6 行）四字段全 `-`。
- 独立扫描 Surcharges 表，发现 **VS - Virgin Atlantic Cargo（第 67 行）也是四字段全 `-`**（`extras.all_fees_dash=True`）。
- adapter 正确识别了两条（本次抽样表 #41 + #48）。
- 业务侧需要确认：VS 是否也属于"有效但无费"还是"误留应删"？（对应业务文档 Q11）这不是 adapter 的问题，是业务文档的遗漏。

### 8.3 P2c — `_to_decimal_or_dash` 对"非数字字符串"与"真空"归并

- 代码中 `_to_decimal_or_dash("abc")` 返回 `(None, False)`，与 `_to_decimal_or_dash(None)` 相同。
- 架构 §4.6 约定"三态：value/dash/empty"；实际实现是"三态：value/dash/其它"，把"非数字字符串"并入 empty 一类。
- 真实 Surcharges 表中 F/G/H/I 四列只出现过数字、`-`、真空三种，没有出现过 "abc" 这类字符串，**所以不触发偏差**。
- 若未来数据真出现"TBD"之类字符串，adapter 会当 `(None, False)` 处理，**不会抛异常，也不会记 warning**。这是 adapter 的一点容忍度，建议架构大师评估是否要补一条针对 Surcharges 的"非数字字符串"warning（类比周表 W-A06）。优先级低，不阻塞。

## 9. 没测到的项（主动声明，不自评"覆盖全面"）

- **落库后**的行为：本 adapter 只产 `ParsedRateBatch`，不写数据库。"批次覆盖 / superseded" 逻辑在批次服务层，不属于本测试范围（架构任务单 §12 也排除了）。
- **原格式回填 writer**：本轮任务单不含（§12），未测。
