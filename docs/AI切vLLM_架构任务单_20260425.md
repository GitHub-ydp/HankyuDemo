## 1. 核心设计决策（先定方向，再拆任务）

### 决策 A：`openai` SDK vs 保留 `httpx` 手写

**结论：选备选 1（保留 httpx 手写）**。

理由：
1. **依赖最小**：引入 `openai` SDK 只为三个 POST 请求不划算，且 `openai` SDK 对 `chat_template_kwargs` 这种 vLLM 扩展字段要通过 `extra_body` 桥接，阅读成本反而更高
2. **Anthropic 分支必须保留**：即便用 `openai` SDK，Anthropic 仍要靠 `anthropic` SDK（或 Bedrock 兼容模式，后者部署复杂不适合 Demo）。两套 SDK 并存反而分散注意力
3. **后续设置页接入成本**：设置页核心是"改运行时参数"（base_url / api_key / model / timeout），httpx 的构造代码就是 4 个字段的字典拼装，改起来比 SDK 直观
4. **单元测试友好**：httpx 用 `respx` / `httpx.MockTransport` 很容易 mock；SDK 要 patch 多层 client 对象

但是：保留 httpx 的前提是**把重复代码抽掉**——把现在 `chat` 和 `chat_with_image` 里两份完全相似的 "build headers / build json / raise / parse content" 流程提取为内部 `_vllm_chat_raw(messages, ...)` / `_anthropic_chat_raw(messages, ...)` 两个私有函数。public API (`chat` / `chat_with_image`) 只负责把 system/user/image 拼成 messages。这样设置页要加 `raw_chat(messages)` 入口时几乎零改动。

**最容易被反驳点**：监工可能质疑"现在有两条分支（vLLM / Anthropic）共用 4 种组合（文本/图片 × 两 provider），总共 4 份相似代码比抽象更简单"。反驳：`chat` 和 `chat_with_image` 的 request/response 构造差异仅在 `messages[0].content` 这一处（字符串 vs list），把这个差异**提前**到 public 层（构造 messages），底层 `_vllm_raw` / `_anthropic_raw` 就能完全复用。这不是"过度抽象"，是"CLAUDE.md 的三次重复阈值已达"。

### 决策 B：Provider 选择策略

**结论：混合方案（静态 env 决定默认 + 单次调用 override 口子）**。

具体：
1. `Settings` 新增 `ai_provider: str = "vllm"`（允许值：`vllm` / `anthropic`）——**默认就是 vllm**，贴合用户约束
2. `_get_provider()` 改成读 `settings.ai_provider`，不再嗅探 key（嗅探逻辑留在 `health_check()`）
3. `chat()` / `chat_with_image()` 新增 `provider: str | None = None` 参数——传 `None` 就用 settings，传 `"anthropic"` 就临时用 Anthropic（给后续设置页的"切换测试"按钮、以及未来 AB 测试留口子）
4. **本轮不实装设置页所需的运行时覆盖注册表**（避免过度设计），但 `provider` 参数的签名留位置，未来加一个 `ai_runtime_config.py` 单例保存当前 override 值，读取优先级 = call 参数 > 运行时单例 > env，三级覆盖——这些都不是本轮工作

**最容易被反驳点**："既然本轮不做设置页，为什么要加 `provider=None` 参数？不是 YAGNI 吗？" 反驳：不加这个参数，测试大师就无法用 pytest 直接测 Anthropic 分支的 timeout 修复（得改 env 重启后端），验收粒度会很粗。参数加上去是**为测试而不是为未来功能**，符合 YAGNI。

### 决策 C：vLLM 差异怎么落到代码

1. **`chat_template_kwargs.enable_thinking`**：硬编码为 `False`。参数化没意义——现在所有业务都要 JSON 输出、不要思考过程；真需要开思考时再加参数。**方式**：`_vllm_raw()` 请求体直接带 `"chat_template_kwargs": {"enable_thinking": False}`
2. **`/no_think` 后缀**：在 `_vllm_raw()` 里**按配置默认开启**（`Settings.ai_auto_no_think: bool = True`），仅当 messages 里最后一条 user 的 text 不以 `/no_think` 结尾时追加。原因：双保险写到文档里，代码层自动补全避免调用方忘记
3. **`max_tokens` 默认**：
   - `Settings.ai_max_tokens_default: int = 512`（文本轻量调用用）
   - `Settings.ai_max_tokens_extract_json: int = 1024`（邮件/图片提取用——prompt 占位 + 答复数组）
   - 调用方显式传 `max_tokens` 时按传入；否则按用途选默认
   - 超限硬上限：`Settings.ai_max_tokens_cap: int = 1536`（留 512 给 prompt，死守 2048）
4. **图片压缩**：新增 `_compress_image(path) -> (bytes, mime)`：
   - 引入 Pillow（`requirements.txt` 加 `Pillow>=10.0.0`）
   - 最长边 > 1280 → 缩至 1280（保持比例）
   - 转 JPEG，quality = `Settings.ai_image_jpeg_quality`（默认 85）
   - HEIC / GIF 等 Pillow 不直接支持的格式 → **不压缩**直接回退原字节（读 mime），在 warnings 里留 log
   - 可关：`Settings.ai_image_compress: bool = True`；设 False 时走旧路径（只给设置页预留）
   - 仅在 `chat_with_image` 调用；Anthropic 分支也走同一压缩（Anthropic base64 越小越快）
5. **prompt 超长兜底**：不在 `ai_client` 层做截断（不知道业务语义），改为**预估 token 数**（粗略 `len(system) + len(user) > 4000 chars ≈ 1500 tokens` 时）在 log 层 `logger.warning` 并把 `max_tokens` 自动降到 `min(max_tokens, 2048 - estimated_prompt_tokens - 128)`，如果压到 ≤ 64 就 raise `PromptTooLongError`。业务层（如 `email_text_parser`）看自己的 SYSTEM_PROMPT 长度自己负责（长邮件要自截断，T-AI-05 任务里覆盖）

**最容易被反驳点**："`/no_think` 自动追加会修改用户 prompt 语义，万一将来业务要思考怎么办？" 反驳：配置项 `ai_auto_no_think=False` 一行 env 就能关，默认关思考符合目前所有业务（JSON 提取不需要推理）。

### 决策 D：Anthropic 分支保留 + 补 timeout

**结论：保留 Anthropic 分支，补 timeout 用 `anthropic.Anthropic(timeout=...)` 构造参数方式**。

- `anthropic` SDK 构造函数接受 `timeout=httpx.Timeout(connect=10, read=N, write=10, pool=10)` 或简单 `timeout=N`（SDK 内部会转成 httpx.Timeout）
- `Settings.ai_timeout_seconds: int = 90` 作为**全局默认**（vllm 和 anthropic 共用）
- vllm 的 httpx.post 把 `timeout=120.0` 这个硬编码改成 `timeout=settings.ai_timeout_seconds`
- `anthropic.Anthropic(api_key=..., timeout=settings.ai_timeout_seconds, max_retries=1)`——**同时把 `max_retries` 从 SDK 默认 2 次改成 1 次**，避免单次请求最长等 270s

- `ai_client.chat()` **不做 fallback 到 Anthropic**——静默切换会掩盖问题。业务层（`email_text_parser.py:76-85`）已有 `except Exception` 兜住并写 `warnings=["AI 解析失败: ..."]`，前端显示警告即可
- 前端设置页未来可加「测试连通性」按钮调 `health_check()`；本轮不做

1. 改 `.env`：
   ```
   AI_PROVIDER=vllm          # 保持 vllm（因为百炼也用 OpenAI-compatible，可以复用 vllm 分支）
   ```

## 4. 风险清单

| # | 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|---|

| R3 | Anthropic model `claude-sonnet-4-20250514` 和 vLLM `qwen3.6-27b` 参数集不同（anthropic max_tokens 上限 8192 vs vllm 2048） | 高 | 切 anthropic 后 max_tokens=1536 太小浪费 | `_resolve_max_tokens(override, purpose, provider)` 增加 provider 参数，anthropic 分支直接返 `override or 4096`，不走 vllm 的 cap。**本版 T-AI-02 的 signature 已预留 provider，具体实现开发大师按此落** |
| R4 | `.env` 留 `VLLM_API_KEY=`（空），代码里有 `vllm_api_key: str = ""` 默认 | 中 | vLLM 401 | `_vllm_raw` 入口先 `if not settings.vllm_api_key: raise ProviderUnavailableError("VLLM_API_KEY 未配置")`。T-AI-02 需加该守卫 |
| R5 | 现 SYSTEM_PROMPT 是为 Claude 写的（中文指令精细，Claude 理解强）；qwen3.6-27b 4bit 量化可能跟不上导致 JSON 提取错误率 upward | 高 | 业务质量下降 | 本轮不改 prompt；T-AI-07 的集成测试用 3 封典型邮件 / 2 张截图做 baseline 评估；若错误率 > 10% 开新业务任务单「qwen prompt 适配」 |
