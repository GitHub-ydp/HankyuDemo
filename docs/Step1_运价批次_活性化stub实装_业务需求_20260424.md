## 1. 历史包袱澄清（最重要，放在前面）

这是业务大师在扫完代码后对两套表关系的结论。下游（架构/开发/测试）以此为准；如果结论在落地时发现不成立，必须回来找业务大师澄清，不得自行改。

### 1.1 两套表的分工（实际代码现状）

| 表 | 归属 | Step1 parser 产出的 `record_kind` | Step2 repository 现状 |
|---|---|---|---|
| `import_batches` | 新表（20260421 迁移新增） | 所有 file_type 共用的"批次表" | 查询入口：只看 `status='active'` |
| `air_freight_rates` | 新表 | `air_weekly`（Air parser 产出） | `query_air_weekly()` 已实装（T-B3） |
| `air_surcharges` | 新表 | `air_surcharge`（Air parser 产出） | `query_air_surcharges()` 已实装（T-B3） |
| `lcl_rates` | 新表 | `lcl`（Ocean parser 产出 LCL 行） | `query_lcl()` **NotImplementedError（v2.0）** |
| `freight_rates` | 老表（加了 `batch_id` FK） | `fcl`（Ocean parser）、`ocean_ngb_fcl`、`ocean_ngb_lcl`（Ocean-NGB parser） | `query_ocean_fcl()` **NotImplementedError（v2.0）** |

### 1.2 关键结论

1. **Air 激活必须写 `air_freight_rates` + `air_surcharges` + `import_batches`（active）**。这是 Demo 端到端路径的唯一硬约束——Step2 `RateMatcher` 通过 `Step1RateRepository.query_air_weekly/query_air_surcharges` 读这两张新表，且**硬过滤 `status='active'`**。不写这三张，Step2 永远 `filled=0`。
2. **Ocean / Ocean-NGB 激活写哪张表，在 v0.1 Demo 不是硬阻塞**。因为 Step2 侧 `query_ocean_fcl / query_lcl` 都是 NotImplementedError，Customer A（v1.0 唯一支持的客户）又是全 Air 场景。Ocean 激活在 Demo 里只被要求"看得到激活成功"（兜底方案 §1.3 第 2 步），不被要求"Step2 能匹配到"。
3. **但为了 Demo 叙事一致性，Ocean / Ocean-NGB 激活也必须真写数据库**（不能再 stub），否则"激活后 imported_rows=0"的 UI 矛盾会在 Demo 现场穿帮。
   - Ocean parser 产出字段（`container_20gp/40gp/40hq`、`carrier_name`、`origin_port_id/destination_port_id`、`lss_cic/baf/ebs/yas_caf`、`valid_from/to`）**正好全部对齐老 `freight_rates` 表结构**——写老表 + 带上新 `batch_id` FK + 落一条 `import_batches(active)` 记录即可。
   - Ocean-NGB parser 产出含 `rate_level`（Lv.1/Lv.2/Lv.3）——老 `freight_rates` 迁移里**恰好加了 `rate_level` 字段**，也是走老表。
4. **老表 `freight_rates` 不能直接丢弃**。原因：
   - `rate_batch_service.get_rate_batch_diff()`（165-157 行）现在用 `FreightRate` 表做差异对比，如果丢掉 `freight_rates`，diff 功能会失效。
   - 前端 `/batches` 有"差异对比"按钮（`RateBatches.tsx:313`），Demo 现场会点。
   - 因此 Ocean / Ocean-NGB 激活必须**写入老 `freight_rates` 表并带上新 `batch_id` FK**——这样 diff 依然能走、激活后的记录也能被下一批 diff 对比到。

### 1.3 关于 "两套表是否都写"

> 业务大师判断：**Air 只写新表（air_freight_rates / air_surcharges）就够**，不需要同步写老 `freight_rates`。理由：
> - Step2 Air 查询不看 `freight_rates`，只看新表。
> - `freight_rates` 的字段结构（carrier_id / 20GP/40GP/40HQ）与 Air 周表（周内 7 天价 × 目的地）**语义不匹配**，强塞进去反而产生脏数据。
> - Air 的 diff 对比业务价值低（Air 半月一变，按周更新本就是"整批替换"模式，diff 不像 Ocean 那样有"同航线涨跌"的业务意义）。
>
> **Ocean / Ocean-NGB 只写老 `freight_rates`**（不写 `lcl_rates`，即便 Ocean parser 会产 LCL 行），理由：
> - `freight_rates` 字段完整覆盖 FCL 所有费用项 + LSS/CIC/BAF/EBS/YAS/CAF，足够承载 Ocean 两种 parser 产出。
> - `lcl_rates` 新表字段（`freight_per_cbm/per_ton`）虽然和 Ocean LCL 产出匹配，但 Step2 不查它，写了也是冷数据，增加后续维护负担。
> - 若后期（v2.0）Step2 要查 Ocean，在老表上继续扩即可，不形成新的数据孤岛。
>
> 若架构大师或开发大师在实装时发现"Air 不写老表会让某处 diff/审计/历史查询坏掉"，必须回来找业务大师重新评估，**不得自行调整结论**。

## 2. 业务目标

### 2.1 一句话目标

让营业/业务人员在前端 `/batches` 点击"真激活"按钮后，**当前批次解析出的费率真的进入数据库**，Step2 立刻能查到。

### 2.2 Demo 场景串联（硬底线）

楢崎 5/14 现场演示时可依次跑完的真实路径：

1. 营业在 `/batches` 页面上传 `【Air】 Market Price updated on  Apr 20.xlsx`（兜底方案 §1.3 第 1 步）。
2. 前端显示 147 条预览（84 条 weekly + 63 条 surcharge，T-B10 实测数）。
3. 营业点"试运行"，看到"将新增 147 条"的摘要。
4. 营业点"真激活"，UI 返回"已激活，imported_rows=147，批次状态 active"。
5. 营业切到 `/pkg` 页面，上传 `Customer A 2-①.xlsx`。
6. 系统识别为 Customer A（high 置信），解析出 5 段 21 行。
7. 自动填入结果：**`filled=2`（AMS + TPE 从刚激活的批次里匹配出来）、`no_rate=3`（ATL/MIA/SYD 在库无）、`skipped=16`**。
8. 下载 cost / sr 两个 xlsx。

**V1 真端到端硬底线 = 第 7 步的 `filled=2` 必须从刚激活的批次真查出来**，不是脚本直插的数据。

### 2.3 不做什么（v0.2+）

- 不做多人审批流（兜底方案 §1.2 不做范围）。
- 不做批次回滚/撤销激活。
- 不做定时自动激活。
- 不做跨 file_type 合并激活。
- 不做 Ocean / Ocean-NGB 激活后 Step2 能查到（Step2 侧 v2.0 再接）——Ocean 激活本轮**只要求入库成功 + UI 反馈成功**。

## 3. 激活成功应达成的状态（验收观察点）

从楢崎的视角、用鼠标+肉眼能观察到的状态：

1. 前端 `/batches` 列表中，被激活的批次 `batch_status` 从 `draft` 变成 `active`（或类似明显的状态 tag）。
2. 前端抽屉"激活"面板下方的 `imported_rows` 是一个大于 0 的具体数字（对 Apr 20 Air 文件应为 147 = 84+63）。
3. 若是 Air file_type：立刻切到 `/pkg` 上传 Customer A 2-①.xlsx，能看到 `filled=2`（AMS/TPE）。
4. 若是 Ocean / Ocean-NGB：UI 返回 `activated=true` + `imported_rows=<具体数>`，不需要 Step2 匹配到。
5. 数据库 `import_batches` 表能查到一条 `status='active'` 的记录，`batch_id` 与前端 URL 上的 UUID 一致，`effective_from/to` 与文件内容匹配。

## 4. 激活的业务规则

### 4.1 老 active 批次的处理（Q-ACT-01 兜底）

**规则**：同一 `file_type`（`air` / `ocean` / `ocean_ngb`）下同一时刻**最多一个 active 批次**。

**激活新批次时**：
- 查 `import_batches` 表，找该 file_type 下所有 `status='active'` 的批次。
- 将这些老 active 批次的 `status` 全部改为 `superseded`。
- 然后将本批次 `status` 设为 `active`。
- 老 active 批次下的**明细记录不删**（`air_freight_rates` / `air_surcharges` / `freight_rates` 的实体记录保留），只是 Step2 查询因为硬过滤 `status='active'`，自然读不到 superseded 批次的数据。

**为什么这样设计**：
- 保留历史记录，允许后期（v2.0）"查上月的报价"。
- Step2 查询简单，不需要复杂的时间窗口逻辑。
- "上週 sheet（Apr 13-19）和当週（Apr 20-26）同时在一个批次里"已由 Air parser 处理（Air adapter `_parse_weekly_sheet` 产出两个 week 的 records 共存一个批次），所以"上週数据保留 vs 当週激活"不是本层关心。

**边界**：
- 不做"选择性激活"（比如只激活某一 sheet）。整批次 all-or-nothing。
- 不做"激活后 N 分钟内可撤销"。一旦激活即生效。
- 不做"一 file_type 允许多个 active"。

### 4.2 Records 的写入策略

**规则**：新批次的 records **全部写入对应明细表，不合并、不覆盖老记录、不去重**。

**Air file_type**：
- 所有 `record_kind=='air_weekly'` 的 records 写入 `air_freight_rates`。
- 所有 `record_kind=='air_surcharge'` 的 records 写入 `air_surcharges`。
- 所有写入记录带上本批次的 `batch_id` 作 FK。

**Ocean file_type**：
- 所有 `record_kind=='fcl'` 的 records 写入老 `freight_rates` 表。
- 所有 `record_kind=='lcl'` 的 records 本轮 **不写入任何表**（兜底方案 §1.2 不做范围 + Step2 不查 LCL），但需在激活响应里统计进 `skipped_rows`，并给一条 warning"LCL 暂不入库（v2.0）"。

**Ocean-NGB file_type**：
- 所有 `record_kind=='ocean_ngb_fcl'` 的 records 写入老 `freight_rates` 表（保留 `rate_level` = Lv.1/Lv.2/Lv.3 字段——这是客户楢崎日文邮件里强调的"三级分档要独立保留"的硬约束）。
- 所有 `record_kind=='ocean_ngb_lcl'` 的 records 本轮不写入（同 Ocean LCL）。

**关于"与老 active 批次的 records 是否冲突"**：
- 不冲突，因为通过 `batch_id` FK 天然隔离。
- 老 active 变成 superseded 后，它的 records 还在表里，只是 Step2 查询不到。

**边界**：
- 不去重。如果同一目的地 / 同一航司出现在老 active 和新 active 两个批次，都保留（通过 batch_id 区分）。
- 不做"增量更新"（只增不改）。

### 4.3 dry_run vs real 模式

**dry_run = true**（"试运行"按钮）：
- 不写任何数据库记录。
- 不改 `import_batches` 状态。
- 不降级老 active 批次。
- 返回"将要导入的条数摘要"——**至少含 `selected_rows`（= 会导入的条数）、`diff_summary`（复用现有 diff 逻辑，只对 Ocean 有意义；Air 可以返回 new_rows=total）**。
- UI 用途：让营业在真激活前心里有数"这次激活会影响多少条"。

**dry_run = false**（"真激活"按钮）：
- 真写数据库。
- 执行 §4.1 + §4.2 规则。
- 返回最终 `imported_rows`、`superseded_batch_ids`（被降级的老批次 UUID 列表，给前端展示"顺带降级了 X 个老批次"）、`batch_status='active'`。

**force 参数**（已存在于 API schema，保留但 v0.1 不做复杂逻辑）：
- 本轮 force=true 和 force=false 行为**一致**（都真激活）。
- force 字段保留作为未来扩展位（比如 v0.2 "在检测到冲突时仍强制激活"），本轮响应里确认"force=true 已接受但当前无差异化行为"。

### 4.4 失败回滚（Q-ACT-03 兜底）

**规则**：激活过程中任何一步失败，**整体回滚，已写入的记录全部撤销**。

**失败的业务语义**：
- 营业看到"激活失败：<原因>"的 UI 反馈。
- 数据库不留任何"半成品"记录：不留失败的 `import_batches` 行，不留已写入一半的明细记录，不降级任何老 active 批次。
- 批次状态退回 `draft`，营业可以重新点"真激活"重试。

**失败的触发场景**（至少覆盖这几种）：
- parser 产出的 records 为空（空批次不允许激活）。
- 写入数据库时违反约束（例如 FK 不存在、字段超长）。
- Ocean / Ocean-NGB 的 `origin_port_id` 或 `destination_port_id` 在 `ports` 表找不到（T-B10 报告未暴露此场景，但 Ocean parser 依赖 `ports` 查询，若 DB 没导入 port 字典会 FK 失败）。

**边界**：
- 不做"部分成功"（写入前 N 条成功，第 N+1 条失败时保留前 N 条）。楢崎业务视角看，"激活成功"等于"整批次可用"，半成品状态会让营业误判。
- 不做自动重试。

### 4.5 空批次拒绝

**规则**：`records` 为空（parser 产出 0 行）时**禁止激活**。

- UI 反馈：`"该批次无可入库数据（0 条），请检查源文件。"`
- 不写 `import_batches` 记录。
- 批次状态维持 `draft`。

**理由**：一个 0 行的 active 批次没有业务意义，只会让 Step2 查到空结果而不自知。

### 4.6 重复激活同一批次

**规则**：已经 `status='active'` 的批次**不允许再次调用 activate**（幂等保护）。

- UI 反馈：`"该批次已激活，当前状态 active。"`
- 不做任何变更。

### 4.7 effective_from / effective_to 的处理

**规则**：`import_batches.effective_from / effective_to` 从 parser 产出的 `ParsedRateBatch.effective_from / to` 直接复制。

- Air parser：取"最新一週 sheet 的 week_start/end"（air.py:102-106 已实现）。
- Ocean parser：取文件里的生效区间（ocean.py 里每行有 `valid_from/to`，批次级取最宽范围；如果找不到，置 NULL）。
- Ocean-NGB parser：同 Ocean。

**业务提醒给下游**：Step2 `RateMatcher` 查 Air 时硬过滤 `effective_week_start <= effective_on <= effective_week_end`（`rate_repository.py:60-62`），其中 `effective_on` 来自 `bidding_orchestrator.py:100` `effective_on or datetime.utcnow().date()`——**Demo 当天是 2026-05-14，而 Apr 20 Air 文件的周窗口是 2026-04-20~04-26，utcnow 查不到**。这是一个独立于激活的隐藏风险（见 §9 风险 R-03），不在本需求范围内，但架构大师做实装前应该把这个风险**向张东旭同步**。

### 5.1 Port / Carrier 字典查询缺失的处理

- Ocean / Ocean-NGB parser 产出的 `origin_port_name / destination_port_name / carrier_name` 可能在 `ports` / `carriers` 字典表找不到对应 id。
- **规则**：找不到时整批激活失败（§4.4），UI 提示"第 N 行：港口 'XXX' 在字典表未找到，请先录入"。
- 不自动创建 port/carrier 字典记录（避免脏数据）。
- 实际上本轮 Demo 走的是 Air 路径，不触发此路径；但 Ocean / Ocean-NGB 文件激活会触发，需要提前规划。

## 7. 失败路径清单

| 编号 | 场景 | 业务反馈 | 数据库状态 |
|---|---|---|---|
| F-ACT-01 | 批次不存在（batch_id 无效） | HTTP 404 + "批次 {id} 不存在" | 无变更 |

## 9. 风险与开放问题

### 9.1 需要监工/张东旭/架构大师拍板的

- **R-01（高）· effective_on 默认值问题（与本需求正交但会影响 Demo 现场）**
  - 事实：`bidding_orchestrator.py:100` `effective_on = effective_on or datetime.utcnow().date()`。
  - Demo 当天是 2026-05-14，但 Apr 20 Air 文件覆盖的周窗口是 2026-04-20~26，`utcnow()` 查不到任何记录，即使激活成功 Step2 仍 `filled=0`。
  - **本需求不解决此问题**，但激活实装做完、Demo 彩排时一定会撞上。
  - **建议**：在激活完成后，张东旭与架构大师决定：
    - (a) 改 `effective_on` 默认值为"最新 active 批次的 effective_to"（侵入 Step2）；
    - (b) Demo 现场用假时间/改系统时间绕过；
    - (c) 临时准备 5/14 当週的 Air 文件（T+24h 内客户能不能给？未必）。
  - 业务大师**偏向 (a)**，最小改动、符合"激活后立刻能用"的用户预期。

- **R-02（中）· draft 批次在内存不持久化**
  - 事实：`_draft_batches` 是进程内字典，服务重启即失。
  - 本需求**不包含**"draft 持久化"，但 Demo 若营业上传后等太久 / 服务意外重启，激活时会 F-ACT-06。
  - **建议**：Demo 当天执行纪律"上传后 1 分钟内激活，不要关浏览器"，走人力工序绕过。

- **R-03（中）· Ocean / Ocean-NGB 的 Port / Carrier 字典未预填**
  - 事实：Ocean parser 用 `origin_port_id / destination_port_id`，需要 `ports` 表先有数据。T-B10 报告未提及，开发大师需确认现状。
  - **建议**：架构大师在拆任务时先核查 `ports` / `carriers` 表当前是否已经预置了 4/21 三份文件里出现的所有港口/船司。如果没有，本需求前置依赖多一条"字典预填"。
