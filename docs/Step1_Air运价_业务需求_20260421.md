## 2. 工作簿结构

### 2.1 `Apr 13 to Apr 19` — 上周周表（历史）

- 结构与本周表完全一致（两个 sheet `max_row=46, max_col=14`，表头均为第 1 行）。
- ❓ 待楢崎确认：同一份文件每次上传都包含"上周 + 本周"两张表，是否意味着上周表应作为历史档永久留存？还是仅用于人眼对比，不入库？

### 2.2 `Apr 20 to Apr 26` — 本周周表（主数据）

- 业务含义：本周生效的航空市场价，本周对外报价以此表为准。
- 有效周窗口：由 sheet 名称决定（`Apr 20 to Apr 26` = 2026-04-20 至 2026-04-26，7 天）。
- 7 天 7 列日报价（出处：`Apr 20 to Apr 26!C1:I1` 列头 `2026/4/20 (Mon) … 2026/4/26 (Sun)`）。

### 2.3 `Surcharges` — 航司附加费表

- 业务含义：每个航司按"目的地区域"收取的两类航空附加费（MYC / MSC）清单；报价 = 周表基础价 + 本表对应航司 × 目的地区域的附加费。
- 结构（出处：原件扫描）：
  - `F2 = "CURRENCY : CNY"`——币种声明写在单元格里，不是列头
  - A 列整列空，数据从 B 列开始
  - 表头在第 4 行：`B4=AREA | C4=FROM | D4=AIRLINES | E4=EFFECTIVE DATE | F4=MYC MIN | G4=MYC FEE/kg | H4=MSC Min | I4=MSC FEE/KG | J4=DESTINATION | K4=REMARKS`
  - `C5:C67` 是一个大纵向合并单元格（统一 `CHINA / SHA`，含"注意生效"竖排汉字）
  - 数据行：第 5–67 行，共 63 条

## 3. 周表业务规则

### 3.1 一行 = 一条业务报价记录

- 业务对象：一行 = `(目的地机场代码, 服务描述/航司组合/直达或中转标识)`。
- 抽样出处：
  - 第 14 行：`Destinations=BKK, Service=CK direct 2 days, 7 日报价 15.5/15.5/18/18/18/18/18, Remark="CK D357"`
  - 第 16 行：`Destinations=BKK, Service="Transfer 3 days SQ/RH", 7 日报价 16×7, Remark="BFS TERMINAL"`
  - 第 17 行：`Destinations=BKK, Service="Transfer NH 2 to 3 days", 7 日报价 14×7, Remark="EES：1ST D5+2ND D5（1M）"`
  - BKK 一个目的地共 4 行（第 14–17 行），因服务方案不同
- 业务规则：同一目的地下多行必须独立留存，不可合并。

### 3.2 目的地列的多种写法

- 绝大多数用 IATA 三字代码：`NRT / KIX / NGO / FUK / BKK / SIN / HKG / TPE / KUL / ICN / HAN / KTI / SGN / JKT / JNB / DEL / BOM / MAA / JFK / LAX / ORD`
- 第 33 行：`DXB/DWC` —— 一行涵盖迪拜两个机场
- 第 40–43 行：`FRA DENSE / FRA VOLUME / AMS DENSE / AMS VOLUME` —— 机场 + 货物密度类型
- 业务规则：三种形态并存，必须按原文保留。

### 3.3 服务/航司列（`Service/+100KG`）

- 列头原文：`Service/+100KG`（出处：`Apr 20 to Apr 26!B1`）
- 含义拆解：
  - `+100KG`：这张表给出的单价是每公斤单价，基于 ≥100 kg 档（FAQ Q3 指出"空运按 kg 计费"）。<100 kg 小票件不在本表覆盖范围。
  - `Service`：半结构化文本，混合航司（可能多个）、直达或中转、天数、MAWB 主体等。典型写法：
    - `CK 2 days service`（第 2 行）
    - `NH 2 days service HHE MAWB` / `NH 2 days service EES MAWB`（第 7、8 行：同航司按主单号分两条）
    - `Transfer 3 days SQ/RH`（第 16 行：中转、多航司任选）
    - `XJ/CI/SQ/NH 2-3days service`（第 34 行：多航司任选 + 天数区间）
- 业务规则：Service 列是半结构化自由文本，不可强行按单一维度拆分。
- ❓ 待楢崎确认：营业是否需要按单个航司代码过滤？（若需要，必须约定多航司串的识别规则）

### 3.4 7 天 7 列日报价

- 一周内价格可能多档（抽样证据）：
  - 第 14 行 BKK CK 直飞：`15.5/15.5/18/18/18/18/18`——周三起涨
  - 第 32 行 JNB SQ：`45/45/45/48/48/48/48`——周四起涨
  - 第 37 行 JFK：`55/55/63/63/63/55/55`——周三至五尖峰
- 业务规则：必须按日保留；不可取均价或首日价。
- ❓ 待楢崎确认：若某日单元格为空，是"当日不接单"还是"沿用前一日"？（本轮抽样未出现空日价）

### 3.5 Remark 列（重点）

- 列头 `Remark (Selling)`（出处：`Apr 20 to Apr 26!J1`）——给营业/销售看的备注。
- 实际分类与出处：

  | 分类 | 原文样本 + 行号 | 业务含义（基于上下文推断） |
  |---|---|---|
  | 航班号/班期限定 | `J14="CK D357"` | CK 航司仅 D357（周三五日）飞此线 |
  | | `J15="TG663 D2457"` + 换行 `TG665 DAILY` | TG663 每周二四五日，TG665 每天 |
  | | `J25="D34567"` | 仅周三四五六日有班 |
  | | `J26="CX 1ST/D246 2ND/D357"` | CX 第一段 D246、第二段 D357 |
  | "Must go" 最低价 | `J18="Must go 22"` (SIN CK)、`J19="Must go 22"` (SIN SQ)、`J20="Must go 9.5"`、`J21="Must go 20.5"`、`J24="Must go 14.25"` | 推测为"对外最低报价 NN（CNY/kg）" |
  | 航站楼/仓位限定 | `J16="BFS TERMINAL"` | 该条仅走 BFS 航站楼 |
  | 货物限制 | `J3="Not accept battery goods"` | 不收电池类货物 |
  | 逐单定价提示 | `J27="Case by case"` 等 | 不能直接按表内价报出，需回购买部门确认（远程线、美线、欧线大量出现） |
  | 班期频率 | `J7="every day, case by case"`、`J30="MU DAILY/GA DAILY"` | 频次说明 |
  | 折扣/补充条款 | `J17="EES：1ST D5+2ND D5（1M）"` | ❓ 具体含义待楢崎确认 |

- 业务规则：Remark 是对外报价的硬约束条件，必须保留原文可回溯；"Must go"决定报价下限；"Case by case"禁止直接引用表内价格。
- ❓ 待楢崎确认："Must go NN"是否等于"对外最低报价"。

### 3.6 数据区范围

- 数据行：第 2–43 行，共 42 条（第 44 行起全空）。
- 目的地覆盖：日本、东南亚、东北亚、南亚、中东、非洲、北美、欧洲。

## 4. Surcharges 业务规则

### 4.1 币种声明的特殊位置

- 事实：`F2 = "CURRENCY : CNY"`（出处：`Surcharges!F2`）。
- 业务规则：本表四个金额字段（MYC MIN / MYC FEE/kg / MSC Min / MSC FEE/KG）统一以 CNY 计价；周表单价无币种声明，❓ 待楢崎确认周表币种。

### 4.2 `AREA` 与 `FROM` 的纵向合并

- 事实：`AREA` 所有数据行值均为 `TC-3`；`C5:C67` 是合并单元格，统一为 `CHINA / SHA`。
- 含义：TC-3 = IATA 空运区划里的亚太区；整张表服务于"中国上海起飞"场景。
- ❓ 待楢崎确认：未来是否会出现第二个 AREA（如 TC-1）或第二个 FROM（如 PVG 分开列）？

### 4.3 `DESTINATION` 列

- 含义：附加费针对的目的地区域（非单机场，而是 IATA TC 区域或自定义区域名）。
- 抽样出处：
  - `J5="TC-1,2"`
  - `J11="Japan"`、`J15="Taiwan"`、`J33="Korea"`
  - `J17="North & South America,UK, Europe & South …"`
  - `J45="ANZ:AU/NZ/POM/NAN"`
  - `J49="INDIAN-SUB : IN/BD/LK/NP/PK"`
- 业务规则：同航司可能有多条不同区域的附加费（MH 共 7 条，出处第 45–51 行）；必须按目的地机场所属区域匹配。
- ❓ 待楢崎确认：区域名受控词表由谁维护？

### 4.4 `EFFECTIVE DATE`

- 含义：生效起始日（无截止日列）。
- 抽样出处：
  - 最近：`E17=2026-04-16`（CX）、`E45=2026-04-16`（MH）
  - 4 月初批量：`E7..E13=2026-04-01`
  - 老日期：`E6=2015-11-01`（BA）、`E26=2012-07-16`（JL TC-1,2）、`E59=2015-06-01`（PO）
- 业务规则：航司附加费一旦定下来可多年不变；老日期代表"长期有效"不代表"过期"；不物理删除。
- ❓ 待楢崎确认：新生效日到来时，旧条留档还是覆盖？

### 4.5 MYC 与 MSC

- 事实：表中同时出现 `MYC MIN / MYC FEE/kg` 与 `MSC Min / MSC FEE/KG`（出处：`Surcharges!F4:I4`）。
- 行业推断（非文件明示）：MYC 通常对应燃油附加费，MSC 通常对应航空安检/战险附加费；REMARKS 普遍写 `"levied on CHG-WT."`（按计费重量收）。
- 业务规则：两笔附加费独立累加，`实际收费 = max(MIN, FEE_per_kg × 计费重量)`。
- ❓ 待楢崎确认（强烈建议必问）：MYC / MSC 是否对应"燃油附加费 / 航空安检附加费"？FAQ 未记录。

### 4.6 缺省值 `-`

- 抽样：`F5="-"`、`G6="-"/H6="-"/I6="-"`（BA 行四字段全是 `-`）、`F20="-"`、`G39=13.1 + H39="-" + I39="-"`（LH 只有 MYC 不收 MSC）。
- 业务规则（推断）：`-` 表示"不适用/不征收/无保底"，不是待填。
- ❓ 待楢崎确认：BA 这种 4 字段全 `-` 的条目是"有效但无费"还是"误留应删"？`-` 与真空值（如 Y8 TC-1 条目的 F66/H66）是否有业务区别？

## 5. 业务异常场景

### 5.1 周表侧

| 异常场景 | 本轮是否出现 | 业务处置 |
|---|---|---|
| 周表某日单元格为空 | 未出现（42 行 × 7 日均有数字） | ❓ 待楢崎确认 |
| 同一目的地多行 | 出现（BKK 4 行、NRT 4 行、NGO 3 行等） | 全部保留 |

| Remark 含 "Must go NN" | 出现（第 18–21、24 行） | 保留并作为最低价业务提示 |
| Remark 含 "Case by case" | 出现（第 22、27、32–43 行） | 保留；不能直接引用表内价 |

### 5.2 Surcharges 侧

| 异常场景 | 本轮是否出现 | 业务处置 |

- 事实：原件含两张同结构周表（Apr 13-19 和 Apr 20-26）。

## 6. 与下游衔接

### 6.1 Step1 范围内

- 检索（Step1 §8.1 `/rates` 页）：至少按"起运 / 目的 / 承运人"过滤；Air 的承运人即航司，但分布在两处（周表 Service 列半结构化 + Surcharges AIRLINES 列）；业务上最小可用维度是"目的地机场代码 + 服务描述关键字"。

### 6.2 为 Step2 预留口子（本轮不实现）

- Step2 里 Air 的预期使用路径：
  1. 按目的机场找行
  2. 按航司偏好从 Service 列筛选
  3. 按期望起飞日取对应列单价
  4. 叠加 Surcharges 对应航司 × 目的地区域的 MYC/MSC
  5. 输出"每公斤单价 + 附加费明细"
- 业务口子：
  - 目的地机场必须可精确匹配（`DXB/DWC`、`FRA DENSE` 等非单一机场要约定展开规则）
  - 航司信息必须可关键词命中
  - 区域名到机场的映射表（Japan → NRT/KIX/NGO/FUK 等）必须存在
- ❓ 待楢崎确认：区域 → 机场列表由谁提供？

### 6.3 每周更新机制

- 批次键：`(file_type=air, effective_from=本周起始, effective_to=本周结束)`。
- 新批次 active、旧同键批次 superseded、不物理删除。
- 业务观察：Surcharges 是航司级长期表（老生效日 2012 年），与周表节奏不同；架构大师需判断是否分开批次节奏（具体技术方案交架构大师）。

## 9. 架构大师做 Air adapter 最易踩的 2 条业务坑

1. **把"目的地机场代码"当成干净单值字段处理** — 真实数据里存在 `DXB/DWC`（双机场共价）、`FRA DENSE` / `FRA VOLUME`（机场+密度分行）、以及 Surcharges 的区域名（Japan / ASEAN / TC-1,2 / INDIAN-SUB …）。若 adapter 一律归一化为机场三字码，42 行里至少 5 行会被错误归一化，Surcharges 与周表的关联也会全部断掉。业务上必须保留"目的地原文 + 机场代码集合"两层信息。
