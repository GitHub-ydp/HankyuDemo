## TD-1 Ocean parser 未保留 `column_index_map`，writer FCL 数值区无法回填入库数据

- **起因**：Step1 Ocean parser（`backend/app/services/step1_rates/adapters/ocean.py`）在 parse 时在内存里构建 `layout` 列映射，但**没有把该映射写入 `record.extras`**；下游 writer 拿不到列位置。
- **writer 当前行为（本轮按任务单 §8.1 兜底走完，已上线）**：
  - LCL sheet：freight_raw / lss_raw / remarks 3 字段能回填
  - FCL JP / FCL Other sheet：**数值区（20FT / 40FT / 40HQ 运费 + BAF / LSS / CIC / CAF 等附加费）完全不触碰**，回填文件里这些 cell 仍是原件模板原值
  - Remark 列（col 18）能回填
- **业务影响（Demo 级）**：
  - 如果用户上传 Ocean 文件 → 入库 → **在系统里编辑 FCL 费率** → 下载回填文件，**Excel 里看到的 FCL 费率仍是原件模板的旧值**，不是编辑后的新值
  - 如果用户只走"上传 → 入库 → 原样下载"，看不出问题（因为模板原值 = 入库来源）
  - **5/14 Demo 风险场景**：楢崎若在演示时临场改一条 FCL 费率再点下载，会发现改动没落到 Excel（LCL 会落，FCL 不会落）
- **触发条件（Demo 必中）**：
  - Demo 本来就不演"系统里编辑费率"功能（Step1 只做解析 + 入库 + 检索 + 原格式回填），所以**触发概率低**
  - 但若楢崎问"如果我改了一条 FCL 费率能下载出来吗"，答案是：需要 TD-1 回炉后才支持
- **回炉方案**：
  1. Ocean parser 的 `_build_fcl_row_payload` 在 `extras` 增加 `column_index_map`（键：field 名；值：1-based column index）
  2. writer `ocean.py` 添加 FCL 数值回填分支：按 `extras.column_index_map` 定位 cell，按 entity 字段回填
  3. 新增 pytest：Ocean FCL round-trip（上传 → 入库 → 手动改数据库一条费率 → writer → 下载 → 验证 cell 值）
  4. 估算：0.5 人天
- **回炉时机**：建议在 Demo 前完成（作为"完整性保障"），或楢崎明确答复 Demo 不演"改费率再下载"后可延后
- **相关测试**：`backend/tests/services/step1_rates/writers/test_ocean_writer.py`（本轮测试走"集合不变"断言，没抓到 FCL 数值回填缺失）

## TD-3 `Content-Disposition filename` 对中文方括号 `【】` 只给了 `filename*=UTF-8''` 编码

- **起因**：writer 下载 API 用 `filename*=UTF-8''{encoded}` 传中文文件名；旧浏览器（如 IE11）显示为 URL-encoded 形式
- **业务影响**：Demo 机器是现代浏览器（Chrome/Edge/Safari），**无影响**
- **回炉方案**：同时返回 `filename="..."`（ASCII fallback）+ `filename*=UTF-8''{encoded}`
- **回炉时机**：可推迟到 v2.0；非 Demo 阻塞

## TD-5 NGB Parser LCL Lv.3 fallback 公式与真实手输不一致（潜在路径）

- **起因**：`backend/app/services/step1_rates/adapters/ocean_ngb.py:_compute_lv_fallback` 对 LCL Lv.3 假设 `ROUNDUP(Lv.1 × 1.2, -1)`，与 FCL 同套公式
- **真实情况**：实测 4 月 NGB 文件，LCL 7 组的 Lv.3（AO/AP 列）全部是**手输硬编码 20**，不是公式；Lv.2 是 `=ROUNDUP(AO{i}*1.1,-1)` 公式
- **当前是否触发**：**不触发**。真实文件 `metadata.formula_fallback_count == 0`，data_only 路径成功拿到所有缓存值
- **业务影响**：若未来某次 LCL Lv.3 单元格未缓存公式且为 None（极小概率，因为它是手输不是公式），fallback 会算出 ~10 而真值是 20
- **回炉方案**：
  - 短期：LCL Lv.3 fallback 触发时额外打 warning（`W-N08-LCL-LV3`），让人工复核
  - 长期：业务侧问楢崎 LCL Lv.3 = 20 的口径来自哪里（手输固定值？还是另一套加价规则？），按答复重写 LCL fallback 算法
- **相关问题**：建议追加 `E-Q-NGB-7`（楢崎确认 LCL Lv.3 业务规则）
- **来源**：测试大师独立验证（2026-04-23），证据见 `/tmp/check_lcl_formula.py`
