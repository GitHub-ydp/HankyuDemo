## 0. 一句话业务定义

> **NGB（Ningbo / 宁波）运价表** = 阪急阪神**宁波**法人（HHENGB）从船司 / Co-loader 拿到的 **FCL+LCL 运价**，按"船社 × 揚港（=Place of Delivery）"做成主数据，并且每条主数据**派生 3 行**：Lv.1（采购成本价）/ Lv.2（成本 ×1.1 向上取整十）/ Lv.3（成本 ×1.2 向上取整十）。

- 业务出处：`docs/Step1_原格式回填Writer_业务需求_20260422.md` §4.3 第 184 行实证："Lv.2 = `=ROUNDUP(R2*1.1,-1)`、Lv.3 = `=ROUNDUP(R2*1.2,-1)`、其他列用 `=A2 / =E2 / =V3` 把 Lv.1 复制到 Lv.2/Lv.3"。
- 真实样本对此完全吻合：`Rate!R3="=ROUNDUP(R2*1.1,-1)"`、`Rate!R4="=ROUNDUP(R2*1.2,-1)"`（实测）。
- "Lv.1 是采购成本价、Lv.2/Lv.3 是 +10% / +20% 报价档"——`docs/Step1_原格式回填Writer_业务需求_20260422.md` §4.3 第 185 行明示。

## 1. NGB 表的业务定位（与 Sea Net Rate 的差异）

> 客户给的不是一套表，是两套表：`【Ocean】 Sea Net Rate_2026_Apr.21 - Apr.30.xlsx`（上海发的"Net Rate"）和 `【Ocean-NGB】 Ocean FCL rate sheet HHENGB 2026 APR.xlsx`（宁波发的 NGB 表）。**它们不是冗余，是两个法人各自的运价底表**。

### 1.1 五条业务差异（实证清单）

| 维度 | 【Ocean】Sea Net Rate（HHE/SHA） | 【Ocean-NGB】（HHE/NGB） | 出处 |
|---|---|---|---|
| **责任法人** | HHE/SHA（阪急阪神**上海**） | HHE/NGB（阪急阪神**宁波**） | `Rate!A2..A104` 全部为 `'HHE/NGB'`；Sea Net Rate 表头 `JP N RATE FCL & LCL!A2='HHE/SHA  Sea Export Net rate'`（写于 Writer 业需 §3-R1 第 93 行） |
| **价格性质** | "Net Rate"（采购净价，不含分级报价档） | **三档价**（Lv.1 采购成本 / Lv.2 +10% / Lv.3 +20%） | `Rate!B2='Lv.1', B3='Lv.2', B4='Lv.3'`；列头 `B1='P/S rate or Net Net'`，业务含义"P/S rate（销售价）或 Net Net（成本价）"两选一 |
| **更新频率** | 每月 2 次（月底/月初 + 月中） | 每月 1 次 | Step1 实施指令 §2 第 53–54 行 |
| **起运港覆盖** | 多港（其他港 sheet）+ 日本主力 + LCL | **宁波周边为主**：NINGBO（默认）/ ZHAPU / ZHOUSHAN | `Rate!I2..I104` 实测仅出现 3 个起运港 |
| **目的港覆盖** | 全球（日本 + 其他全球港 + LCL） | **仅日本 10 个港口**（TOKYO / YOKOHAMA / NAGOYA / OSAKA / KOBE / HAKATA / MOJ(I) / HIROSHIMA / SHIMIZU） | `Rate!N2..N104` 实测，去重后 10 个港口名 |

### 1.2 为什么客户给两套表（业务理由，非客户原话）

- 阪急阪神在中国有**上海法人**和**宁波法人**两个独立运价主体，各自和船司谈价。
- NGB 表里 `Agent = HHE/NGB`（实测 `Rate!A2`），且 NGB 表把 **Lv.1/2/3 三档**做进去，意味着 NGB 营业拿到的是"采购价 + 已经算好的对外报价档"——这与 Sea Net Rate 上海版"只给 Net 价、由营业自己加价"是两种维护模式。
- 业务推断（❓ 待楢崎确认）：两表是否会合并 / 互相覆盖？本轮 Step1 实施指令 §2 把它们当**两个独立 file_type** 处理（`ocean` / `ocean_ngb`），无合并需求；本轮 parser 也按"独立两条产线"理解，不互查。

## 2. 工作簿 sheet 清单（实证）

| Sheet 名 | 真实尺寸 | 业务用途 | 本轮 parser 是否解析 | 出处 |
|---|---|---|---|---|
| `sample` | 1064 行 × 55 列（实测仅 20 行非空） | **填表示例 / 模板说明**：第 1 行是日文表头（`'船社' / '揚港' / '荷渡地' / 'コンテナタイプ'` 等），第 3 行是英文表头复制；下面 17 行是空白模板 | **不解析**（Step1 实施指令 §3.3 第 110 行明示"跳过不导"） | `sample!D1='有効\n期限'`、`sample!F1='船社'`、`sample!K1='揚港...'` |
| `Rate` | 104 行 × 55 列（数据 99 行 + 1 行表头 + 4 个空行） | **真实费率主数据**，本轮 parser **唯一处理目标** | **是** | Step1 实施指令 §3.3 第 111 行"只处理这个" |
| `Shipping line name` | 46 行 × 12 列 | **船社 / Co-loader 字典**：列头 `A5='船会社名' / B5='略称' / C5='URL' / H5='備考'`，共 40 家船司中文名 + 略称（如 `'COSCO CONTAINER LINES' / 'COS'`、`'OCEAN NETWORK EXPRESS' / 'ONE'`、`'SITC CONTAINER LINES' / 'SIT'`） | **不入主表**；Step1 实施指令 §3.3 第 112 行"船司字典，可作为 lookup，不入主表" | `Shipping line name!A2='＊船社・CO-LOADER一覧'` |

### 2.1 三个 sheet 的"日文 vs 英文"事实

- `sample` sheet 表头是**日文**（`'船社' / '揚港' / '荷渡地' / '有効期限' / 'コンテナタイプ' / '備考'`，实测 `sample!F1='船社'`）。
- `Rate` sheet 表头是**英文**（`'Agent' / 'P/S rate or Net Net' / 'Shipping line / co-loader' / 'Port of Discharge' / 'Place of Delivery' / 'Type of container' / 'Remarks'`，实测 `Rate!A1='Agent'`）。
- `Shipping line name` sheet 是**日文**（`'船会社名' / '略称' / '備考'`）。
- 业务含义：sample 和字典 sheet 是"内部维护"用的日文，Rate 是"对外可交付"的英文；这意味着 Rate sheet 的列名英文化是有意为之，不是翻译错误。
- 业务约束：parser 只读 `Rate` 的英文表头；不要尝试用 sample 或 Shipping line name 的字段去校验 Rate（它们字段集不一致——sample 有 17 列、Rate 有 55 列，列序不同）。

## 4. NGB 与 Sea Net Rate 不同的特殊业务规则（必须列清楚）

### 4.1 R-NGB-1：Lv.1 / Lv.2 / Lv.3 三行一组，必须作为 3 条独立记录入库

- **业务规则**：每个 `(Shipping line, Origin port, Place of Delivery, FCL/LCL)` 组合在 Rate sheet 里**占连续 3 行**：Lv.1（行 N）/ Lv.2（行 N+1）/ Lv.3（行 N+2），由 B 列的 `Lv.1 / Lv.2 / Lv.3` 区分。
- **实证**：本表共 99 行数据，分成 **33 组**，每组刚好 3 行（验证脚本输出 `groups: 33, group sizes: [3], non-3 groups: 0`）。
- **业务出处**：Step1 实施指令 §1.1 第 35 行"分级报价 Lv.1 / Lv.2 / Lv.3 作为 Ocean-NGB 的一等数据保留"；§3.3 第 116 行"同一条航线有 Lv.1/Lv.2/Lv.3 三行，必须作为不同等级保留"；Writer 业需 §4.3 第 183 行同义。
- **不允许**：把三行折叠为一条带数组字段的记录（这会破坏 Writer 端"按行回填"的对称性，Writer 业需 §10 坑 1 第 313 行）。

### 4.2 R-NGB-2：Lv.2 / Lv.3 的运费值在原件里是 Excel 公式，parser 必须按"已计算结果"取值

- **业务规则**：parser 应当**取公式计算后的数值**入库（即 `data_only=True` 模式取得的值），而**不是**记录公式字符串。
  - 实证：行 3 的 R/S/T 公式 `=ROUNDUP(R2*1.1,-1)` / `=ROUNDUP(S2*1.1,-1)` / `=ROUNDUP(T2*1.1,-1)`，data_only 模式实测分别为 `100 / 170 / 170`（Lv.1 是 90/150/150，乘 1.1 = 99/165/165，向上取整十 = 100/170/170）。
- **业务理由**：营业部门做 Step2 报价时，需要直接读到"对外报价档的具体数字"；如果 parser 入库的是公式字符串，下游一切检索 / 比价 / 异常值校验都做不了。
- **Writer 端的 1687 公式不在 parser 责任内**：B-Q-W4 已关闭，Writer 端通过"模板法 load → 不对公式 cell 赋值"自动保留公式（`docs/待楢崎确认问题清单_20260422.md` 第 52 行；Writer 业需 §10 坑 1）。**parser 端无需保留公式**——parser 只输出"数值结果"，Writer 端从模板独立保留公式形态，两侧不冲突。
- **风险**：若原件保存时 Excel 没把公式结果缓存（即 data_only 模式取到 None），parser 必须能 fallback 到"按业务规则现场计算"——`Lv.2 = ROUNDUP(Lv.1 * 1.1, -1)`、`Lv.3 = ROUNDUP(Lv.1 * 1.2, -1)`。**❓ 待楢崎确认 Q-NGB-1**：fallback 计算结果若与未来某次原件实测不一致（即客户手工把公式改成了别的乘数），以哪个为准？默认值：以**原件 data_only 值**为准；若 None 才用业务规则 fallback，并打 warning。

### 4.3 R-NGB-3：表无合并单元格、无冻结窗口

- **实证**：`Rate!freeze_panes=None`、`Rate!merged_cells.ranges=[]`（空列表）。
- **业务含义**：与 Sea Net Rate 的 JP sheet"2 行配对 + B 列 Destination 合并"完全不同——NGB 没有合并、没有"配对"概念，**每行独立**。Lv.1/2/3 三行的"分组"是**靠 B 列值识别**，不是靠合并范围识别。
- **业务约束**：parser 不可假设有合并；也不可借用 Sea Net Rate 的"按合并块切组"思路（Step1 实施指令 §3.2 第 91 行讲的是 Ocean，不适用 NGB）。

### 4.4 R-NGB-4：Lv.2/Lv.3 行的"非运费列"普遍是 `=<Lv.1 同列>` 公式

- **实证**：行 3 / 行 4（Lv.2/Lv.3）的 A/E/F/G/H/I/K/L/M/N/P/Q/U/V/W/X/Y/Z/AA/AB/AD/AF 等列大多是 `=<Lv.1 cell>` 形式（如 `A3='=A2', E3='=E2', F3='=F2'`）。
- **业务含义**：原件用公式确保"改 Lv.1 一处，Lv.2/Lv.3 联动"。
- **parser 行为**：与 R-NGB-2 同——取 data_only 值，把每行展开为完整记录（每行都有完整的 Agent / Origin port / Place of Delivery / 等字段值），**不要**把 Lv.2/Lv.3 写成"继承自 Lv.1，缺字段"的稀疏记录。
- **少数例外**：实测 `AC3=400`（DOC 费 Lv.2 直接写数字 400 而不是 `=AC2`），`AC2=AC3=AC4=400`——值相同但不走公式。**parser 无须区分**，统一按 data_only 值处理即可。

### 4.5 R-NGB-5：起运港 NINGBO / ZHAPU / ZHOUSHAN 三种，必须按原文保留

- **实证**：`I2='NINGBO'`（行 2–37 大多）；`I43='ZHAPU'`（NOS 船司 ZHAPU 港）；`I68='ZHOUSHAN'`（TCLC 船司 ZHOUSHAN 港）。
- **业务含义**：宁波周边三个港口都属于"宁波法人"覆盖范围，但**起运港不同**，对客户意味着上门提货 / 报关地不同，**不能合并**。
- **parser 行为**：保留 I 列原文；如果 I 列是公式（如 `I3='=I2'`），data_only 模式应取到 `'NINGBO'` 字符串。

### 4.6 R-NGB-6：4 个空行（41/42/67/83）必须跳过

- **实证**：`Rate` sheet 第 41、42、67、83 行整行 1–55 列全空。
- **业务含义**：原件里手工留白的"组分隔"——41/42 行分隔 NOS HIROSHIMA 段和 NOS TOKYO 段（船司虽同但起运港变 ZHAPU）；67 行分隔 NOS（ZHAPU 起）和 TCLC（ZHOUSHAN 起）；83 行分隔 FCL 全段和 LCL 全段。
- **parser 行为**：跳过空行；**禁止**把空行当作"分组哨兵"产出空记录。

### 4.7 R-NGB-7：FCL 段与 LCL 段的列布局不同（同表两套字段集）

- **业务规则**：E 列 = `'FCL'` 时，主运费在 R/S/T（柜型 3 列）；E 列 = `'LCL'` 时，主运费在 AO/AP（按方/按吨 2 列），R/S/T **空**。
- **实证**：行 2 (`E2='FCL'`) `R2=90, S2=150, T2=150, AO2/AP2=None`；行 84 (`E84='LCL'`) `R84/S84/T84=None, AO84=5, AP84=5`。
- **parser 行为**：按 E 列值分流——FCL 行只填柜型字段，LCL 行只填 LCL 字段；**禁止**把 R 列的 None 当作"运费缺失"打 warning。

### 4.8 R-NGB-8：附加费列存在"币种+数值+单位"混写为字符串的情况

- **实证**：
  - `V2='240/TEU'`（FAF/BAF 字段）
  - `W2='30/TEU'`（YAS/CAF 字段）
  - `AF2='USD30/BL'`（ENS/AFR 字段）
  - `AG2='USD145/TEU'`（LSF/LSS 字段）
  - `AI2="CNY20/20'\nCNY30/40'"`（ISPS 字段，**含换行 + 按柜型两档**）
  - `BA84='AFS USD30'`（LCL Other 1 字段）
- **业务含义**：原件采购部门为了"一格说清楚"把币种、数值、单位、按柜型分段全部塞进字符串。
- **parser 行为**（本轮）：**按字符串原文保留**；**不**强行解析为 `(currency, amount, unit, container_type_breakdown)` 的结构化字段。理由：
  1. 字符串格式不统一（`240/TEU` vs `USD30/BL` vs `CNY20/20'\nCNY30/40'`），强行结构化会引入大量边角 bug；
  2. Writer 端要原文回填（Writer 业需 §3-R3、§3-R4），结构化后再拼回去等于多此一举；
  3. Demo 当天楢崎只验"原文是否一字不差"，不验结构化质量。
- **Step2 风险**：未来 Step2 报价生成时如需把这些字符串拆开计算实际收费，**留给 Step2 处理**（本轮明确不做，参 §6 边界）。

### 4.9 R-NGB-9：日期列必须是 datetime（不是字符串）

- **实证**：`C2..D104` 全为 `datetime.datetime(2026, 4, 1)` / `datetime.datetime(2026, 4, 30)`。
- **业务含义**：批次有效期；下游入库要按日期排序、Writer 端要保持 Excel 日期类型（Writer 业需 §3-R7 第 143 行）。
- **parser 行为**：保留 datetime/date 类型；**禁止**强转字符串。

### 4.10 R-NGB-10：批次有效期来源 = Lv.1 行的 C/D 列（不在 sheet 元信息区）

- **业务规则**：与 Sea Net Rate 不同（Sea Net Rate 的 effective_from 在 `B3:D3` 元信息区），NGB 表**没有 sheet 级元信息行**，effective_from / to 直接从 Lv.1 数据行的 C/D 列读取。
- **实证**：`Rate!C2 = D2 = 2026-04-01 / 2026-04-30`，且全表所有数据行的 C/D 列均为同一对日期。
- **parser 行为**：取**任意一条 Lv.1 行的 C/D 列**作为整批 effective_from / effective_to；如果一份原件里出现"多种 (C, D) 组合"，必须打 warning 并以**最早 from + 最晚 until** 作为兜底。
- **❓ 待楢崎确认 Q-NGB-2**：未来是否会出现"同一份月度 NGB 表里，不同船司有不同生效期"？默认值：不会；若出现则 parser 取 min(from) / max(to) 并打 warning，由楢崎手工处理。

### 4.11 R-NGB-11：船司略称必须和 `Shipping line name` sheet 字典对得上

- **实证**：Rate 表 F 列出现 `SINO / COSCO / DONGJIN / SITC / NOS / TCLC / Coloader` 共 7 种；`Shipping line name` 字典里 B 列略称包含 `'COS'（COSCO）/ 'DJS'（DONGJIN）/ 'SIT'（SITC）` 等 40 项。
- **冲突点**：Rate 表 F 列写的是**全名首段**（`COSCO / DONGJIN / SITC`）而**不是**字典里的略称（`COS / DJS / SIT`）；`SINO`（SINOTRANS）/ `NOS` / `TCLC` 这三个略称**字典里没有**。
- **parser 行为**（本轮）：保留 F 列原文；**不强制**与字典做 join；不打 warning（因为字典缺项是常态，不是错误）。
- **❓ 待楢崎确认 Q-NGB-3**：`Shipping line name` 字典是否会更新以覆盖 `SINO / NOS / TCLC` 三个船司？默认值：不会，字典是日本侧维护的"参考链接"，与中方实际使用的船司列表本就不完全重合；本轮 parser 不做关联。

## 10. 业务风险（Demo 现场可能暴露）

| 风险编号 | 场景 | 影响 |
|---|---|---|
| BR-NGB-1 | parser 把 Lv.2/Lv.3 行的"主标识列继承公式"（`A3='=A2'` 等）当作"缺失"处理，导致 Lv.2/Lv.3 记录的 Agent / Origin / Destination 字段为空 | `/rates` 页过滤 Lv.2/Lv.3 时全部过滤不到；楢崎当场看不到分级报价 |

## 11. 与下游（架构大师 / Writer）的衔接

### 11.1 Parser → Writer 字段对称性约定

- Writer 端已实现（`backend/app/services/step1_rates/writers/ocean_ngb.py`），通过 `record.row_index` + `record.column_index_map` 驱动按"物理行 + 列号"回填。
- **业务侧要求 parser 输出**：
  - 每条记录带 `row_index`（原件物理行号，2..104）。
  - 每条记录带 `sheet_name = 'Rate'`（Writer 端会跳过 `sample` 和 `Shipping line name`）。
  - 每条记录带 `column_index_map`：把"业务字段→具体单元格值"映射，Writer 据此回填；其中**公式 cell 不要写入**（Writer 的 `is_formula_cell` 守卫会拒绝写公式 cell，参 Writer 业需 §10 坑 1 + B-Q-W4 实证），即 parser 在 column_index_map 里只放 Lv.1 行的运费数值列；Lv.2/Lv.3 行的运费列**不要塞进 column_index_map**（让 Writer 端公式自动联动）。
  - 同时输出"规整化字段"（如 `effective_from`、`effective_to`、`origin_port`、`destination_port`、`shipping_line`、`rate_level`、`twenty_ft_rate`、`forty_ft_rate`、`forty_hc_rate`、`lcl_per_cbm`、`lcl_per_ton` 等业务名）供 Step1 检索 / Step2 报价使用。规整化字段名由架构大师定，业务侧只规定**必须有**。
- **Writer 不需要 parser 输出公式字符串**——Writer 走"模板法 load 原件 → 改数据 cell → save"，公式 cell 自动保留（B-Q-W4 关闭实证）。

### 11.2 Parser 输出的批次元信息

- `file_type = ocean_ngb`
- `source_file = 原文件名`
- `effective_from = 2026-04-01`（取 Lv.1 行 C 列）
- `effective_to = 2026-04-30`（取 Lv.1 行 D 列）
- `records = 99 条`
- `warnings = [...]`（按 §10 列表，把"业务上正常的现象"避免误打 warning；只对真正异常的情况打）

### 11.3 不在 parser 责任内的事

- 入库后的批次 active/superseded 切换 —— 是 batch 服务的职责
- 与现有 `ocean` adapter 的去重 / 互查 —— 业务上两表是独立法人，不互查
- Demo 模板生成（templates 目录里的"空壳"模板）—— Writer 已经 OK，parser 不管
