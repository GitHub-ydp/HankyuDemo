## 需求 0：本轮范围裁定（最重要，架构大师按这个拆任务）

### 0.1 三个候选范围

- 范围 A（小）：只做 AI 参数设置 —— Provider / Base URL / API Key / Model / 超参 / 图片压缩。不含用户管理，不含系统其他可调项。
- 范围 B（中）：AI 参数 + 其他系统参数（邮件 IMAP、CORS、上传大小等 `config.py` 里剩下的可调项）。
- 范围 C（大）：AI 参数 + 用户管理（列用户、改角色、禁用启用、重置密码）+ 权限/角色概念落地到后端。

### 0.2 业务大师推荐：**范围 A（小）**

判断依据（三条全部成立才推 A）：

1. **用户原话只明确要求 AI 参数**。"AI 的地址、key、各种参数"是本轮原文里唯一具体可抓的需求；"控制用户"是上轮的一句口语化提及，没有给出"要哪些字段、谁能改谁、改完立即生效还是下次登录生效"等任何细节。材料未明，按规则不能脑补。
2. **Demo 5/14 的刚需只是"服务器地址会变"**。用户原话"因为我们的服务器地址可能会变，到时候直接程序中修改就可以了"——这是本轮的业务根因，只要 Base URL / API Key 能在 UI 改、改完下次 AI 调用生效，核心诉求就达成。用户管理在 5/14 Demo 现场楢崎不会点到（memory `project_demo_scope_v1` 明示 Demo = Step1+Step2 + Customer A only）。
3. **范围 C 会引爆认证系统重构**。当前后端**没有** user model、**没有** `/auth/*` 路由、**没有** role 字段落库（代码证据：`frontend/src/contexts/AuthContext.tsx:29-36` 自己实现了 djb2 哈希写 localStorage，注释原文"上线前换成后端 bcrypt"）。要做"用户管理"必须先做"后端鉴权系统"，这一步拉出来至少 2~3 天工作量，且与"服务器地址会变"这个当下痛点无关。**做了就是半成品**，不做 Demo 也不穿帮。
4. **范围 B 的增量价值低**。邮件 IMAP 已在 Step2 外的 `EmailConnection` 组件里有独立配置入口（`frontend/src/components/EmailConnection`），CORS/上传大小不是业务人员该碰的参数，塞进设置页会让页面重心偏离"我就想改 AI 地址"这个单一诉求。

### 0.3 本轮承诺的范围边界（写给开发大师）

- 做：AI 相关 11 个字段的查看与修改（见 §1）
- 做：一个"测试连通性"按钮，查当前 AI 是否可达
- 做：设置页的三语 i18n
- 做：设置页的菜单入口（齿轮图标）
- 不做：用户/角色/权限管理（留 TODO，写进 §7）
- 不做：邮件 IMAP、CORS、上传大小、数据库 URL 等其他系统参数
- 不做：配置变更历史、版本回滚、导入导出
- 不做：多租户、多环境配置切换

## 需求 1：AI 参数设置（核心）

### 1.1 可调字段总集（11 项）

出处：`backend/app/core/config.py:49-75`；AI 客户端消费点：`backend/app/services/ai_client.py`

| 业务分组 | 字段业务名 | 字段性质 | 业务含义 |
|---|---|---|---|
| Provider 切换 | AI 提供方 | 单选 | 当前请求走哪个 AI 后端（vLLM / Anthropic） |
| Provider 切换 | 超时秒数 | 数字 | 单次 AI 调用最长等待 |
| Provider 切换 | 自动 /no_think | 开关 | vLLM 分支给 user 文本末尾加 `/no_think`（Qwen 思考关闭） |
| vLLM 连接 | vLLM 服务地址 | 文本 | 用户最关心的"会变的服务器地址"，形如 `http://43.133.197.65:8000/v1` |
| vLLM 连接 | vLLM API Key | 密文 | 访问 vLLM 的鉴权 |
| vLLM 连接 | vLLM 模型名 | 文本 | 形如 `qwen3.6-27b` |
| vLLM 连接 | 启用思考 | 开关 | 是否允许思考（默认关） |
| vLLM 连接 | 启用模板参数 | 开关 | 是否带 `chat_template_kwargs`（阿里云百炼 400 时要关） |
| Anthropic 连接 | Anthropic API Key | 密文 | Claude 分支的 key |
| Anthropic 连接 | Anthropic 模型名 | 文本 | 形如 `claude-sonnet-4-20250514`；当前代码硬编码，UI 上暴露出来便于切版本 |
| Token/超参 | 默认 max_tokens | 数字 | `ai_max_tokens_default`，范围 64~8192 |
| Token/超参 | 抽取 JSON 的 max_tokens | 数字 | `ai_max_tokens_extract_json`，范围 64~8192 |
| Token/超参 | max_tokens 上限 | 数字 | `ai_max_tokens_cap`，vLLM 死守的天花板，范围 256~4096 |
| 图片压缩 | 启用压缩 | 开关 | 开关控制 `ai_image_compress` |
| 图片压缩 | 最长边像素 | 滑块 | 范围 640~2048，步长 64 |
| 图片压缩 | JPEG 质量 | 滑块 | 范围 60~95，步长 5 |

（注：上表列了 16 行但合并成 11 个业务字段组，架构大师自行判断是否拆多字段或合并 group。）

### 1.2 每个字段的交互逐字段定义

#### 1.2.1 AI 提供方（Provider）

- 控件：单选 radio（两个选项：vLLM / Anthropic），不做多选，不做"同时启用"。
- 切换时弹确认框：**"切换 AI 提供方会改变下一次 AI 调用的后端，确定切换到 Anthropic 吗？"**（反之亦然）。原因：切错后 Step2 PKG 自动填表会整条链路直接 fail。
- 当前选中的 provider 在标题旁用 tag 显著显示："当前：vLLM"。
- 切换后不立即调用远端验证，仅做"保存成功提示"；**验证由用户自己点"测试连通性"按钮**。

#### 1.2.2 vLLM 服务地址（Base URL）

- 控件：**自由输入 + 快捷下拉**。
- 快捷下拉 3 个预设（点了填入输入框，仍可编辑）：
  - 生产 vLLM：`http://43.133.197.65:8000/v1`
  - 本地开发：`http://localhost:8000/v1`
  - 阿里云百炼（回滚兜底）：`https://dashscope.aliyuncs.com/compatible-mode/v1`
- 前端 validate：必须以 `http://` 或 `https://` 开头，否则保存按钮禁用 + 红色提示"地址必须以 http:// 或 https:// 开头"。
- 不 validate 路径是否以 `/v1` 结尾（阿里云是 `/compatible-mode/v1`，死规则会误杀）。

#### 1.2.3 API Key（vLLM / Anthropic 都适用）

- 展示：默认**只显示后 4 位，前面一律 `••••••••••••`**。例如 `sk-abcdefg1234` 显示为 `••••••••1234`。
- 编辑：右侧放一个"修改"按钮。点击后切换为明文输入框（新输入，不自动回填旧值）。
- 取消编辑：再点"取消"回到脱敏展示。
- 保存前二次确认："确定要修改 API Key 吗？改错会导致 AI 调用失败。"
- **空值策略**：允许保存为空，但保存后前端立即给一条警告 toast：`API Key 已清空，该 Provider 下一次调用会失败`。不硬拦截，因为可能用户就是临时清空用于切到另一个 provider。

#### 1.2.4 模型名（Model）

- 控件：文本输入。不做下拉，不做按 provider 自动填充（常见值会过期，反而误导）。
- 旁边一行灰字提示（三语）："常见值：vLLM 用 `qwen3.6-27b`；Anthropic 用 `claude-sonnet-4-20250514`"。

#### 1.2.5 max_tokens 三档

- 三个独立数字输入框，每个加 placeholder 显示默认值（512 / 1024 / 1536）。
- 前端 validate：`ai_max_tokens_default` ≤ `ai_max_tokens_cap`；`ai_max_tokens_extract_json` ≤ `ai_max_tokens_cap`。不满足时保存按钮禁用 + 红字提示。

#### 1.2.6 超时（Timeout）

- 数字输入，单位"秒"，默认 90，范围 **10~600**。超界用前端 validate 拦截。

#### 1.2.7 三个开关（自动 /no_think、启用思考、启用模板参数）

- Ant Design Switch 控件。
- 每个旁边有 `?` tooltip 图标，tooltip 内容三语：
  - 自动 /no_think：用于 vLLM/Qwen 模型，在 user 文本末尾追加 `/no_think` 以关闭思考链，节省 token。Anthropic 下不生效，但可以开着不影响。
  - 启用思考：仅 vLLM/Qwen 有效。关闭可显著降 max_tokens 消耗。建议保持关闭。
  - 启用模板参数：仅 vLLM 生效。切到阿里云百炼若被 HTTP 400 拒绝请关闭此项。

#### 1.2.8 图片压缩三项

- "启用压缩"：Switch。
- "最长边像素"：Slider，范围 640~2048，步长 64，当前值实时显示。
- "JPEG 质量"：Slider，范围 60~95，步长 5，当前值实时显示。
- 三项合并放在"图片压缩"分组框内。开关关闭时，两个 Slider 灰掉（disabled），但数值保留。

### 1.3 "测试连通性"按钮

- 位置：Provider 分组右上角，独立按钮，文案"测试连通性 / 接続テスト / Test Connection"。
- 行为：调 `ai_client.health_check()`（架构大师已预留），返回 `{provider, ok, latency_ms, detail}`。
- UI 反馈：
  - 成功：绿色气泡 Toast，"vLLM 连接正常，延迟 142ms"；
  - 失败：红色气泡 Toast，显示 `detail` 内容（后端应给人话错误："Base URL 未配置" / "API Key 未配置" / "HTTP 503 服务不可达" / "超时 90 秒"）。
- 本按钮**不保存**任何修改，仅测当前已保存的配置。**未保存的编辑**不参与测试——UI 上若有未保存变更，按钮禁用并提示"请先保存再测试"。这避免"测了用户看到的但后端配置是旧的"这种错位。

## 需求 2：持久化策略

### 2.1 业务大师推荐：**方案 β — DB 存单行 `app_settings` 表**

（具体字段/表名由架构大师定，这里只说业务约束。）

判断依据：

1. **方案 α 写 `.env`**：后端进程启动时 `pydantic-settings` 只读一次 `.env`，写文件后必须重启 uvicorn 才生效。Demo 现场重启后端会让运价 seed、已上传批次、前端 session 全部可能出问题，不可接受。
2. **方案 γ 内存单例 + DB merge**：实现最灵活，但是内存单例在多 worker（uvicorn `--workers 4`）下各 worker 各自一份，改完可能 4 个 worker 里 3 个还拿旧值。排除。
3. **方案 β 的业务承诺可交付**：
   - 保存即写 DB，AI 客户端在每次 `chat()` 调用前从 DB 读一次"当前 effective 配置"覆盖 `settings.*` 默认值；
   - 多 worker 下每个 worker 都读同一张表，天然一致；
   - 启动时 `.env` 仍然是默认值来源（fallback），DB 里的记录覆盖 env（DB 优先级 > env）；
   - 不做内存缓存（为了业务简洁，不值得引入缓存失效问题）；性能影响：每次 AI 调用多一次 DB 读，单行查询，可接受。

### 2.2 业务上的硬约束

- **DB 里只存"与默认值不同"的项**（推荐）。这样当有人手动改了 `.env` 默认值后，未被用户覆盖的字段会跟着默认值变化。但架构大师若判断"全量存"更易维护，也可接受，不是业务红线。
- **敏感字段（API Key）在 DB 里不要明文落盘**。最低要求：base64 或 AES-Sym 加密；加密 key 从 env 注入（`APP_SETTINGS_ENC_KEY`）。本条是红线。
- **DB 里不要存"用户改配置的历史记录"**（本轮不做审计，见 §7）。

## 需求 3：生效方式

- **保存成功后立即生效**。下一次 AI 调用读新值。
- **不中断正在进行的 AI 请求**。正在 `await` 的 `chat()` 调用继续用旧配置跑完；新请求拿新配置。
- **不提供"重启后端"按钮**。风险过大，违反用户"不能让演示中断"的 Demo 红线。
- 若用户手动改了 DB 里的 `app_settings` 跳过 UI（例如 psql 直连），本轮不保证实时生效；允许下一次请求才读到。

## 需求 4：UI 交互设计

### 4.1 入口位置

- **顶部栏右上角加一个齿轮图标**（`Icon name="settings"`），与现有的 Reset/Help/Bell 三个 icon 并列，插在"help"右侧。
- 不进左侧侧边栏。理由：设置页是**运维/管理员**属性的入口，业务流操作（仪表盘/运价/PKG 等）优先占用侧边栏；齿轮属于标准 SaaS 形态，楢崎能秒认。
- 点击齿轮跳 `/settings`（架构大师注册路由）。

### 4.2 页面结构

- **单页多 section**，不分 Tab。理由：本轮只有 AI 一个类别，分 Tab 反而空洞。
- Section 列表（从上到下）：
  1. Section A：Provider 切换（含 Provider 单选 + 超时 + 自动 /no_think + 测试连通性按钮）
  2. Section B：vLLM 连接（5 个字段）
  3. Section C：Anthropic 连接（2 个字段）
  4. Section D：Token 超参（3 个字段）
  5. Section E：图片压缩（3 个字段）
- 每个 Section 有标题 + 一段三语业务说明（`help-text`）。

### 4.3 保存按钮

- 页面底部固定一条操作栏，三个按钮：
  - **保存**（主按钮，绿色）—— 有未保存变更时点亮，无变更时灰掉
  - **取消**（次按钮）—— 丢弃本次编辑，恢复到最后一次保存的值
  - **重置为默认**（危险按钮，右侧）—— 把所有字段回到 `.env` 默认值（不是回到"保存过的值"），**需要二次确认**："重置为默认会清空所有在 UI 上改过的 AI 配置，包括 API Key，确定继续吗？"
- **不做**"编辑模式 → 查看模式"切换，所有输入框常驻可编辑，降低点击成本。

### 4.4 未保存离开拦截

- 有未保存变更时切换路由 / 关闭标签页：弹 `Modal.confirm`："设置尚未保存，确定离开吗？"三选一：离开丢弃 / 留下 / 保存后离开。

### 4.5 敏感字段二次确认

- 已在 §1.2.3 写明 API Key 修改前有二次确认。其他字段（地址、模型名等）**不加二次确认**，避免点击疲劳。

### 4.6 风格一致性（用户硬约束）

参照 `frontend/src/pages/RateList.tsx`（最主干的业务页）和 `frontend/src/pages/RateBatches.tsx`（含 Drawer/Modal 交互），硬对齐以下 5 点：

1. **卡片背景色**：沿用现有 `app-card` 或等价变量，**不**引入新的灰白风；
2. **按钮组件**：Ant Design `<Button>`，主按钮用现有主题绿 `#4FAFB5`（从 Login.tsx 可见的品牌色），危险按钮用 `var(--danger)`；
3. **表单输入框**：Ant Design `<Input>` / `<InputNumber>` / `<Select>` / `<Switch>` / `<Slider>`，不自定义 CSS 外观；
4. **间距节奏**：Section 之间间距与 `RateList.tsx` 卡片间距一致（现状 24px 左右）；
5. **三语 i18n 集中到 `frontend/src/i18n/{zh,ja,en}.json` 的 `settings.*` 段下**，不散在组件里硬编码。

### 4.7 三语 i18n key 清单

在 `frontend/src/i18n/{zh,ja,en}.json` 下新增 `settings` 段，至少包含：

## 需求 5：权限控制

- **现状**：后端无 user model、无 role 字段、无 `/auth/*` 路由，所有认证在前端 localStorage（`AuthContext.tsx:1-148`）。
- **本轮策略**：**所有已登录用户都能进设置页**。不做 admin-only 拦截。
  - 理由：本轮也没 admin 概念，假装拦也是 UI 层假拦截，徒增复杂度；
  - 真实权限控制等后端认证系统补齐后再说，留 TODO 写进 §7。
- **未登录用户**仍然被 `ProtectedRoute.tsx` 拦到 /login（现状行为不变）。

## 需求 7：本轮不做范围（防开发大师越界）

下列功能全部明确**不**做，有需求走下一轮评审：

1. 用户管理（列用户、改角色、禁用启用、重置密码）
2. 后端认证系统迁移（bcrypt 落库、`/auth/*` 路由、JWT session 等）—— 与设置页耦合的硬块，但不是本轮任务
3. 角色/权限模型（admin / ops / viewer 等）
4. 邮件 IMAP 参数、CORS、上传大小、数据库 URL 等非 AI 的系统参数
5. 配置变更历史、版本回滚、变更审计表
6. 配置导入/导出（JSON 上传下载）
7. 多租户、多环境（dev/stg/prod）配置切换
8. 配置预设模板库（"加载企业 A 的预设配置"）
9. AI 调用监控看板（QPS、延迟、失败率可视化）—— 本轮只给一个按钮测连通
10. AI cost/计费统计
11. 前端只读模式（"查看配置但不能改"的 viewer 权限）

## 需求 8：失败场景与边界

| 场景 | 期望行为 |
|---|---|
| API Key 保存为空 | 允许保存，保存后 toast 警告"已清空，调用会失败"；不硬拦截 |
| Base URL 不是 http(s):// 开头 | 前端 validate 拦截，保存按钮禁用 + 输入框红框提示 |
| Base URL 是 http(s):// 但远端不存在 | 保存成功，但"测试连通性"按钮会失败 |
| 切到 Anthropic 但 Anthropic API Key 未配置 | 保存成功；下一次 AI 调用时 `ProviderUnavailableError`；前端捕获后显示红色错误 |
| max_tokens default > cap | 前端 validate 拦截 |
| Timeout < 10 或 > 600 | 前端 validate 拦截 |
| 图片 JPEG 质量 < 60 或 > 95 | Slider 本身范围限制，不可能越界 |
| 测试连通性时后端 health_check 本身抛异常 | UI 按"失败"处理，toast 显示 `detail: internal error, see backend logs` |
| 未保存切路由 | `Modal.confirm` 拦截 |
| 两个用户同时编辑（并发写） | 后写赢（last-write-wins），本轮不做乐观锁 |
| DB 里 app_settings 表缺失（首次部署） | 后端读不到时回退 `.env` 默认值，保存时自动建/插，不报错 |
| DB 里字段类型对不上（老代码升级后） | 架构大师负责 migration；业务侧不做兜底 |

## 需求 9：验收清单（楢崎/赵总/张东旭在 UI 上能验证的事实）

1. 已登录用户在顶部栏右上角能看到齿轮图标，点击跳转到 `/settings`，页面标题显示"系统设置 / システム設定 / System Settings"。
2. 页面分 5 个可见的 section：Provider / vLLM / Anthropic / Token 超参 / 图片压缩。
3. Section A 中"AI 提供方"默认选中的 radio 与当前 `settings.ai_provider` 一致（未改过的情况下应为 vLLM）。
4. 修改"vLLM 服务地址"为一个合法的 http(s):// 地址后点"保存"，右下角出现"保存成功"toast；刷新页面后值仍然保留。
5. API Key 字段默认显示为 `••••••••` + 后 4 位；点"修改"切为明文输入框可编辑；不点"修改"无法改动。
6. 修改 API Key 保存时弹出"确定要修改 API Key 吗"的二次确认；确认后保存成功。
7. 点击"测试连通性"按钮：vLLM 在线时弹绿色 toast 含延迟毫秒数；vLLM 故意改错地址后弹红色 toast 含具体错误信息。
8. 切换 Provider 时弹出切换确认 Modal；拒绝则保持原 Provider 不变。
9. 修改未保存时点击左侧任意菜单项，弹出"尚未保存"拦截 Modal。
10. 点击"重置为默认"弹危险二次确认；确认后所有字段回到 `.env` 默认值，包括 API Key 被清空。
11. 切换语言到日文/英文，section 标题、字段名、按钮文案全部同步切换，无中文残留。
12. 任意 AI 调用（例如 Step2 PKG 自动填表）在配置修改保存后，**下一次调用**走新配置（可通过改错 Base URL + 触发一次 PKG 填表 → 观察到 `ProviderUnavailableError` 报错来反向验证）。
